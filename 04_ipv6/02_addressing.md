# IPv6 アドレッシング — 128 ビットの読み方と配り方

## 概要

この章では、IPv6 アドレスの表記法・種別・スコープ・構造(RFC 4291)を扱う。
「1インタフェース = 複数アドレスが正常」という IPv4 との発想の違いと、
それを秩序立てるアドレス選択(RFC 6724)までを本章の範囲とする。
前提知識は[前章](01_why_ipv6.md)(なぜ 128 ビットか、基本ヘッダ)と
[第1部02章](../01_fundamentals/02_routing_table_basics.md)(CIDR、経路集約)。
アドレスを実際に自動構成するプロトコル(NDP、SLAAC)は
(後述: `03_ndp_slaac.md`)で扱う。

## 導入 — 128 ビットをどう書くか

### 表記法 — 16 ビット × 8 グループ

IPv6 アドレスの表記は **RFC 4291** が定める。128 ビットを
16 ビットずつ8グループに割り、各グループを16進数で書いて
コロンで区切る。

```text
2001:0db8:0000:0000:0000:0000:0000:0001
```

このままでは長すぎるので、2つの省略規則がある。

1. **各グループの先頭のゼロは省略できる**: `0db8` → `db8`、`0000` → `0`
2. **連続するオールゼロのグループを、アドレス中1回だけ `::` に
   置き換えられる**: 上の例は `2001:db8::1`

`::` が「1回だけ」なのは曖昧さを避けるためである。
`2001:db8::1::2` のように2箇所を潰すと、省略されたゼロが
それぞれ何グループ分か復元できなくなる。

プレフィックスの表記は IPv4 の CIDR と同じ `/n` 形式で、
`2001:db8::/32` は「上位 32 ビットが一致する範囲」を意味する。
IPv4 のようなドット区切りのネットマスク表記は存在しない。

### 「書き方が2通りある」ことの運用コスト — RFC 5952

省略規則は**任意**である。つまり同じアドレスに合法な表記が複数ある。

```text
2001:db8:0:0:1:0:0:1     省略なし(先頭ゼロだけ省略)
2001:db8::1:0:0:1        前のゼロ列を圧縮
2001:db8:0:0:1::1        後のゼロ列を圧縮
2001:DB8::1:0:0:1        大文字
```

4つはすべて同じアドレスである。人間には些細な違いだが、
ログの grep、設定ファイルの diff、監視システムの突合といった
**文字列としての比較**はすべて破綻する。この問題への答えが
**RFC 5952** で、機械がアドレスを出力するときの正規形を定めた —
(1) 16進は小文字、(2) 先頭ゼロは必ず省略、(3) `::` は最長のゼロ列に
適用(同長なら最初のほう)、(4) 1グループだけのゼロを `::` にしない。
上の例の正規形は `2001:db8::1:0:0:1` である。
自作ツールがアドレスを扱うときは、文字列比較の前に
正規化(パースして比較)するのが鉄則である。

もう1つ、コロンにまつわる実務の罠として、ポート番号の併記がある。
`2001:db8::1:80` は「ポート 80」ではなくそれ全体が1つのアドレスに
見えてしまう。URL などでは **`[2001:db8::1]:80`** のように
角括弧で囲む(RFC 3986)。

### 発想の転換 — 「ホストのアドレス」は1つではない

表記より重要な、IPv4 との根本的な違いを先に述べておく。
IPv4 では「1インタフェースに1アドレス」が原則で、複数持たせるのは
例外的な構成だった。IPv6 では**1つのインタフェースが複数のアドレスを
同時に持つのが正常状態**である。

- リンク内専用の**リンクローカルアドレス**(必ず1つ持つ)
- インターネットで使う**グローバルユニキャストアドレス(GUA)**
  (プライバシーのための一時アドレスを含めて複数になりうる)
- 組織内用の **ULA** を併せ持つ設計もある

「このホストの IP アドレスは何か」という IPv4 の問いは、IPv6 では
「この通信に**どのアドレスを使うか**」という選択の問題に変わる。
それぞれのアドレスには**届く範囲(スコープ)**と**寿命**が付いており、
選択の規則(RFC 6724)まで含めて初めて体系が閉じる。
本章はこの順 — 空間の地図 → 種別とスコープ → 選択 — で進む。

## 理論

### アドレス空間の全体地図(RFC 4291)

まず 128 ビット空間のどこに何がいるかの地図を持つ。

| プレフィックス | 用途 | 定義 |
|---|---|---|
| `::/128` | 未指定アドレス(アドレス未確定時の送信元。IPv4 の 0.0.0.0 相当) | RFC 4291 |
| `::1/128` | ループバック(IPv4 の 127.0.0.1 相当。/8 も取らず1個だけ) | RFC 4291 |
| `::ffff:0:0/96` | IPv4-mapped アドレス(`::ffff:192.0.2.1`。デュアルスタックの API 内部表現で、原則ワイヤ上には現れない) | RFC 4291 |
| `64:ff9b::/96` | NAT64 の well-known prefix((後述: `05_transition_technologies.md`)) | RFC 6052 |
| `2000::/3` | **グローバルユニキャスト(GUA)** | RFC 4291 |
| `2001:db8::/32`、`3fff::/20` | ドキュメント用(本書の例もこれ) | RFC 3849、RFC 9637 |
| `fc00::/7` | **ユニークローカル(ULA)** | RFC 4193 |
| `fe80::/10` | **リンクローカル** | RFC 4291 |
| `ff00::/8` | **マルチキャスト** | RFC 4291 |
| 上記以外 | 予約(未割り当て) | — |

注目すべきは、グローバルユニキャストに現在使われているのが
`2000::/3` — 全空間の **1/8** — にすぎないことである。
それでも 2^125 個あり、設計を誤ったと分かってもやり直せる余白が
確保されている。「潤沢さを前提にする」とは、無計画に使うことではなく、
**保守的に配ってもなお困らない**ということである。

ブロードキャストアドレスはこの地図に存在しない。前章で見たとおり、
「全員へ」はマルチキャストの1グループ(ff02::1)にすぎない。

### GUA の構造 — 階層割り当てと集約

GUA の内部構造は3層である(RFC 3587、RFC 4291)。

```text
|         n ビット           |   64 - n ビット   |        64 ビット        |
+---------------------------+------------------+------------------------+
| グローバルルーティング       |   サブネット ID    |    インタフェース ID     |
| プレフィックス              |                  |    (IID)               |
+---------------------------+------------------+------------------------+
 ↑ どの組織か(世界に対する座標)  ↑ 組織内のどの      ↑ サブネット内の
                               サブネットか        どのインタフェースか
```

グローバルルーティングプレフィックスは IANA → RIR → ISP → 組織の
階層で配られる。典型的なサイズ感は:

- ISP(LIR)には **/32** 程度(サブネット ID まで 2^16 の顧客ブロック)
- 組織(サイト)には **/48**、家庭向けには **/56** が代表的
  (RFC 6177 は「一律 /48」の旧方針を改め、/48〜/56 の幅を認めた)
- 個々のサブネット(VLAN、リンク)は **/64**

/48 の組織はサブネット ID に 16 ビット = **65,536 個のサブネット**を
持てる。IPv4 で「このセグメントは 60 台だから /26」とホスト数から
逆算していた作業は消滅し、アドレス設計は**サブネットの数と階層**だけを
考える仕事になる。実務では 16 進1文字 = 4 ビットの
**ニブル境界**(/48、/52、/56、/60、/64)で切るのが定石である。
表記上で階層が読み取れ、DNS の逆引き(ip6.arpa はニブル単位の委任)とも
整合するからである。

この階層割り当ての狙いは[第1部02章](../01_fundamentals/02_routing_table_basics.md)の
経路集約そのものである。ISP は世界に向けて /32 を1本広告すれば済み、
顧客の増減は外に波及しない。IPv4 の経路表を肥大させた
「希少ゆえの細切れ割り当て」を、潤沢さによって構造的に避けている。

### なぜ /64 で切るのか

サブネットが原則 /64 なのは慣例ではなく仕様である。
RFC 4291 は、(バイナリ 000 で始まる特殊な範囲を除く)ユニキャスト
アドレスの IID を **64 ビットとする**と規定し、
Ethernet 上のアドレス自動構成(RFC 2464、SLAAC)も 64 ビット IID を
前提に定義されている。/64 より長いプレフィックスのサブネットでは
SLAAC が動かない(設計の経緯と論点の整理は RFC 7421)。

「1サブネットに 2^64 個は無駄では」という直感には、こう答えられる —
IID を十分大きくランダムにできることが、端末が**誰にも聞かずに
アドレスを自己生成して衝突をほぼ心配しない**(SLAAC、プライバシー
アドレス)ための土台であり、64 ビットは「収容数」ではなく
**自己生成の名前空間**として使われている。

例外が1つある。ルータ間のポイントツーポイントリンクには
**/127** を使ってよい(RFC 6164)。端末のいないリンクで /64 を使うと、
存在しない 2^64 個のアドレスへのパケットがルータの近隣探索
(アドレス解決の待ち行列)を溢れさせる攻撃(RFC 6583)が成立するため、
その余地自体を消す運用である。「原則 /64、ルータ間 /127」が
現代の設計の定石になっている。

### スコープ — 届く範囲がアドレスに内蔵されている

IPv4 では「このアドレスはどこまで届くか」は予約表と運用の約束
(RFC 1918 は経路表に載せない、など)だった。IPv6 では**スコープ**という
概念が体系に組み込まれており(RFC 4007)、アドレス自身が自分の
有効範囲を宣言している。

**リンクローカルアドレス**(fe80::/10)は、そのリンク(L2 セグメント)
上でのみ有効なアドレスである。3つの性質を押さえておく。

1. **必ずある。** IPv6 が有効なインタフェースは、外部の設定・サーバ・
   ルータの一切なしに、リンクローカルアドレスを必ず1つ自己生成する
   (前章の設定例で「何もしていないのに付いていた」アドレスがこれである)。
2. **プロトコルの土台である。** NDP・RA((後述: `03_ndp_slaac.md`))は
   リンクローカルで動く。OSPFv3 のネイバーもリンクローカルで張られ、
   [第3部05章](../03_bgp/05_mp_bgp.md)で見た「IPv6 ネクストホップの
   グローバル+リンクローカル2個併記」(RFC 2545)のリンクローカル側も
   これである。さらに、RA で配られるデフォルト経路のネクストホップも
   リンクローカルになる — **IPv6 のデフォルトゲートウェイは
   fe80:: で始まるのが正常**である。
3. **リンクを越えず、リンクごとに重複しうる。** ルータは
   リンクローカル宛てのパケットを絶対に転送しない。そして eth0 の先の
   `fe80::1` と eth1 の先の `fe80::1` は別のノードでありうる。

3つ目の性質から**ゾーン ID** が必要になる。`fe80::1` という文字列だけ
では宛先が一意に決まらないため、「どのリンクの話か」を
`fe80::1%eth0` のように付記する(RFC 4007)。前章の
`ping ff02::1%eth0` の `%eth0` の正体がこれである。IPv4 に
このような構文が無かったのは、リンク内専用アドレスという概念自体が
体系に無かったからである(169.254.0.0/16 = RFC 3927 は後付けの例外)。

### ULA — プライベートアドレスの後継、ただし NAT の前提ではない

**ULA**(Unique Local Address、RFC 4193)は組織内利用のための
ユニキャスト空間 fc00::/7 である。実際に使えるのは L ビット = 1 の
**fd00::/8** で、構造が面白い。

```text
| 8 ビット |     40 ビット      |  16 ビット   |     64 ビット     |
+---------+-------------------+-------------+------------------+
|   fd    |  グローバル ID      | サブネット ID |  インタフェース ID |
|         |  (乱数で自己生成)   |             |                  |
+---------+-------------------+-------------+------------------+
```

RFC 1918 との決定的な違いは、40 ビットの**グローバル ID を乱数で
生成する**ことである。10.0.0.0/8 は世界中の組織が同じ 10.1.1.0/24 から
使い始めるため、組織の統合・VPN 接続のたびにアドレス重複と
リナンバリングが発生した。fd00::/8 では2組織がたまたま同じ /48 を
引く確率は 2^-40(約 10^-12)であり、**登録機関なしで実用上一意**が
成り立つ。「潤沢さは乱数による無調整の一意性を買える」という、
IPv6 の設計思想の縮図である。

もう1つの違いは使い方の想定である。RFC 1918 は NAT とセットで
「グローバルの代わり」に使うものだった。ULA は **NAT の前提ではない** —
1インタフェース複数アドレスの原則の上で、**GUA と併存**させ、
組織内の通信(ISP を替えても変わらない安定した内部アドレス)には ULA を、
インターネットへは GUA を、アドレス選択(次節)に使い分けさせる設計である。
「ULA だけを配って RFC 1918 と同じ感覚で使う」と何が起きるかは、
トラブルシューティング③で見る。

### マルチキャストアドレスの構造

ff00::/8 の内部構造にもスコープが埋め込まれている。

```text
|  8 ビット  | 4 ビット | 4 ビット |          112 ビット           |
+-----------+---------+---------+------------------------------+
| 11111111  |  flags  |  scope  |         グループ ID           |
+-----------+---------+---------+------------------------------+
```

scope フィールド(RFC 7346)の主な値は、1 = インタフェースローカル、
**2 = リンクローカル**、5 = サイトローカル、e = グローバル。
つまり `ff02::1` は「リンク内(2)の全ノードグループ」と
アドレス自身が読める。同じグループ ID でもスコープ違いは別アドレスである。
flags は最下位ビットが T(transient)で、0 なら IANA 登録の恒久グループ、
1 なら動的な一時グループを示す。

覚えておくべき well-known グループは少数である。

| アドレス | 意味 |
|---|---|
| `ff02::1` | リンク上の全ノード |
| `ff02::2` | リンク上の全ルータ |
| `ff02::5` / `ff02::6` | OSPFv3 の AllSPF / AllDR([第1部05章](../01_fundamentals/05_igp_overview.md)の 224.0.0.5/6 に対応) |
| `ff02::1:2` | DHCPv6 リレー/サーバ |
| `ff02::1:ffXX:XXXX` | **要請ノードマルチキャスト**(プロトコル動作の詳細で後述) |

### エニーキャスト

IPv6 はエニーキャスト — **同じユニキャストアドレスを複数のノードに
割り当て、経路制御で最寄りの1台に届ける**配送形態 — を体系に
明示的に含めた(RFC 4291)。重要なのは、エニーキャストアドレスに
専用のフォーマットが**ない**ことである。ワイヤ上ではユニキャストと
区別できず、「複数の場所から同じプレフィックスが広告されている」という
経路制御上の状態がエニーキャストの実体である。

RFC 4291 が予約する具体例はサブネットルータエニーキャスト
(IID 全ゼロ = そのサブネットのルータのどれか)だが、実務で目にする
エニーキャストの多くはむしろ広域のもの — DNS ルートサーバのように
同一プレフィックスを世界中の拠点から BGP で広告する構成 — である。
[第2部05章](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)の
エニーキャストゲートウェイも「同じアドレスを複数の場所に置き、
最寄りに処理させる」という同型の発想だった。

### 複数アドレスの世界の秩序 — アドレス選択(RFC 6724)

ここまでで、1台のホストが リンクローカル + GUA(複数)+ ULA を
持ちうることが分かった。では `curl http://[2001:db8::1]/` を実行した
瞬間、OS は**送信元**をどう決めるのか。宛先に A と AAAA の両方が
あるとき、**宛先**をどう選ぶのか。この規則が **RFC 6724**
(Default Address Selection)である。仕組みの骨格だけ押さえる。

**送信元選択**は、候補アドレスを規則の連鎖でふるい落とす —
主なものは (1) 宛先と同じアドレスがあればそれ、(2) **スコープを
宛先に合わせる**(リンクローカル宛てにはリンクローカルを、
グローバル宛てには GUA を)、(3) **deprecated(後述)より preferred を
優先**、(4) 既定では**一時アドレスを恒久アドレスより優先**、
(5) 宛先との**最長一致プレフィックス**。

**宛先選択**(A/AAAA の選択を含む)は、プレフィックスごとの
優先度表(policy table)による。既定の抜粋:

| 宛先プレフィックス | 優先度 | 意味 |
|---|---|---|
| `::1/128` | 50 | ループバック |
| `::/0` | 40 | IPv6 一般(GUA) |
| `::ffff:0:0/96` | 35 | **IPv4**(宛先が IPv4 のときはこの形で表に掛ける) |
| `2002::/16` | 30 | 6to4(歴史的) |
| `fc00::/7` | **3** | **ULA** |

GUA(40)> IPv4(35)がデュアルスタックで「IPv6 があるなら IPv6」と
なる根拠である。そして **ULA(3)< IPv4(35)** — ULA しか持たない
ホストは、AAAA があっても IPv4 を選ぶ。これは「ULA は組織内用であり、
グローバルな宛先への到達性を約束しない」という設計意図の反映であり、
バグではない(トラブルシューティング③)。

## プロトコル動作の詳細

### インタフェース ID の生成① — Modified EUI-64

下位 64 ビットの IID を端末はどう作るか。最初に標準化された方式が
**Modified EUI-64**(RFC 4291 Appendix A)で、48 ビットの MAC アドレス
から機械的に導出する。

```text
 MAC アドレス:            52:54:00:12:34:56
                              │
 ① 真ん中に ff:fe を挿入:  52:54:00:ff:fe:12:34:56
                              │
 ② 第1オクテットの U/L ビット(下位から2ビット目)を反転:
    52 = 0101 0010  →  0101 0000 = 50
                              │
 IID:                     5054:00ff:fe12:3456
 リンクローカル:           fe80::5054:ff:fe12:3456
```

前章の設定例に現れた `fe80::5054:ff:fe12:3456` の正体がこれである
(MAC 52:54:00:12:34:56 の変換結果)。`ff:fe` が真ん中に見える IID は
EUI-64 由来だと読める。U/L ビットの反転は「手動設定の簡便なアドレス
(fe80::1 など)が『ローカル管理』の意味論に自然に収まるように」という
配慮だったが、RFC 7136 により現在この意味論は放棄され、
IID は不透明なビット列として扱うことになっている。

EUI-64 は決定的で衝突しにくい反面、重大な問題を抱えていた —
**IID が MAC 由来で世界中どこでも同じ**になる。ノート PC が
自宅・職場・カフェでプレフィックスだけ違う同じ IID を使えば、
アドレスの下位 64 ビットだけで個人の移動が追跡できてしまう。

### インタフェース ID の生成② — 一時アドレスと安定不透明 ID

追跡問題への答えは2段階で標準化された。

- **一時アドレス**(temporary address、RFC 8981、旧 RFC 4941):
  乱数で IID を生成し、**短い寿命で次々に乗り換える**。既定では
  1日程度で新しい一時アドレスが preferred になり、古いものは
  deprecated を経て消える。外向きの通信の送信元として使われ
  (RFC 6724 の規則(4))、「同じ端末だと突合できない」ことを目的とする。
- **安定不透明 ID**(RFC 7217): サーバ用途などでは安定した
  アドレスも必要である。RFC 7217 は
  ハッシュ(秘密鍵, プレフィックス, インタフェース, ...)で IID を作る —
  **同じネットワークでは常に同じ**(DNS 登録や ACL に耐える)、
  **ネットワークが変われば別の値**(ネットワーク横断の追跡は不能)、
  そして MAC を漏らさない。**RFC 8064 は新規実装の既定を
  EUI-64 ではなくこの方式にすべき**と勧告しており、現代のクライアント
  OS の既定は概ね「安定不透明 ID + 一時アドレス」の組み合わせである。

結果として典型的なクライアントは、1つのプレフィックスに対して
安定アドレス(受信・登録用)と一時アドレス(発信用)の**2つ以上の
GUA を同時に持つ**。IPv4 の感覚で「端末のアドレスは1つ」と考えて
いると、ログにもフィルタにも穴が開く(トラブルシューティング②)。

### アドレスの一生 — tentative / preferred / deprecated

IPv6 のアドレスには状態と寿命がある(RFC 4862 の用語。
プロセスの詳細は(後述: `03_ndp_slaac.md`))。

```text
 生成 ──▶ tentative ──重複なし──▶ preferred ──preferred_lft 満了──▶ deprecated ──valid_lft 満了──▶ 削除
          (重複検査中。            (通常状態。              (新規の送信元には使わない。
           まだ使えない)            送受信に使う)             既存の接続は継続できる)
```

- **tentative**: 生成直後、同じアドレスを使う者がリンクにいないかの
  重複アドレス検出(DAD、03 章)が終わるまでの状態。
- **preferred**: 通常状態。`ip addr` の `preferred_lft` が残っている間。
- **deprecated**: `preferred_lft` が尽きた状態。既存の接続はそのまま
  使い続けてよいが、**新しい通信の送信元には選ばれない**
  (RFC 6724 規則(3))。一時アドレスの世代交代や、プレフィックスの
  付け替え(リナンバリング)を接続を切らずに行うための猶予状態である。

`ip addr` で常に表示されている `valid_lft` / `preferred_lft` は
この状態機械の残り時間であり、値は RA((後述: `03_ndp_slaac.md`))で
配られ、更新され続けている。

### 要請ノードマルチキャスト — 「全員を起こさない」の実装

前章で「NDP は無関係なノードを起こさない」と述べた。その土台が
**要請ノードマルチキャストアドレス**(solicited-node multicast address、
RFC 4291)である。作り方は機械的で、ユニキャストアドレスの
**下位 24 ビット**を `ff02::1:ff00:0/104` に接合する。

```text
 ユニキャスト:      2001:db8:1:2:5054:ff:fe12:3456
                                          └──┬──┘
                                     下位 24 ビット = 12:34:56
 要請ノード:        ff02::1:ff12:3456
```

各ノードは、自分が持つ**すべての**アドレス(リンクローカル、GUA、
一時アドレス…)について対応する要請ノードグループに参加する。
アドレス解決したい側は、相手のアドレスから同じ計算でグループを求めて
そこへ照会を送る — 受信するのは下位 24 ビットが一致するごく少数
(ほとんどの場合、本人だけ)である。ARP のブロードキャストが
「全端末への割り込み」だったのと対照的に、宛先を知っていることが
そのまま「本人しか聞いていないチャネル」を指すよう設計されている。
照会のプロトコル(近隣要請/近隣広告)は 03 章で扱う。

### L2 へのマッピング(RFC 2464)

IPv6 マルチキャストは Ethernet 上では、宛先 MAC
**33:33 + IPv6 アドレスの下位 32 ビット**のマルチキャストフレームで
運ばれる(RFC 2464)。

```text
 ff02::1:ff12:3456  ──▶  宛先 MAC 33:33:ff:12:34:56
 ff02::1(全ノード)  ──▶  宛先 MAC 33:33:00:00:00:01
```

NIC はハードウェアの受信フィルタで自分の参加グループの MAC だけを
拾うため、「起こさない」はソフトウェアに届く前の段階で実現される。
なお L2 スイッチから見ればこれらは通常のマルチキャストフレームであり、
既定ではフラッディングされる([第1部01章](../01_fundamentals/01_l2_l3_recap.md)) —
帯域まで絞りたい場合に MLD スヌーピング(IGMP スヌーピングの
IPv6 版)が使われる、という関係も押さえておくとよい。

## 設定例 — 複数アドレスを読む(任意)

SLAAC が動いている Linux ホストの典型的な見え方:

```bash
$ ip -6 addr show dev eth0
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 state UP
    inet6 2001:db8:1:2:8f3a:1c22:9d10:77be/64 scope global temporary dynamic
       valid_lft 86375sec preferred_lft 14375sec
    inet6 2001:db8:1:2:5054:ff:fe12:3456/64 scope global dynamic mngtmpaddr
       valid_lft 86375sec preferred_lft 14375sec
    inet6 fe80::5054:ff:fe12:3456/64 scope link
       valid_lft forever preferred_lft forever
```

本章の内容がすべて詰まっている — `scope global` の GUA が2つ
(`temporary` = 一時アドレス。外向き通信の送信元はこちらが選ばれる)、
`scope link` のリンクローカル、そして各行の `valid_lft` /
`preferred_lft`(アドレスの一生の残り時間)。

デフォルト経路のネクストホップがリンクローカルであることも確認できる:

```bash
$ ip -6 route show default
default via fe80::1 dev eth0 proto ra metric 1024 expires 1795sec
```

参加中のマルチキャストグループには、全ノード(ff02::1)と、
保有アドレスごとの要請ノードグループが見える:

```bash
$ ip -6 maddress show dev eth0
2:  eth0
    inet6 ff02::1:ff12:3456
    inet6 ff02::1:ff10:77be
    inet6 ff02::1
```

リンクローカル宛ての操作にはゾーン ID が要る:

```bash
$ ping -c 1 fe80::1          # ゾーン未指定 → エラーになる
$ ping -c 1 fe80::1%eth0     # どのリンクの fe80::1 かを指定
```

## トラブルシューティング

### ① 「ゲートウェイが fe80 で始まっている」は故障ではない

IPv4 の感覚で経路表を見ると、`default via fe80::...` は
「グローバルでないアドレスがゲートウェイになっている、何かおかしい」に
見える。これは**正常**である。RA はルータ自身のリンクローカルアドレスを
送信元として届き、デフォルト経路のネクストホップはそれになる。
ネクストホップに必要なのは「同一リンクで L2 解決できること」だけであり
([第1部02章](../01_fundamentals/02_routing_table_basics.md))、
リンクローカルはその条件を常に満たす。むしろ ISP を替えても
変わらない安定したネクストホップである。
逆に、リンクローカル宛ての ping やソケット接続が
`Invalid argument` 等で失敗するときは、ゾーン ID の指定漏れを疑う —
`fe80::1` だけでは OS はどのインタフェースから出すべきか決められない。

### ② 送信元アドレスが変わり続ける — 一時アドレスと ACL

「特定サーバからの接続だけ許可する IP ベースの ACL が、数日おきに
突然弾き始める」— 典型犯は一時アドレス(RFC 8981)である。
既定の OS は外向き通信の送信元に一時アドレスを選び、それは
1日程度で世代交代する。ログの突合も同様に壊れる(同じ端末の通信が
日ごとに別アドレスに見える)。

- 観察: `ip -6 addr` で `temporary` フラグと `preferred_lft` を見る。
  世代交代の周期と「弾かれ始めた時刻」が一致するはずである。
- 対策の方向: 発信元を固定したいホスト(サーバ、運用端末)では
  一時アドレスを無効化する(Linux では
  `net.ipv6.conf.eth0.use_tempaddr=0`)か、アドレスではなく
  プレフィックス(/64)単位で ACL を書く。クライアントの
  プライバシー保護としては正しい既定であり、「無効化が常に正解」
  ではないことに注意。

### ③ ULA だけ配ったら IPv6 が使われない

「組織内に fd00::/8 を配り、AAAA も登録したのに、通信を見ると
全部 IPv4 で流れている」— これは RFC 6724 の仕様どおりの動作である。
宛先選択の優先度表で ULA(3)は IPv4(35)より低く、
**ULA しか送信元候補がないホストは、AAAA があっても A を選ぶ**。
「ULA = RFC 1918 の置き換え、NAT66 を足せばインターネットにも出られる」
という IPv4 的な設計は、アドレス選択の段階で意図的に排除されている。

- 観察: `ip -6 addr` に `scope global` が ULA しかないことを確認する。
  `curl -6` で明示すれば ULA 発の通信自体は(内部宛てなら)通る。
- 対策の方向: インターネットに IPv6 で出るなら GUA を配る
  (ULA は併存させてよい — 複数アドレスの原則)。組織内通信だけ
  IPv6 化したい場合も、この優先度の帰結(内部宛て = ULA どうしなら
  最長一致で IPv6 が選ばれる)を理解した上で設計する。

### ④ 表記ゆれで grep・監視から漏れる

`2001:db8:0:0:1:0:0:1` を設定に書き、ログには `2001:db8::1:0:0:1` と
出力され、監視ツールには大文字で登録されている — 同じアドレスが
文字列としては三様になり、突合・検索がすべてすり抜ける。
IPv4 には無かった、表記の自由度が生む IPv6 固有の運用問題である。

- 対策の方向: 人間が書く場所(設定、台帳、監視)にも **RFC 5952 の
  正規形**(小文字・最長ゼロ列を `::`)を規約として強制する。
  アドレスを扱うスクリプトでは文字列比較を禁じ、パースして比較する
  (Python の `ipaddress` モジュール等)。tcpdump や FRR などの
  出力は基本的に RFC 5952 準拠なので、「機械の出力に人間が合わせる」
  と覚えるのが早い。

## 演習・確認問題

1. `2001:0db8:0000:0000:0001:0000:0000:0001` の RFC 5952 正規形を示せ。
   また `2001:db8:0:0:1::1` という表記が正規形でない理由を
   規則に照らして説明せよ。
2. IPv6 のインタフェースが同時に持ちうるアドレスを3種類以上挙げ、
   それぞれのスコープと役割を説明せよ。「ホストの IP アドレスは
   1つ」という IPv4 的前提が IPv6 で成り立たない理由をまとめよ。
3. /48 を割り当てられた組織がサブネットに使える数を示せ。また
   「ホスト数からのプレフィックス長の逆算」が IPv6 の設計で
   消滅する理由と、それでもルータ間リンクに /127 を使う理由
   (RFC 6164)を説明せよ。
4. リンクローカルアドレスにゾーン ID(%eth0)が必要な理由を、
   GUA には不要な理由と対比して説明せよ。
5. ULA のグローバル ID を乱数で生成する設計意図を、RFC 1918 の
   アドレス重複問題と対比して説明せよ。また ULA しか持たない
   デュアルスタックホストが IPv4 を優先する理由を RFC 6724 の
   優先度表から説明せよ。
6. MAC アドレス `00:0c:29:aa:bb:cc` から Modified EUI-64 の IID と
   リンクローカルアドレスを導出せよ。この方式が抱える追跡問題と、
   RFC 8981・RFC 7217 がそれぞれ何を解決するかを述べよ。
7. アドレス `2001:db8::abc:def0` に対応する要請ノードマルチキャスト
   アドレスと、それが Ethernet 上で使う宛先 MAC アドレスを導出せよ。
   この仕組みが ARP のブロードキャストと比べて「誰を起こすか」を
   どう変えるか説明せよ。

## まとめ

- IPv6 アドレスは 16 ビット×8 グループの16進表記で、省略規則により
  同一アドレスに複数の合法表記がある。機械出力の正規形は RFC 5952 —
  運用では文字列比較を避け、正規化して扱う。
- 空間の地図: GUA は 2000::/3、リンクローカルは fe80::/10、ULA は
  fd00::/8、マルチキャストは ff00::/8。ブロードキャストは存在しない。
  GUA は「ルーティングプレフィックス+サブネット ID+64 ビット IID」の
  階層構造で、割り当ては経路集約が構造的に効くよう配られる。
  サブネットは原則 /64(ルータ間は /127)。
- IPv6 では届く範囲(スコープ)と寿命(preferred/deprecated)が
  アドレスに内蔵され、1インタフェースが複数のアドレスを持つのが
  正常状態。どれを使うかは RFC 6724 のアドレス選択が決める —
  ULA が IPv4 より優先されないのは仕様である。
- IID の生成は EUI-64(MAC 由来、追跡問題)から、一時アドレス
  (RFC 8981)+安定不透明 ID(RFC 7217、RFC 8064 が既定に勧告)へ
  移った。要請ノードマルチキャスト(下位 24 ビット接合)が
  「全員を起こさない」アドレス解決の土台であり、その上で動く
  NDP・SLAAC を次章で扱う。
