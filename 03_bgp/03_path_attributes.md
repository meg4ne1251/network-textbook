# パスアトリビュート — BGP は経路をどう選ぶか

## 概要

この章では、BGP の UPDATE で経路とともに運ばれる**パスアトリビュート**の意味論と、
それらを材料に最良経路を決める**経路選択プロセス(Decision Process)**を扱う。
前提知識は [BGP の基礎の章](01_bgp_basics.md) の UPDATE の構造(NLRI + 属性一式)と
3段の RIB、[iBGP と eBGP の章](02_ibgp_ebgp.md) の広告規則
(AS_PATH 前置・NEXT_HOP 書き換え・LOCAL_PREF と MED のスコープの予告)である。

## 導入 — 経路は「宛先」ではなく「属性の束」である

IGP の経路は素朴だった。宛先プレフィックスと、メトリックという1個の数値。
経路選択は「数値の最小化」であり、そこに解釈の余地はない
([第1部](../01_fundamentals/02_routing_table_basics.md) の3段フィルタの①である)。

BGP の経路は違う。1本の経路は**プレフィックスに属性の束が貼り付いたもの**であり、
「どこから来たか(AS_PATH)」「誰に投げればよいか(NEXT_HOP)」
「組織としてどれくらい好むか(LOCAL_PREF)」「隣の AS はどの入口を勧めているか
(MED)」といった、性質の異なる情報が1本の経路に同居している。
いわば経路の**履歴書**である。

なぜこんな構造が要るのか。[BGP の基礎の章](01_bgp_basics.md) の3要求を思い出そう。
AS 間の経路選択は距離の最小化ではなく**契約と方針**で決まる。
方針とは「顧客経由を優先する」「AS X は避ける」「入口はこちらにしてほしい」といった、
互いに独立な複数の判断基準の重ね合わせである。単一の数値には畳み込めない。
だから BGP は、判断材料を**属性として個別に運び、決まった順序で比較する**という
設計を採った。属性には2つの顔がある:

- **経路選択の材料** — 属性を決まった順序で比較して、最良の1本を決める
- **ポリシーの荷札** — 経路に印を付け、境界で書き換え、遠くの判断を操作する

本章はこのうち「属性の意味」と「比較の順序」を扱う。
属性を**書き換える**道具立て(route-map、prefix-list、コミュニティ)は
[次章](04_policy_control.md) の主題である。本章が意味論、次章が操作法という分担になる。

## 理論

### 属性はどう運ばれるか — TLV と4つの分類

パスアトリビュートは UPDATE の Path Attributes 部に、
**型・長さ・値(TLV)**の並びとして格納される(RFC 4271 Section 4.3):

```text
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Attr. Flags  |Attr. Type Code|   ← 属性型 = フラグ(1) + 型番号(1)
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Length(1 または 2 オクテット)|
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+
|  Value(可変長)...
+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+

Attr. Flags:
 0 1 2 3 4 5 6 7
+-+-+-+-+-+-+-+-+
|O|T|P|E|0 0 0 0|
 │ │ │ └─ Extended Length: Length フィールドを 2 オクテットにする
 │ │ └─── Partial: 経路上の誰かがこの属性を理解しないまま中継した印
 │ └───── Transitive: 理解できない場合でも隣へ引き継ぐべきか
 └─────── Optional: 0 = well-known(全実装必須)/ 1 = optional(拡張)
```

フラグの Optional / Transitive の組み合わせで、属性は4つに分類される
(RFC 4271 Section 5):

| 分類 | 意味 | 代表例 |
|---|---|---|
| well-known mandatory | 全実装が理解し、全 UPDATE に必須 | ORIGIN、AS_PATH、NEXT_HOP |
| well-known discretionary | 全実装が理解するが、有無は状況次第 | LOCAL_PREF、ATOMIC_AGGREGATE |
| optional transitive | 拡張。理解できなくても**中継する** | COMMUNITY、AGGREGATOR、拡張コミュニティ |
| optional non-transitive | 拡張。理解できなければ**捨てる** | MED、MP_REACH_NLRI |

この分類の急所は「**知らない属性を受け取ったらどうするか**」まで
仕様が決めていることである(RFC 4271 Section 9):

- 未知の well-known 属性 → エラー(そもそも起こってはならない)
- 未知の optional transitive 属性 → **Partial ビットを立てて、そのまま隣へ運ぶ**
- 未知の optional non-transitive 属性 → 静かに無視し、隣へは渡さない

つまり BGP は、**古い実装が新しい属性の運び屋になれる**ように最初から
設計されている。コミュニティも拡張コミュニティも 32 ビット AS 番号対応も、
既存のインターネットを止めずに導入できたのはこの仕組みのおかげである。
[ケイパビリティ交渉](01_bgp_basics.md)がセッション単位の拡張性なら、
こちらは経路情報単位の拡張性である
(ただし「中身を検査せず運ぶ」ことの代償が後年問題になる。
トラブルシューティングの節で再訪する)。

### 属性カタログ — 必須の3つ

**ORIGIN(Type 1、well-known mandatory)** は、経路情報が最初に BGP に
入ったときの**由来**を示す1オクテットである。値は3つ:
IGP(0)= 明示的な設定(network 文など)で注入された、
EGP(1)= 旧 EGP プロトコル(RFC 904)由来(歴史的遺物であり現在は使われない)、
INCOMPLETE(2)= それ以外([再配送](../01_fundamentals/03_static_vs_dynamic.md)が典型)。
経路選択では値の小さいほう(IGP < EGP < INCOMPLETE)が好まれる。
「胸を張って広告された経路のほうが、どこかで拾われてきた経路より信用できる」
という程度の、弱い品質表示である。

**AS_PATH(Type 2、well-known mandatory)** は
[BGP の基礎の章](01_bgp_basics.md) で見たパスベクタの本体である。
実際のフォーマットは AS 番号の単純な列ではなく、**セグメント**
(型 1 オクテット + AS 数 1 オクテット + AS 番号列)の並びになっている。
セグメント型は2つ:

- **AS_SEQUENCE**: 通過順に並んだ AS 番号列。通常の経路はこれ1本である
- **AS_SET**: 順序を持たない AS 番号の集合。[経路集約](../01_fundamentals/02_routing_table_basics.md)
  の BGP 版で複数経路を1本に畳んだとき、失われる AS_PATH 情報を
  「この集約にはこれらの AS が混ざっている」という集合として残すためにある。
  ただし集約に AS_SET を使う運用は現在は推奨されない(RFC 6472 / BCP 172)。
  ループ検出と集約の両立には後述の ATOMIC_AGGREGATE を使う

AS_PATH は書き換えの対象でもある。自 AS 番号を**複数回**積んで広告する
**AS_PATH プリペンド**は、経路をわざと長く見せて他 AS からの流入を
別経路へ誘導する、最も広く使われた(そして最も過信されがちな)
トラフィック制御である。効果の限界は動作の節で扱う。

なお 32 ビット AS 番号(RFC 6793)への移行では、この属性自体が
optional transitive の拡張の実例になった。旧実装とのセッションでは
AS_PATH に AS_TRANS(23456)を置き、本当の 4 オクテット AS 番号列を
**AS4_PATH** という optional transitive 属性で別送する。
旧実装は意味を知らないまま AS4_PATH を中継し、その先の新実装が
両者を突き合わせて本来の道筋を復元する — 「古い実装が新しい属性の
運び屋になる」設計が実際に世界規模の移行を支えた例である。

**NEXT_HOP(Type 3、well-known mandatory)** の意味論は
[前章](02_ibgp_ebgp.md) で扱い尽くした(eBGP で書き換え、iBGP で保持、
next-hop-self)。1点だけ予告を足すと、この属性が運べるのは IPv4 アドレス
1個だけであり、IPv6 経路や EVPN 経路のネクストホップは MP-BGP の
**MP_REACH_NLRI 属性(Type 14、RFC 4760)**の中へ移設されている。
詳細は [05 章](05_mp_bgp.md) で扱う。

### 属性カタログ — 選択を駆動する2つ

**LOCAL_PREF(Type 5、well-known discretionary)** は
「**この経路を AS としてどれくらい好むか**」を表す 4 オクテットの度数で、
大きいほど好まれる。決定的に重要なのはスコープである:
iBGP ピアへの広告には**必ず付け**、eBGP ピアへは**付けない**
(RFC 4271 Section 5.1.5。受け取っても無視する)。
つまり LOCAL_PREF は **AS の外に一歩も出ない、組織内部の意思統一の道具**である。

なぜ「内部専用」なのか。どの経路を好むかは各 AS の契約と経済の問題であり、
隣の AS に指図される筋合いのものではないからである。
境界ルータが入力ポリシーで「この経路は顧客経由だから 200」「これは高価な
回線だから 50」と値付けし、iBGP がその値を AS 全体へ運ぶ。
結果、**AS 内の全ルータが同じ物差しで出口を選ぶ**ようになる。
既定値は 100 が実装慣例である(RFC 4271 は既定値を定めていない)。

**MED(MULTI_EXIT_DISC、Type 4、optional non-transitive)** は逆向きの道具である。
自 AS と隣接 AS が**複数の地点で接続している**とき、
「入ってくるならこちらの入口を使ってほしい」という希望を隣接 AS へ伝える
4 オクテットの値で、**小さいほど好まれる**。

スコープは LOCAL_PREF と対をなす。eBGP で隣接 AS に渡り、
その AS の中では iBGP で運ばれるが、**さらに先の AS へは伝搬してはならない**
(RFC 4271 Section 5.1.4)。入口の希望は隣どうしの間でだけ意味を持つ
ローカルな調整事項であり、世界に知らせることではないからである。

さらに重要な但し書きがある。MED は既定で
**同一の隣接 AS から受けた経路どうしでしか比較されない**。
AS X が言う「MED 10」と AS Y が言う「MED 20」は、
別々の組織が別々の基準で付けた値であり、比較に意味がないからである
(メトリックがプロトコル間で比較できないのと同じ理屈である)。
この制約が MED を「効いているのかどうか分かりにくい属性」の代表にしている
(トラブルシューティングの節で再訪する)。

### 属性カタログ — 集約の痕跡と、次章以降への予告

**ATOMIC_AGGREGATE(Type 6、well-known discretionary)** は値を持たないフラグで、
「この経路は集約であり、より詳細な経路情報(AS_PATH の一部)が
失われている」ことを下流に知らせる。
**AGGREGATOR(Type 7、optional transitive)** は集約を行った
AS 番号とルータの BGP Identifier を記録する。どちらも経路選択には関与しない、
集約の**出所表示**である。

残る主要な属性は、それぞれ専用の章で扱う:

- **COMMUNITY(Type 8、RFC 1997)** と**拡張コミュニティ(Type 16、RFC 4360)**、
  **ラージコミュニティ(Type 32、RFC 8092)** — 経路に貼る「荷札」。
  [第2部で使った RT(ルートターゲット)](../02_vlan_vxlan_evpn/05_evpn_vxlan.md) の
  正体は拡張コミュニティの一種である。詳細は [04 章](04_policy_control.md)
- **MP_REACH_NLRI / MP_UNREACH_NLRI(Type 14 / 15、RFC 4760)** —
  MP-BGP の本体。詳細は [05 章](05_mp_bgp.md)
- **ORIGINATOR_ID / CLUSTER_LIST(Type 9 / 10、RFC 4456)** —
  ルートリフレクタのループ防止。詳細は [06 章](06_large_scale_design.md)

### スコープの設計 — 属性はどこまで届くか

ここまでの属性を「届く範囲」で並べ直すと、BGP の設計思想が一枚で見える:

| 属性 | 届く範囲 | 役割 |
|---|---|---|
| LOCAL_PREF | 自 AS の中だけ | 出口選択の意思統一 |
| MED | 隣接 AS まで | 入口の希望の伝達 |
| AS_PATH・ORIGIN | 全世界(境界ごとに加工) | ループ検出・道筋と由来の共有 |
| COMMUNITY 類 | 貼った者と了解した者の間 | 合意ベースの荷札(04章) |

**意思決定は内に閉じ、希望は隣まで、事実は世界へ。**
[前章](02_ibgp_ebgp.md) の対比表で予告した「スコープの違いがポリシー設計の
道具立てになる」の中身がこれである。範囲の異なる属性を重ねることで、
「世界には事実だけを見せ、判断は自分で下し、隣とは希望を交換する」という
AS 間関係の作法がプロトコルに焼き込まれている。

## プロトコル動作の詳細

### 経路選択プロセス — 3つのフェーズ

同じプレフィックスへの経路を複数のピアから受け取ったとき、
Loc-RIB に載せる1本を決めるのが**経路選択プロセス(Decision Process、
RFC 4271 Section 9.1)**である。全体は3フェーズに分かれる:

1. **Phase 1 — 度数の計算**: 受信した経路ごとに「好みの度合い
   (degree of preference)」を計算する。iBGP から受けた経路は運ばれてきた
   LOCAL_PREF をそのまま使い、eBGP から受けた経路は自分の入力ポリシーで
   値付けする(その結果が iBGP へ広告するときの LOCAL_PREF になる)
2. **Phase 2 — 最良経路の選択**: 宛先プレフィックスごとに、度数最大の経路を
   選ぶ。同点なら後述の多段の比較(tie-breaking)で1本まで絞り、Loc-RIB へ載せる
3. **Phase 3 — 配布**: Loc-RIB の経路を出力ポリシーに通し、
   Adj-RIB-Out を経て各ピアへ広告する(集約もここで行われる)

比較の前提を2つ確認しておく。第一に、比較されるのは**同一プレフィックス**の
経路どうしだけである(203.0.113.0/24 と 203.0.113.0/23 は別の宛先であり、
両方が Loc-RIB に載る。どちらでパケットが転送されるかは
[第1部の3段フィルタ](../01_fundamentals/02_routing_table_basics.md) の③、
転送時のロンゲストマッチの仕事である)。第二に、そもそも**無効な経路は
土俵に上がらない**。ネクストホップが再帰解決できない経路
([前章](02_ibgp_ebgp.md))や、AS_PATH に自 AS を含む経路
(ループ検出)は Phase 1 の前に除外されている。

### 多段の比較 — 誰の意見がどの順で聞かれるか

Phase 2 の同点決勝は、RFC 4271 Section 9.1.2.2 では次の順序である。
**上から順に比較し、差がついた時点で終了する**:

| 段 | 比較する材料 | 勝つ値 | 意味 |
|---|---|---|---|
| 1 | LOCAL_PREF(度数) | 大 | **自 AS の意思** |
| 2 | AS_PATH の AS 数 | 小 | 世界の形(道筋の短さ) |
| 3 | ORIGIN | 小(IGP < EGP < INCOMPLETE) | 由来の確かさ |
| 4 | MED(同一隣接 AS 間のみ) | 小 | **隣接 AS の希望** |
| 5 | eBGP 学習 > iBGP 学習 | — | 出口は近いほうがよい |
| 6 | ネクストホップへの IGP コスト | 小 | 自分からの近さ |
| 7 | BGP Identifier | 小 | 決定性の担保 |
| 8 | ピアのアドレス | 小 | 最後の決着 |

この順序は恣意的な仕様ではなく、**意見の優先順位の宣言**として読める。
最初に聞かれるのは自組織のポリシー(LOCAL_PREF)であり、
契約と経済の判断が他のすべてに優先する。次が世界の客観的な形(AS_PATH 長)、
その次が相手の希望(MED)。**自分の意思 > 世界の事実 > 隣の希望**という
序列が、そのまま比較の段になっている。

段5と段6は「出口までの近さ」の連続した表現である。
iBGP 経路より eBGP 経路が勝つのは「他の境界ルータまで AS 内を運ぶより、
自分の目の前の境界から出すほうが早い」からであり、
段6はその iBGP 版 —「複数の境界が同点なら、IGP 的に一番近い境界から出す」
— である。自 AS 内の運搬距離を最小化するこの振る舞いは
**ホットポテトルーティング**(熱い芋は早く手放す)と呼ばれ、
トランジット AS の既定の行動原理になっている。

段7・8に意味論はない。ここまで全部同点だった経路の間で、
**どのルータで計算しても同じ答えになる**ことだけを保証する決着装置である
(比較に一意な順序があるからこそ、BGP は分散計算でも決定的でいられる)。

細部の規定を2つ補足する。AS_PATH の AS 数の勘定では
AS_SET は中身の個数によらず**1**と数える。また MED を持たない経路は
**最小値(0)を持つものとして扱う**、つまり最も好まれる側に倒すのが
RFC 4271 の規定である。

### 実装の現実 — 表にない2つの段

実装(Cisco 系、FRR など)の比較リストには、RFC にない段が挟まっている。
[管理距離](../01_fundamentals/02_routing_table_basics.md) と同様、
「RFC 非標準だが実務で必ず出会う慣例」として押さえておく:

- **Weight(段1の前)**: ルータローカルの優先度。属性ではないので
  **一切広告されず**、そのルータの上でだけ効く。LOCAL_PREF より先に
  比較される最強の値であり、「この1台だけ例外的な選択をさせる」ための
  非常口である
- **古い経路の優先(段7の前)**: eBGP 経路どうしなら、BGP Identifier の
  比較へ進む前に「先に受け取っていた経路」を維持する実装が多い。
  経路のフラップのたびにベストが揺れて転送が波打つのを防ぐ安定化だが、
  「再起動したら選択が変わった」「2台で選択が食い違う」という
  **再現性のなさ**の源でもある(トラブルシューティングで再訪する)

### ウォークスルー — 2本の経路、2つの結末

構成を用意する。自 AS 64500 は2つの上流 AS に接続している
(マルチホーム)。宛先 203.0.113.0/24 は AS 64511 が生成する:

```text
              ┌── AS 64496(上流 A)──────────────┐
       eBGP ┌─ R1                                  ├─ AS 64511
 AS 64500   │  │ iBGP(フルメッシュ)              │   203.0.113.0/24 を生成
       eBGP └─ R2 ── AS 64510 ── AS 64497 ────────┘
              (上流 B)
```

AS 内のルータ R3(iBGP のみ)には、同じ宛先への経路が2本届く
(R1・R2 とも next-hop-self 設定済みとする):

| | 経路 A(R1 経由) | 経路 B(R2 経由) |
|---|---|---|
| AS_PATH | 64496 64511(長さ 2) | 64510 64497 64511(長さ 3) |
| NEXT_HOP | R1 のループバック | R2 のループバック |
| LOCAL_PREF | 100(既定) | 100(既定) |

**結末1(ポリシーなし)**: 段1は同点(100 対 100)。段2で AS_PATH 長 2 対 3、
**経路 A の勝ち**。R3 だけでなく AS 内の全ルータが同じ計算で A を選び、
外向きトラフィックはすべて R1 から出ていく。

**結末2(ポリシーあり)**: 上流 B のほうが回線が安いので、平常時は B を
使いたいとする。R2 の入力ポリシーで、AS 64510 から受ける経路に
LOCAL_PREF 200 を付ける。すると段1で 100 対 200、**経路 B の勝ち**。
AS_PATH が1ホップ長いことは、もう誰にも顧みられない —
比較は差がついた段で終わるからである。

これが「意思は形に勝つ」の実演である。LOCAL_PREF が比較の先頭にある以上、
**AS_PATH 長は「誰もポリシーを設定しなかったときの既定の良識」**にすぎない。
インターネットの経路選択は最短経路探索ではなく、
各 AS のポリシーの重ね合わせであり、その力関係が比較の順序に刻まれている。

### 入口の制御 — 広告の向きとトラフィックの向き

ウォークスルーで動かしたのは**出ていく**トラフィック(egress)である。
出口の選択は自分の Loc-RIB の勝敗だから、LOCAL_PREF で完全に支配できる。
では**入ってくる**トラフィック(ingress)はどう制御するか。

原理を1つ確認する。**トラフィックは、経路広告が流れた向きと逆向きに流れ込む**。
自分のプレフィックスの広告が届いた先の AS が、その広告(が勝ち残った経路)に
向かってパケットを送ってくるからである。つまり入口の制御とは、
**自分が出す広告の細工**であり、最終判断は常に相手側の経路選択プロセスが下す。
自分にできることは強い順に:

1. **広告しない** — その経路を広告しなかった出口には、原理的に入ってこない
   (最強。ただし経路を絞るほど冗長性も失う)
2. **MED を付ける** — 隣接 AS に入口の希望を数値で伝える。
   ただし届くのは隣までで、比較されるのは段4 — 相手の LOCAL_PREF と
   AS_PATH 長が同点だったときだけである
3. **AS_PATH プリペンドする** — 劣らせたい出口からの広告に自 AS を
   複数回積み、道筋を長く見せる。遠くの AS にも伝わる唯一の細工だが、
   効くのは段2 — 相手やその先の AS が LOCAL_PREF で
   判断していれば届かない
4. **コミュニティで依頼する** — 「この経路の LOCAL_PREF を下げてほしい」
   といった依頼を荷札として送り、相手のポリシーに実行してもらう
   (相手がその荷札の意味を公開・実装している場合に限る。04 章で扱う)

段1(相手の意思)に割り込む手段は 4 の「お願い」しかないことに注意してほしい。
**出口は命令できるが、入口は頼むしかない** — これは BGP の欠陥ではなく、
各 AS の自律(まさに Autonomous System)という前提の帰結である。

## 設定例(補助)

ウォークスルーの R3 で経路の属性と選択理由を観察する。
FRRouting では `show bgp ipv4 unicast <プレフィックス>` が属性の一覧を表示する:

```text
router# show bgp ipv4 unicast 203.0.113.0/24
BGP routing table entry for 203.0.113.0/24, version 42
Paths: (2 available, best #2, table default)
  64510 64497 64511
    10.255.0.2 (metric 20) from 10.255.0.2 (10.255.0.2)
      Origin IGP, metric 0, localpref 100, valid, internal
      Last update: Mon Jul  6 10:02:33 2026
  64496 64511
    10.255.0.1 (metric 10) from 10.255.0.1 (10.255.0.1)
      Origin IGP, metric 0, localpref 100, valid, internal, best (AS Path)
      Last update: Mon Jul  6 10:01:12 2026
```

読み方はそのまま本章の理論の確認になる。各ブロックの1行目が AS_PATH、
2行目がネクストホップと(括弧内に)そこまでの IGP コスト(段6の材料)、
3行目に ORIGIN・MED(metric 表記)・LOCAL_PREF が並ぶ。
そして best の後ろの **(AS Path)** が「どの段で勝負がついたか」の表示である。
R2 で LOCAL_PREF 200 を付ける結末2を実施すると、この表示が
best (Local Pref) に変わり、勝者が入れ替わる — 比較の段が
そのまま出力に現れるので、選択理由の切り分けに直結する。

LOCAL_PREF の値付けそのものは route-map(次章)の仕事だが、
形だけ先に見せておく:

```text
route-map FROM-UPSTREAM-B permit 10
 set local-preference 200
!
router bgp 64500
 address-family ipv4 unicast
  neighbor 198.51.100.9 route-map FROM-UPSTREAM-B in
 exit-address-family
```

「eBGP から受けた経路に入力ポリシーで度数を付ける」という
Phase 1 の記述が、設定の形にそのまま対応している。

## トラブルシューティング

### 症状: AS_PATH プリペンドしたのに流入が変わらない

- プリペンドは段2(AS_PATH 長)への細工である。トラフィックを送ってくる
  AS が段1(LOCAL_PREF)で判断している — 例えば「顧客から学んだ経路は
  無条件に最優先」というポリシーは商用 ISP のほぼ普遍的な慣行である —
  なら、段2は永遠に比較されず、何回積んでも効かない
- 切り分けは「相手側でどう見えているか」の確認からになる。多くの ISP や
  IX が提供するルッキンググラス(外部から経路表を覗ける公開窓口)で
  自分のプレフィックスを引き、どの経路が best に選ばれ、
  LOCAL_PREF に何が付いているかを見る。相手のポリシーが原因なら、
  打つ手はプリペンドの増量ではなく、コミュニティによる依頼(04 章)か
  広告の停止である

### 症状: MED を付けたのに入口が変わらない

- まず段4に届いているか。相手 AS の中で LOCAL_PREF・AS_PATH 長・ORIGIN が
  同点になっていなければ MED は見られもしない
- 次に**比較の相手**。MED が比較されるのは同一の隣接 AS から受けた経路
  どうしだけである。相手 AS があなたと直接、複数地点でピアしているなら
  比較されるが、あなたへの経路を「あなたから直接」と「別の AS 経由」の
  2通りで持っている場合、その2本の間で MED は既定では比較されない
- 最後に**実装オプションの不一致**。異なる AS 間でも MED を比較する
  always-compare-med、MED なしを最悪値として扱う missing-as-worst など、
  MED の解釈を変えるノブが各実装にあり、両側の想定が揃っていないと
  「こちらの計算では効くはず」が成立しない。MED は関係者の設定を
  揃えて初めて機能する、調整コストの高い属性である

### 症状: 同じ経路を受けている2台で、選択がなぜか食い違う

- 故障とは限らない。段6(ネクストホップへの IGP コスト)は**ルータごとに
  値が違って当然**の材料であり、置かれた場所によって最寄りの出口が
  違うだけかもしれない(ホットポテトの正常動作)
- IGP コストまで同点なのに食い違う場合は、実装の「古い経路の優先」を疑う。
  受信した**順序**が2台で違えば選択も違いうるし、セッションを
  リセットした側だけ選択が変わる、という再現性のない挙動になる。
  比較を BGP Identifier まで確実に進めて決定性を取り戻すノブ
  (FRR・Cisco 系の bgp bestpath compare-routerid)を使うか、
  そもそも上流の段(LOCAL_PREF 等)で意図的に差を付けて、
  たまたまの同点に選択を委ねない設計にする
- MED が絡む同点は特に厄介である。「同一隣接 AS 間のみ比較」という
  規則のため、候補が3本以上あると比較の組み合わせ順で結果が変わりうる
  (これを是正する deterministic-med 系のオプションも実装にはある)。
  「MED は隣との2者間調整に限って使う」と割り切るのが安全である

### 症状: 特定の UPDATE を受けるとセッションが切れる/経路が消える

- 属性の構文エラーである。RFC 4271 の原則では UPDATE の属性に不正を
  見つけたら NOTIFICATION(UPDATE Message Error)を送ってセッションを
  切る — しかしこの規則は「未知の optional transitive 属性は検査せず運ぶ」
  設計と組み合わさると悲劇を生む。どこか遠くの実装が生成した壊れた属性が、
  中継者たちには検査されないまま世界を巡り、**それを理解する実装に
  届いたところで一斉にセッションを落とす**のである(実際に地域規模の
  障害として複数回発生している)
- このため RFC 7606 が扱いを改訂した。可能な限りセッションは維持し、
  問題の経路だけを**撤回として扱う**(treat-as-withdraw)。
  現代の実装は概ねこれに従うため、症状は「セッション断」から
  「特定の経路だけ現れない」に変わっている。ログの属性エラーの記録と、
  tcpdump/Wireshark での UPDATE の属性デコード(Wireshark は属性を
  型名まで展開して表示する)が切り分けの道具になる

## 演習・確認問題

1. パスアトリビュートの4分類を挙げ、「未知の属性を受け取ったときの動作」が
   分類ごとにどう規定されているかを述べよ。またこの規定が BGP の拡張性に
   果たした役割を、AS4_PATH を例に説明せよ。
2. LOCAL_PREF と MED について、値の向き(大小どちらが好まれるか)、
   届く範囲、想定された用途をそれぞれ対比して述べよ。
3. LOCAL_PREF が eBGP ピアへ広告されないのはなぜか。「AS の自律」の
   観点から説明せよ。
4. 経路選択プロセスの比較で、LOCAL_PREF が AS_PATH 長より先に
   評価されることの設計上の意味を述べよ。またその帰結として、
   AS_PATH プリペンドが効かないのはどのような場合か。
5. 「MED は同一の隣接 AS から受けた経路どうしでしか比較されない」という
   規則の理由を、メトリックがプロトコル間で比較できない理由
   (第1部)との類推で説明せよ。
6. ホットポテトルーティングとは何か。経路選択プロセスのどの段が
   それを実現しているかを2つ挙げよ。
7. 比較の最後に BGP Identifier という「意味のない値」が置かれているのは
   なぜか。分散計算の決定性という観点から述べよ。また実装の
   「古い経路の優先」がこの決定性をどう損なうかを説明せよ。
8. 自 AS のプレフィックスについて、入ってくるトラフィックの経路を
   制御したい。取りうる手段を効果の強い順に4つ挙げ、それぞれの限界を
   1行ずつ述べよ。

## まとめ

- BGP の経路はプレフィックスと属性の束であり、属性は「選択の材料」と
  「ポリシーの荷札」を兼ねる。4分類(well-known / optional ×
  transitive / non-transitive)は未知の属性の扱いまで規定した
  後方互換な拡張の仕組みである
- スコープが設計の鍵である: LOCAL_PREF は AS 内(意思)、MED は
  隣接 AS まで(希望)、AS_PATH・ORIGIN は全世界(事実)
- 経路選択プロセス(RFC 4271 Section 9.1)は LOCAL_PREF → AS_PATH 長 →
  ORIGIN → MED → eBGP/iBGP → IGP コスト → BGP Identifier の多段比較で、
  差がついた段で終わる。順序は「自分の意思 > 世界の事実 > 隣の希望」の宣言である
- 出口は LOCAL_PREF で命令できるが、入口は相手の選択の材料
  (広告の有無・MED・プリペンド・コミュニティ)を差し出して頼むしかない
- Weight・古い経路の優先・MED の解釈ノブは RFC 外の実装慣例であり、
  「なぜこの経路が選ばれたか」の再現性を崩す典型でもある。
  属性を書き換える道具立ては次章で扱う
