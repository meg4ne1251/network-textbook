# BGP の大規模設計 — iBGP フルメッシュを卒業する

## 概要

この章では、iBGP フルメッシュの規模の破綻を救う2つの構造 —
**ルートリフレクタ(RFC 4456)**と**コンフェデレーション(RFC 5065)** —
を扱い、あわせて現代のデータセンターにおける BGP 設計(RFC 7938、
EVPN ファブリックの経路配布)を見る。第3部の最終章である。前提知識は
[02 章](02_ibgp_ebgp.md)(iBGP の再広告禁止とフルメッシュ)と
[03 章](03_path_attributes.md)(経路選択プロセス)。
[第2部で保留にした「スパインをルートリフレクタにする構成」](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)
の正体も、この章で明らかになる。

## 導入 — 規則は正しいのに、構成が持たない

### n(n-1)/2 の再演

[02 章](02_ibgp_ebgp.md)の結論を思い出そう。iBGP には
「iBGP から学んだ経路を iBGP へ再広告してはならない」という規則
(RFC 4271 Section 9.2)があり、AS 内の全スピーカーに経路を行き渡らせるには
**全員が全員と直接ピアを張る**しかなかった。セッション数は n(n-1)/2 —
50 台で 1225 本、100 台で 4950 本。しかも1台の追加が既存全台の設定変更を
要求する。

この形の破綻には、本書ですでに2回出会っている。
[静的ルーティングが規模で破綻するパターン](../01_fundamentals/03_static_vs_dynamic.md)と、
[ヘッドエンドレプリケーションの静的リスト](../02_vlan_vxlan_evpn/04_vxlan_control_plane.md)である。
そしてどちらのときも、解決の方向は「構成の全体像を知る仕事を、
構造の工夫で局所化する」ことだった。iBGP でも同じことをしたい。

### 何を壊さずに、何を緩めるか

ただし、再広告禁止の規則は気まぐれな制限ではない。
[02 章で導出したとおり](02_ibgp_ebgp.md)、iBGP の広告では AS_PATH に
自 AS が積まれないため、**AS の内側にはループを検出する情報が存在しない**。
再広告を禁止し「経路は発信者から直接1ホップで聞く」形に限定することが、
そのままループ防止だったのである。

だから、規則を緩めて「又聞き」を許すなら、**代わりのループ検出を
持ち込まなければならない**。これから見る2つの解は、どちらも同じ原理 —
[第1部で学んだパスベクタ](../01_fundamentals/04_distance_vector_link_state.md)、
すなわち「通ってきた道筋を情報自体に載せ、自分の印を見つけたら捨てる」—
を AS の**内側に再輸入**する。

- **ルートリフレクタ**: 中継者の印(ORIGINATOR_ID / CLUSTER_LIST)を
  経路に積む。AS の中に「ミニ AS_PATH」を作る
- **コンフェデレーション**: AS の中に小さな AS を作り、
  AS_PATH の仕組みそのものを内部で使う(AS_CONFED_SEQUENCE)

## 理論

### ルートリフレクタ(RFC 4456)

#### 発想 — 禁止規則に、例外を1つ開ける

ルートリフレクタ(Route Reflector、以下 RR)の考え方は率直である。
再広告禁止の規則はそのままに、**指定された特別なルータだけは、
iBGP から学んだ経路を iBGP へ再広告(反射、reflect)してよい**とする。

RR に経路を反射してもらうルータを**クライアント**と呼び、
RR と配下のクライアントの集合を**クラスタ**と呼ぶ。クライアントは
RR とだけピアを張ればよく、クライアント同士のフルメッシュは不要になる。
セッション数は n(n-1)/2 から「クライアント数 × RR 数」のオーダーへ落ち、
ルータの追加は「新ルータと RR の間にセッションを張る」だけの
局所的な作業になる。

RFC 4456 の設計で決定的に巧いのは、**クライアント側に一切の変更が
要らない**ことである。クライアントは自分がクライアントであることすら
知らなくてよく、ただの iBGP スピーカーとして振る舞う。RR かどうかは
RR 側の設定だけで決まり、既存の AS に1台ずつ漸進的に導入できる。
この展開の軽さが、後述するコンフェデレーションではなく RR が
実務の主流になった最大の理由である。

#### 反射の3規則

RR は、最良経路(Loc-RIB に載せた経路)を、学んだ相手の種別に応じて
次の規則で配る(RFC 4456 Section 6):

| 経路を学んだ相手 | 配る先 |
|---|---|
| eBGP ピア | すべてのクライアントと非クライアント(通常の BGP と同じ) |
| 非クライアントの iBGP ピア | クライアントへ反射する |
| クライアントの iBGP ピア | すべての非クライアントと、(発信者を除く)他のクライアントへ反射する |

つまり再広告禁止が破られるのは下2行 — RR がクライアントとの間で
経路を中継するときだけである。非クライアント同士の関係は従来のままなので、
**RR と非クライアント(および RR 同士)は依然フルメッシュが必要**である。
AS 全体としては「フルメッシュを組むのは少数の RR だけ、
多数のクライアントはクラスタにぶら下がる」という2階建てになる。
RR がさらに増えて RR 間フルメッシュが苦しくなれば、RR を別の RR の
クライアントにする**階層化**もできる。

#### 又聞きの代償① — ループと、その検出

反射は「又聞き」だから、経路が巡って発信者や中継者に戻る可能性が生まれる。
RFC 4456 はパスベクタの原理どおり、**道筋の印を2つの新しい
パスアトリビュート**(いずれも optional non-transitive)として経路に積む。

- **ORIGINATOR_ID(Type 9)**: この経路を AS 内に持ち込んだ発信者の
  BGP Identifier。最初に反射する RR が付ける。**自分の ID が
  ORIGINATOR_ID に入った経路を受け取ったルータは、それを無視する**
  (自分の話が一周して戻ってきた)
- **CLUSTER_LIST(Type 10)**: 経路が通ってきたクラスタの ID の列。
  RR は反射のたびに自分の**クラスタ ID**(既定では自身の BGP Identifier、
  設定で変更可)を先頭に前置する。**自分のクラスタ ID がリストに
  含まれる経路を受け取った RR は、それを無視する**

AS_PATH と見比べてほしい。「前置」「自分の印で破棄」という動作まで
そっくり同じである。AS 間で AS 番号が果たす役割を、AS 内では
発信者 ID とクラスタ ID が果たす — パスベクタの二段構えである。

なお両属性が non-transitive なのは、[05 章で見た MP-BGP の器と同じ理由による](05_mp_bgp.md)。
これらは AS 内のループ検出情報であり、AS の外に意味を持たない。
万一境界から漏れても、理解しない実装のところで静かに止まる。

#### 又聞きの代償② — 情報の間引き(パスハイディング)

もう1つの代償は、ループより目立たないが設計上は重要である。
反射の規則をよく読むと、RR が配るのは**自分の経路選択プロセスが選んだ
最良の1本だけ**である。フルメッシュなら各ルータは同じ経路を
複数の境界ルータから直接聞き、全候補を自分の視点で比較できた。
RR 経由では、**候補の絞り込みが RR の視点で先に行われてしまう**。

これは2つの形で現れる。

- **経路多様性の喪失**: クライアントはバックアップ候補を知らないため、
  最良経路の障害時に「RR が次の最良を選び直して反射してくる」までの
  時間、代替経路を持たない。フェイルオーバーが一段遅くなる
- **ホットポテトの歪み**: [03 章で見たとおり](03_path_attributes.md)、
  経路選択の終盤には「ネクストホップへの IGP コスト」の比較がある。
  RR はこれを**自分の位置**から計算するので、RR の選んだ出口が
  クライアントにとっての最寄りの出口とは限らない。AS に出口が
  複数あるとき、フルメッシュなら各自が最寄りの出口を選べたはずが、
  全員が「RR にとっての最寄り」に誘導される

緩和策は2方向ある。設計面では、RR をトポロジ上の自然な集約点
(POP ごと、地域ごと)に置き、クラスタを物理構成に沿わせる。
プロトコル面では、**ADD-PATH(RFC 7911)** — 同一プレフィックスの
複数経路を並べて広告できるようにするケイパビリティ
([05 章の交渉の枠組み](05_mp_bgp.md)に載る拡張)— で
間引き自体を緩める。本書では存在の言及にとどめる。

また、RFC 4456 は RR が反射時に経路の属性(NEXT_HOP、AS_PATH、
LOCAL_PREF など)を**変更しないこと**を求めている。RR はあくまで
コントロールプレーンの分配器であり、反射でネクストホップが変わらない以上、
**RR 自身がデータパス上にいる必要はない**(転送に参加しない専用機や
仮想マシンを RR にする設計も普通に成立する)。反射時の属性変更は
経路選択の視点をさらに歪め、転送ループの原因にもなりうる。

#### 冗長化とクラスタ ID の設計

RR はクラスタの単一障害点になるので、実務では**1つのクラスタに
RR を2台以上**置き、クライアントは全 RR とピアを張る。このとき
クラスタ ID の設計に2流儀がある。

- **2台に同じクラスタ ID を与える**: RR1 が反射した経路を RR2 は
  (自クラスタ ID を見て)破棄する。重複経路を持たないぶんメモリに
  優しいが、破棄された側の RR は代替情報を持たない
- **既定のまま別々の ID にする**: 互いの反射経路を保持し合う。
  経路は重複して持つが、情報の欠けに強い

現代は経路メモリより堅牢性を取り、**別々の ID(既定)のままにする**のが
一般的な推奨である。いずれにせよ「クラスタ ID を共有した瞬間、
その ID を持つ RR 間では経路が流れなくなる」ことは、
トラブルシューティングの節で再訪する。

### コンフェデレーション(RFC 5065)

#### 発想 — AS の中に AS を作る

もう1つの解は、視点がまったく違う。フルメッシュが n の2乗で破綻する
なら、**n を小さくすればよい** — AS を複数の**メンバー AS**
(member-AS、サブ AS とも)に分割し、それぞれに内部専用の AS 番号
(通常はプライベート AS 番号)を振る。メンバー AS の内側では従来どおり
iBGP(フルメッシュ、または RR)を張り、メンバー AS の**間**は
eBGP の変種で接続する。ループ検出は AS_PATH の仕組みがそのまま —
ただし専用のセグメントタイプで — 面倒を見てくれる。

```text
        AS 65000(外の世界から見える姿)
 ┌───────────────────────────────────────────┐
 │   メンバー AS 65101         メンバー AS 65102   │
 │  ┌──────────────┐           ┌──────────────┐ │
 │  │  R1 ─── R2   │  confed   │  R3 ─── R4   │ │
 │  │   ╲   ╱     │←─ eBGP ─→│   ╲   ╱     │ │
 │  │ (iBGP フル    │           │ (iBGP フル    │ │
 │  │  メッシュ)    │           │  メッシュ)    │ │
 │  └──────────────┘           └──────────────┘ │
 └───────────────────────────────────────────┘
    ↑ 外部の AS 64510 から見ると、単一の AS 65000
```

#### 「外から見れば1つの AS」を守るための特殊規則

メンバー AS 間のセッションは eBGP そのものではない。素の eBGP に
してしまうと、AS 全体としての一体性が壊れる — LOCAL_PREF は
[AS の外に出せない属性](03_path_attributes.md)だから出口選択の意思統一が
メンバー AS 単位に分断され、内部専用の AS 番号が世界に漏れてしまう。
そこで RFC 5065 は、メンバー AS 間の eBGP(confederation eBGP と呼ばれる)に
次の特殊規則を課す。

- **AS_PATH には専用セグメントを使う**: メンバー AS 境界の通過では、
  通常の AS_SEQUENCE ではなく **AS_CONFED_SEQUENCE(セグメントタイプ 3)**
  /**AS_CONFED_SET(タイプ 4)**にメンバー AS 番号を積む。ループ検出は
  通常の AS_PATH と同じ「自分の番号があれば破棄」
- **コンフェデレーションの外に出る瞬間、AS_CONFED_\* セグメントを
  すべて取り除き**、公開 AS 番号(コンフェデレーション ID)を1つ前置する。
  外部から内部構造は一切見えない
- **NEXT_HOP・MED・LOCAL_PREF を変更せずに運ぶ**: 意思統一(LOCAL_PREF)と
  転送情報(NEXT_HOP)は AS 全体で連続させる
- **経路選択で AS_CONFED_\* セグメントは AS_PATH 長に数えない**:
  内部の分割の仕方が経路の「長さ」に影響してはならない

つまり confederation eBGP は「AS_PATH の形式だけ eBGP、
属性の意味論は iBGP」という中間的なセッションである。
[04 章の well-known コミュニティの表](04_policy_control.md)に登場した
**NO_EXPORT_SUBCONFED** の伏線もここで回収できる — NO_EXPORT が
「AS(コンフェデレーション全体)の外に出すな」であるのに対し、
NO_EXPORT_SUBCONFED は「**メンバー AS** の外に出すな」、
すなわち confederation eBGP のピアにも広告しない、より狭い指定である。

#### RR との使い分け — なぜ RR が主流になったか

| 観点 | ルートリフレクタ | コンフェデレーション |
|---|---|---|
| 変更の範囲 | RR にする少数のルータだけ(クライアントは無改造) | **全ルータ**(AS 番号・ピア構成の再設計) |
| 漸進的な導入 | 容易(1クラスタずつ) | 困難(移行は大工事) |
| ループ検出 | ORIGINATOR_ID / CLUSTER_LIST | AS_CONFED_SEQUENCE |
| 経路の見え方 | RR が選んだ最良1本(パスハイディング) | メンバー AS 境界ごとに選択(同様の間引きはある) |
| 分割の単位 | 論理(クラスタ) | 構造(メンバー AS)。**メンバー AS ごとに IGP を分離できる** |
| ポリシーの適用 | 反射では属性をいじらない(規則) | メンバー AS 間で(限定的に)ポリシーを適用できる |
| 実務での採用 | **主流** | 少数派(歴史的な大規模網、組織統合の経緯がある網) |

コンフェデレーションの強みは、メンバー AS ごとに IGP を分けられる
構造の分割にある。巨大な網や、AS どうしの合併で生まれた網には
合理性があった。しかし「全ルータの設定を作り替える」導入コストは重く、
「RR を数台指定するだけ」の手軽さには勝てなかった。現代の新規設計で
迷ったら RR、と考えてよい。2つを併用する(メンバー AS の内部を
RR で組む)ことも仕様上は可能である。

### 現代の大規模設計 — データセンターと BGP

#### アンダーレイを eBGP で組む(RFC 7938)

[第2部](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)で予告した
「大規模 DC では IGP の代わりに BGP でアンダーレイを組む設計」を
ここで扱う。RFC 7938(Informational)は、大規模データセンターの
リーフ・スパイン網で **eBGP を唯一のルーティングプロトコル**として使う
設計を整理した文書である。

発想を逆から言うとわかりやすい。この網に必要なのは、ループバックと
リンクの経路を配り、スパイン本数分の ECMP を効かせることだけである。
リンクステート IGP でももちろん組めるが、数千台規模になると
フラッディングの範囲制御が設計課題になる。一方 eBGP なら:

- **伝搬の範囲と粒度を制御しやすい**: BGP は「聞いたことを選んで話す」
  プロトコルであり、経路の要約・フィルタが
  [ポリシーの関所](04_policy_control.md)として最初から備わっている
- **障害の影響が局所化する**: リンク断は当該セッションの経路撤回として
  伝わり、LSDB の再同期や SPF 再計算のような AS 全体のイベントを起こさない
- **1つのプロトコルで済む**: IGP + iBGP の2階建て
  ([02 章の積層関係](02_ibgp_ebgp.md))を運用せず、eBGP 1層で完結する

具体的な設計の骨子は次のとおり。各リーフ(または各ラック)に
**専用のプライベート AS 番号**を振り(4オクテット AS 番号
(RFC 6793)のプライベート領域 4200000000〜4294967294 を使えば
枯渇しない)、スパイン層は**層で1つの AS 番号を共有**する。
セッションはすべて p2p リンク上の直結 eBGP(TTL 1)で、
ループ検出は AS_PATH に任せる。同一宛先へスパイン経由の等コスト経路を
複数使うために、「AS_PATH 長が同じなら中身の AS 番号が違っても
ECMP に載せる」実装ノブ(multipath relax などと呼ばれる。RFC 外の
実装慣例)を有効にするのが定石である。

ここで面白いのは役割の逆転である。第1部以来「IGP は土台、BGP は積み荷」
という分業を基本構図としてきたが、RFC 7938 の網では **BGP 自身が
土台(アンダーレイ)になる**。プロトコルの性質(ポリシー、局所化、
TCP 上の差分配布)が要求に合うなら、AS 間・AS 内という本来の区分に
こだわる必要はない — BGP が[05 章で見たとおり](05_mp_bgp.md)
汎用の配布機構である以上、これも自然な適用先の1つである。

#### EVPN ファブリックの経路配布 — スパインを RR にする

この章の道具立てで、[第2部の EVPN ファブリック](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)の
経路配布を設計してみよう。オーバーレイ(EVPN、AFI 25 / SAFI 70)の
経路は全リーフに行き渡る必要がある。リーフ同士の iBGP フルメッシュは
リーフ数の2乗で破綻する — まさに本章の問題そのものである。

定石は、**アンダーレイは IGP(OSPF / IS-IS)で組み、オーバーレイは
ループバック間の iBGP とし、スパインをルートリフレクタにする**構成である。

- 全リーフは全スパインと物理的に接続している(リーフ・スパインの定義)
  ので、「クライアントは RR とだけピアを張る」構成が物理トポロジと
  一致する。クラスタ設計に迷いがない
- スパインは [VTEP ではなく、VNI も MAC も関知しない](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)が、
  **EVPN 経路の反射には参加する**。データプレーンの無関与と
  コントロールプレーンの中継は独立 — 「RR はデータパスにいなくてよい」の
  好例である(逆に、トンネルの終点はあくまでリーフなので、反射で
  NEXT_HOP が変わらないことが本質的に効いている)
- RR の弱点だったパスハイディングも、この網では痛まない。EVPN の
  経路(RT-2 / RT-3)のネクストホップは発信 VTEP そのものであり、
  同一 NLRI の候補が複数の出口から届く状況が(マルチホーミングを除き)
  そもそも少ない

アンダーレイも eBGP(RFC 7938)にした上で、オーバーレイの EVPN を
その eBGP セッションに多重する、全部 eBGP の設計もある。この場合
スパインは RR ではなく普通の eBGP 中継点になるが、EVPN 経路の
ネクストホップ(= トンネル終点)を書き換えさせない設定
(next-hop-unchanged 相当)が必要になるなど、規則の例外が増える。
本書では「iBGP + スパイン RR」を基本形として覚えることを勧める。

## プロトコル動作の詳細

### ORIGINATOR_ID と CLUSTER_LIST のフォーマット

RFC 4456 Section 8 の定義:

```text
ORIGINATOR_ID(Type 9、optional non-transitive)
+---------------------------------------------------------+
| BGP Identifier(4 オクテット)                           |
+---------------------------------------------------------+
  経路を AS 内に持ち込んだルータの BGP Identifier。
  最初に反射する RR が付与する(すでに付いていれば触らない)。

CLUSTER_LIST(Type 10、optional non-transitive)
+---------------------------------------------------------+
| クラスタ ID の列(4 オクテット × 反射された回数)       |
+---------------------------------------------------------+
  RR が反射するたびに、自分のクラスタ ID を先頭に前置する。
```

破棄(ループ検出)の規則を整理すると:

| 検査するもの | 検査する者 | 破棄の条件 |
|---|---|---|
| AS_PATH | すべての BGP スピーカー | 自 AS 番号が含まれる(AS 間のループ) |
| ORIGINATOR_ID | すべての iBGP スピーカー | 自分の BGP Identifier と一致(発信者に戻った) |
| CLUSTER_LIST | RR | 自分のクラスタ ID が含まれる(クラスタを一周した) |

### 経路選択プロセスへの追加(RFC 4456 Section 9)

[03 章の多段比較](03_path_attributes.md)に、RR 環境では2つの修正が入る。

- **BGP Identifier の比較では、ORIGINATOR_ID があればその値を使う**。
  反射された経路は「RR の ID」ではなく「本来の発信者の ID」で比較される —
  又聞きでも比較の土俵を発信者ベースで揃えるためである
- **BGP Identifier の比較の直後に「CLUSTER_LIST が短い経路を優先」を挿入する**
  (属性がなければ長さ 0)。反射の段数が少ない、
  つまり発信者に近い聞き方をした経路を選ぶ

いずれも比較の最終盤(ほとんどタイブレーク)であり、LOCAL_PREF や
AS_PATH 長など上位の段に差があればそちらで決まることは変わらない。

### 反射のウォークスルー

境界ルータ R1 が eBGP で受けた経路が、RR 経由でクライアント R2 に
届くまでを追う。R1・R2 は RR のクライアントで、互いにピアを張っていない。

```text
  AS 64500 ──(eBGP)── R1 ──(iBGP)── RR ──(iBGP)── R2
                        [クライアント]  [クラスタ ID    [クライアント]
                                        192.0.2.100]
```

1. **AS 64500 → R1(eBGP)**: 経路 203.0.113.0/24 を受信。
   AS_PATH = `64500`、NEXT_HOP = 相手のアドレス
2. **R1 の取り込み**: [next-hop-self](02_ibgp_ebgp.md) により NEXT_HOP を
   自ループバック 10.255.0.1 に書き換え、LOCAL_PREF 100 を付けて
   iBGP(RR 向け)へ広告する。ORIGINATOR_ID / CLUSTER_LIST はまだない
3. **RR の反射**: RR は経路選択プロセスでこの経路を最良と判定。
   クライアントから学んだ経路なので、規則により(発信者 R1 を除く)
   全クライアントと全非クライアントへ反射する。このとき
   **ORIGINATOR_ID = 10.255.0.1(R1 の BGP Identifier)を付与**し、
   **CLUSTER_LIST の先頭に 192.0.2.100 を前置**する。
   AS_PATH・NEXT_HOP・LOCAL_PREF には触らない
4. **R2 の受信**: ORIGINATOR_ID は自分の ID ではない → 受け入れ。
   NEXT_HOP 10.255.0.1 を IGP で再帰解決して使う。
   経路を「くれた」のは RR だが、パケットが向かう先は R1 である
5. **もし反射が R1 に届いたら**: R1 は ORIGINATOR_ID = 自分の ID を見て
   無視する(手順3で発信者は除外されるが、冗長 RR 構成などで
   経路が回り込んでも、この検査が最後の安全弁になる)

区間ごとの属性の変化をまとめる:

| 区間 | AS_PATH | NEXT_HOP | LOCAL_PREF | ORIGINATOR_ID | CLUSTER_LIST |
|---|---|---|---|---|---|
| AS 64500 → R1(eBGP) | `64500` | 相手アドレス | (付かない) | なし | なし |
| R1 → RR(iBGP) | `64500` | **10.255.0.1**(self) | 100 | なし | なし |
| RR → R2(反射) | `64500`(不変) | 10.255.0.1(不変) | 100(不変) | **10.255.0.1** | **192.0.2.100** |

### コンフェデレーションの AS_PATH の変化

導入の節の図(AS 65000 = メンバー AS 65101 + 65102)で、
AS 64500 から来た経路が外部 AS 64510 へ抜けるまで:

```text
 ① AS 64500 → R1(eBGP)          AS_PATH: 64500
 ② R1 → R2(65101 内の iBGP)     AS_PATH: 64500(変更なし)
 ③ R2 → R3(confed eBGP)         AS_PATH: (65101) 64500
                                    └ AS_CONFED_SEQUENCE に 65101 を前置
 ④ R3 → R4(65102 内の iBGP)     AS_PATH: (65101) 64500(変更なし)
 ⑤ R4 → AS 64510(真の eBGP)     AS_PATH: 65000 64500
                                    └ (65101) を除去し、公開 AS 65000 を前置
```

- ③で LOCAL_PREF・NEXT_HOP・MED はそのまま渡る(iBGP 的な意味論)
- ④の R4 が経路選択で AS_PATH 長を数えるときは `(65101)` を無視して
  長さ 1 と数える
- ⑤の除去を**忘れる実装があってはならない**。AS_CONFED_\* セグメントが
  真の eBGP ピアに届いた場合、受信側はこれを属性エラーとして扱う
  (RFC 7606 では treat-as-withdraw)

## 設定例(補助)

FRRouting で、スパイン(AS 65000、ループバック 192.0.2.100)を
2台のリーフに対する RR にする例。[第2部のファブリック](../02_vlan_vxlan_evpn/05_evpn_vxlan.md)の
「スパイン = RR」を最小構成で再現したものである。

```text
router bgp 65000
 bgp cluster-id 192.0.2.100
 neighbor 10.255.0.1 remote-as 65000
 neighbor 10.255.0.1 update-source lo
 neighbor 10.255.0.2 remote-as 65000
 neighbor 10.255.0.2 update-source lo
 !
 address-family l2vpn evpn
  neighbor 10.255.0.1 activate
  neighbor 10.255.0.1 route-reflector-client
  neighbor 10.255.0.2 activate
  neighbor 10.255.0.2 route-reflector-client
 exit-address-family
```

見どころは3つ。**RR であることは `route-reflector-client` という
「相手の指定」だけで決まり**、リーフ側の設定には RR の痕跡が一切ない
(クライアント無改造の原則)。また `route-reflector-client` が
**address-family の中**にあるとおり、反射するかどうかは
[ファミリごとの設定](05_mp_bgp.md)である(EVPN だけ反射し、
IPv4 ユニキャストは反射しない構成も成立する)。`bgp cluster-id` を
省略すれば BGP Identifier がクラスタ ID になる(既定)。

クライアント側で反射された経路を見ると(代表的な表示。
IPv4 ユニキャストの例):

```text
leaf2# show bgp ipv4 unicast 203.0.113.0/24
BGP routing table entry for 203.0.113.0/24
Paths: (1 available, best #1)
  64500
    10.255.0.1 (metric 20) from 192.0.2.100 (10.255.0.1)
      Origin IGP, localpref 100, valid, internal, best
      Originator: 10.255.0.1, Cluster list: 192.0.2.100
```

`from 192.0.2.100`(経路をくれたのは RR)なのにネクストホップは
`10.255.0.1`(パケットが向かうのは R1)— **配布の経路と転送の経路が
別物**であることが1行に現れている。`Originator` / `Cluster list` の
行で、この経路が反射されてきたこと、どのクラスタを何段通ったかが読める。

## トラブルシューティング

### 症状: RR を導入したのに、クライアントに経路が配られない

- まず `route-reflector-client` の設定漏れを疑う。指定がなければ
  RR にとってそのピアはただの iBGP ピアであり、iBGP から学んだ経路は
  再広告禁止の**正常動作**として配られない。[02 章の「フルメッシュの欠け」](02_ibgp_ebgp.md)と
  同じ見え方をするが、原因は規則ではなく指定漏れである
- MP-BGP 環境では**ファミリを確認する**。`route-reflector-client` は
  address-family ごとの設定なので、「IPv4 は反射されるのに EVPN が
  来ない」なら EVPN 側の指定漏れが典型である
  ([05 章の切り分けの層構造](05_mp_bgp.md)の②と③の間に
  「反射の指定」という段が増えたと考えるとよい)

### 症状: 冗長 RR ペアの片方経由の経路だけが捨てられる

- 2台の RR に**同じクラスタ ID** を設定していないかを確認する。
  同一 ID の構成では、RR1 が反射した経路を RR2 が CLUSTER_LIST 検査で
  破棄するのは**仕様どおりの動作**である。それが意図した設計
  (重複経路の削減)なのか、コピペ設定の事故なのかを切り分ける
- 事故の典型は「`bgp cluster-id` を含む設定を丸ごと複製して2台目を
  作った」ケースである。堅牢性を優先するなら ID は既定(各自の
  BGP Identifier)のままにし、明示設定は消す

### 症状: クライアントの選ぶ出口が期待と違う / 障害時の切り替えが遅い

- パスハイディングを疑う。`show bgp` で当該プレフィックスの候補数を
  RR 上とクライアント上で比べれば一目瞭然である — RR には複数候補が
  あるのにクライアントには1本しか届いていないなら、それが間引きである
- 出口の歪みは故障ではなく、**RR の視点で IGP コストが比較されている**
  構造の帰結である。対処は設計側にある: RR の配置・クラスタ分割を
  トポロジに沿わせる、または ADD-PATH(RFC 7911)対応を検討する
- 反射経路の属性を RR のポリシーで書き換えていないかも確認する。
  RFC 4456 が警告するとおり、反射時の属性変更は選択の歪みや
  ループの原因になる

### 症状: コンフェデレーションで、隣のメンバー AS の経路が使えない

- `show bgp` で経路自体は見えるのにベストにならない・RIB に載らない
  なら、[02 章と同じネクストホップ解決](02_ibgp_ebgp.md)をまず疑う。
  confed eBGP は **NEXT_HOP を書き換えずに**渡すため、メンバー AS ごとに
  IGP を分割している設計では「隣のメンバー AS の境界ルータの
  ループバック」が自分の IGP に存在せず、再帰解決に失敗する
- 対処は2択で、境界ルータのループバック(またはリンク)を
  メンバー AS 間で相互に再配送・リークするか、confed eBGP の境界で
  next-hop-self を使う。「NEXT_HOP はどの経路情報で解決されるはずか」
  から逆算すれば、どちらの欠けかが特定できる

## 演習・確認問題

1. iBGP の再広告禁止(RFC 4271 Section 9.2)がループ防止として機能する
   理由を述べ、ルートリフレクタとコンフェデレーションがそれぞれ
   「何を緩め、代わりにどんなループ検出を持ち込んだか」を対比せよ。
2. RR の反射3規則を、経路を学んだ相手(eBGP / 非クライアント /
   クライアント)ごとに述べよ。RR と非クライアントの間に
   フルメッシュが残る理由も説明せよ。
3. ORIGINATOR_ID と CLUSTER_LIST について、誰がいつ付与・更新し、
   誰がどんな条件で経路を破棄するかを AS_PATH と対比して説明せよ。
   両属性が optional non-transitive である理由も述べよ。
4. パスハイディングとは何か。経路多様性とホットポテトルーティングの
   2つの観点から説明し、設計面・プロトコル面の緩和策を1つずつ挙げよ。
5. confederation eBGP が通常の eBGP と異なる点を、AS_PATH の扱い・
   LOCAL_PREF・NEXT_HOP・経路選択(パス長の数え方)について述べよ。
   NO_EXPORT と NO_EXPORT_SUBCONFED の違いも説明せよ。
6. 新規に AS 内部の経路配布を設計するとき、RR とコンフェデレーションの
   どちらを選ぶべきか。導入コスト・漸進性・実務での採用状況の観点から
   論ぜよ。
7. EVPN ファブリックで「スパインを RR にする」構成が自然である理由を、
   物理トポロジとの一致・データパス無関与・NEXT_HOP 不変の3点から
   説明せよ。アンダーレイを RFC 7938 流の eBGP にした場合に
   何が変わるかも述べよ。

## まとめ

- iBGP フルメッシュの n(n-1)/2 の破綻に対する2つの解は、どちらも
  パスベクタの原理(道筋を情報に載せ、自分の印で捨てる)を
  AS の内側に再輸入したものである
- ルートリフレクタ(RFC 4456)は再広告禁止に例外を開け、
  ORIGINATOR_ID / CLUSTER_LIST でループを検出する。クライアント無改造で
  漸進導入できる手軽さから実務の主流となった。代償は又聞きの構造に
  由来するパスハイディング(経路多様性の喪失と出口選択の歪み)である
- コンフェデレーション(RFC 5065)は AS をメンバー AS に分割し、
  AS_CONFED_SEQUENCE でループを検出する。LOCAL_PREF・NEXT_HOP・MED は
  メンバー AS を跨いで保持され、外部からは単一の AS に見える
- RFC 7938 は大規模 DC のリーフ・スパイン網で eBGP を唯一の
  ルーティングプロトコルとして使う設計を示した。「IGP は土台、
  BGP は積み荷」の分業は絶対ではなく、BGP が土台にもなりうる
- EVPN ファブリックの定石は「アンダーレイ IGP + ループバック間 iBGP +
  スパイン RR」。反射で NEXT_HOP が変わらないことが、データパスに
  関与しないスパインへの RR 配置を可能にしている

これで第3部は完結である。セッションと FSM(01章)、2つの規則(02章)、
属性と選択(03章)、ポリシー(04章)、多目的化(05章)、そして
大規模化(本章)— この道具立ての上で、BGP は第2部の EVPN を、
第4部の IPv6 経路を、第5部の VPN を運ぶ。
