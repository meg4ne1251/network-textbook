# ポリシー制御 — 経路の選別と属性の書き換え

## 概要

この章では、BGP の経路を選別し、属性を書き換えるための道具立て —
**prefix-list**、**AS_PATH フィルタ**、**コミュニティ**、そしてそれらを束ねる
**route-map** — を扱う。前提知識は [BGP の基礎の章](01_bgp_basics.md) の
3段の RIB(ポリシーの関所)と、[パスアトリビュートの章](03_path_attributes.md) の
属性の意味論・経路選択プロセスである。前章が「属性の意味」なら、本章は「属性の操作」である。

## 導入 — 関所には検問の道具が要る

[BGP の基礎の章](01_bgp_basics.md) で、BGP にはピアとの受け渡しの前後に
必ず**ポリシーの関所**があると述べた。[前章](03_path_attributes.md) では、
関所で下される判断の材料 — 属性の意味と比較の順序 — を見た。
残る問いは実際的である。**関所に何を置けば、意図した検問ができるのか。**

検問の中身を分解すると、どんなポリシーも3つの要素に還元できる:

- **どこで** — どのピアとの、どちら向き(受信 in / 広告 out)の関所か
- **どの経路に** — 何を基準に経路を選り分けるか(**マッチ条件**)
- **何をするか** — 通す/落とす、そして通すなら属性をどう書き換えるか(**アクション**)

IGP にはこの道具立てが(再配送の注入点フィルタ程度しか)存在しなかったことを
思い出してほしい。リンクステートの大前提は「全ルータが同じ地図を持つ」ことであり、
情報を選り分けたり書き換えたりすることは原理からの逸脱だからである
([第1部](../01_fundamentals/04_distance_vector_link_state.md))。
BGP は逆で、**選別と書き換えこそがプロトコルの本体**である。
経路選択プロセスの各段(LOCAL_PREF、MED、AS_PATH 長)は、
ポリシーが属性を書き換えることによって初めて「操作できる判定」になる。

本章の道具は2系統に分かれる。経路を**選別する**道具(prefix-list、
AS_PATH フィルタ)と、選別・書き換えの**荷札**になる属性(コミュニティ)である。
そして route-map が両者を「マッチ+アクション」の形に束ねる。
最後に、これらを組み合わせた実務の定石 — AS 間の**商業関係に基づくポリシー** —
を1本のウォークスルーとして組み立てる。

## 理論

### ポリシーはどこで効くか — 3段の RIB との対応

道具の前に、適用点を正確にしておく。
[BGP の基礎の章](01_bgp_basics.md) の3段の RIB に重ねると、
ポリシーの関所は2箇所である:

```text
 ピアから受信                                            ピアへ広告
     │                                                      ▲
     ▼                                                      │
 Adj-RIB-In ──▶【入力ポリシー(in)】──▶ 経路選択 ──▶ Loc-RIB ──▶【出力ポリシー(out)】──▶ Adj-RIB-Out
  (ピアごと)     選別・書き換え      (Phase 1〜2)              選別・書き換え        (ピアごと)
```

- **入力ポリシー**は、受信した経路を「自分の経路選択の土俵に上げるか、
  どんな属性で上げるか」を決める。前章の Phase 1(度数の計算)で
  「eBGP から受けた経路は入力ポリシーで値付けする」と述べたのはここである
- **出力ポリシー**は、Loc-RIB で勝った経路を「このピアに見せるか、
  どんな属性で見せるか」を決める(Phase 3)

どちらも**ピアごと・方向ごと**に独立に設定できる。同じ経路が、
ピア A には LOCAL_PREF 200 で受け入れられ、ピア B からの同じ広告は
捨てられる、ということが普通に起こる — それがポリシーである。

### 選別の道具① — prefix-list

最も基本的な選別基準は「**宛先プレフィックスそのもの**」である。
prefix-list は、プレフィックスの値と**長さの範囲**の組で経路を選り分ける:

```text
ip prefix-list <名前> seq <番号> permit|deny <プレフィックス>/<L> [ge <X>] [le <Y>]
```

マッチの条件は2つの独立した検査の AND である:

1. 経路のプレフィックスの**上位 L ビット**が、記述したプレフィックスと一致する
2. 経路のプレフィックス**長**が範囲内にある —
   ge も le もなければ**ちょうど L**、`ge X` があれば X 以上(上限は le か 32)、
   `le Y` があれば Y 以下(下限は ge か L)

なぜ長さの範囲指定が要るのか。ルーティングの世界では
[ロンゲストマッチ](../01_fundamentals/02_routing_table_basics.md) がある以上、
「10.0.0.0/8 を許可」だけでは足りないからである。実務の要求は常に
「**この集約の配下の、どの細かさまでを**許すか」という形をとる —
細かすぎる経路(ホストルート)を受ければテーブルが荒れ、
細かい経路は集約より常に勝つ(ロンゲストマッチ)ため、
経路ハイジャックの主要な形態も「より細かいプレフィックスの広告」である。
長さの範囲は、この攻防の主戦場を直接記述するための構文である。

代表的な書き方を挙げる。**この読み替えがすべての基本**になる:

| 記述 | マッチする経路 | 用途 |
|---|---|---|
| `permit 0.0.0.0/0` | デフォルトルート**のみ**(長さちょうど 0) | 上流からデフォルトだけ受ける |
| `permit 0.0.0.0/0 le 32` | **すべての経路**(上位0ビット一致=無条件、長さ 0〜32) | 「any」の慣用表現 |
| `permit 10.0.0.0/8 le 24` | 10/8 配下で /8〜/24 の経路 | 集約と適度な詳細だけ許す |
| `deny 0.0.0.0/0 ge 25` | /25 より細かいすべての経路 | ホストルート級の混入を拒否 |

複数行の prefix-list は**シーケンス番号順に評価され、最初にマッチした行で確定**する
(permit なら通過、deny なら不通過)。どの行にもマッチしなければ**暗黙の deny**である。
この「先勝ち+暗黙の deny」は本章のすべての道具に共通する評価規則なので、
ここで覚えてしまってほしい。

### 選別の道具② — AS_PATH フィルタ

宛先ではなく「**来歴**」で選びたいこともある。「AS 64496 が生成した経路だけ」
「AS 64510 を経由する経路は使わない」といった条件である。
AS_PATH は AS 番号の列だから、選別の道具は**正規表現**になる
(Cisco 系では as-path access-list と呼ばれる)。最低限の語彙は4つで足りる:

| パターン | 意味 |
|---|---|
| `^$` | AS_PATH が空 = **自 AS 生成**の経路(iBGP で受けた自 AS 発の経路) |
| `^64496_` | **隣の 64496 から直接**学んだ経路(列の先頭が 64496) |
| `_64511$` | **64511 が生成**した経路(列の末尾が 64511) |
| `_64510_` | 経路がどこかで **64510 を経由**する |

`_`(アンダースコア)は区切り(先頭・末尾・AS 番号の間)にマッチする
メタ文字で、「64496」が「164496」に部分一致してしまう事故を防ぐ。
`^$` は最重要である — 「**自分が生成した経路だけを外に広告する**」という、
すべての末端 AS が持つべき最低限の出力ポリシーがこの3文字で書ける。

### 荷札の属性 — コミュニティ(RFC 1997)

prefix-list も AS_PATH フィルタも、経路が**もともと持っている情報**で選別する。
だが実務のポリシーはしばしば、経路自体からは読み取れない区別を必要とする —
「これは**顧客から**受けた経路」「これは**大阪の拠点で**受けた経路」。
受信の文脈は、受信の瞬間を過ぎれば失われる。
それを経路に**書き留めておく**ための属性がコミュニティである。

**COMMUNITY 属性(Type 8、optional transitive)**は **RFC 1997** で定義された、
**32 ビット値の集合**である(1本の経路に何個でも付けられる)。
表記は上位 16 ビットと下位 16 ビットをコロンで区切った **AS:値** が慣例で、
「上位に自分の AS 番号、下位に組織内で決めた意味番号」を入れる
(64500:100 など。名前空間を AS 番号で区切ることで他組織との衝突を避ける)。

決定的に重要なのは、**プロトコルはコミュニティの意味を定めない**ことである。
64500:100 が何を意味するかは AS 64500 が決め、その意味は
**貼る者と読む者の合意**の中にしか存在しない。つまりコミュニティは
経路に貼る**自由記述の荷札**であり、使い方は2系統に分かれる:

- **情報タグ**: 受信の文脈(どの関係の相手から、どの拠点で、どの種類の経路として
  受けたか)を入口で記録し、AS 内の別の場所(出口)のポリシーが読んで判断する。
  荷札を貼る者と読む者が**同一 AS 内**にいる使い方
- **アクション依頼**: 「この荷札が付いた経路は LOCAL_PREF を下げる」といった
  処理内容を **ISP が顧客向けに公開**し、顧客が荷札を貼って広告することで
  相手のポリシーを**遠隔操作**する使い方。[前章](03_path_attributes.md) の
  入口制御の第4の手段「コミュニティで依頼する」の実体である

例外的に、**意味が世界共通で予約されたコミュニティ**がある
(well-known communities。値域 0xFFFF0000〜0xFFFFFFFF が予約されている):

| 名前 | 値 | 受け取った BGP スピーカーの動作 |
|---|---|---|
| NO_EXPORT | 0xFFFFFF01 | この経路を **eBGP ピアへ広告しない**(AS の外に出さない) |
| NO_ADVERTISE | 0xFFFFFF02 | この経路を**誰にも広告しない** |
| NO_EXPORT_SUBCONFED | 0xFFFFFF03 | NO_EXPORT のコンフェデレーション版([06 章](06_large_scale_design.md)) |
| GRACEFUL_SHUTDOWN | 0xFFFF0000 | 経路の優先度を最低(LOCAL_PREF 0)にする(**RFC 8326**) |
| BLACKHOLE | 0xFFFF029A | この宛先へのトラフィックを**廃棄する**(**RFC 7999**) |

NO_EXPORT の典型は「隣接 AS には細かい経路を見せてトラフィックを誘導するが、
世界には集約だけを見せる」— 細かい経路に NO_EXPORT を貼って広告すれば、
隣接 AS の中では使われ、その先へは伝搬しない。
GRACEFUL_SHUTDOWN と BLACKHOLE は「アクション依頼」が標準化されるほど
普遍的だった実例で、動作の節で再訪する。

### コミュニティの拡張 — 型付きと 32 ビット AS 対応

**拡張コミュニティ(Type 16、RFC 4360)**は、コミュニティを 8 オクテットに広げ、
先頭に**型フィールド**を持たせたものである。自由記述だった荷札が
「型を見れば構造と意味が機械的に定まる」構造化された荷札になり、
これが VPN 系の機能の基盤になった —
[第2部で使った RT(ルートターゲット)](../02_vlan_vxlan_evpn/05_evpn_vxlan.md) の
正体は、拡張コミュニティの Route Target 型(どの VRF / MAC-VRF に
取り込むかの仕分けラベル)である。型ごとに transitive かどうかも定義される。

**ラージコミュニティ(Type 32、RFC 8092)**は 32 ビット AS 番号
(RFC 6793)への対応である。AS 番号が 4 オクテットになると、
RFC 1997 の「上位 16 ビットに AS 番号」という慣例が物理的に成立しない。
そこで **4 オクテット × 3**(グローバル管理者 = AS 番号:値1:値2)へ拡張した。
32 ビット AS 番号を持つ組織が名前空間付きの荷札を使うには
ラージコミュニティが必要になる。

3種のコミュニティはいずれも optional transitive(またはそれに準ずる)であり、
原理的には AS を越えて運ばれる。ただし**実際に遠くまで届くかは別問題**である —
送信するかどうか自体が設定項目である実装が多く、また多くの ISP は
境界で他組織のコミュニティを洗い流す(トラブルシューティングで再訪する)。

### 束ねる道具 — route-map

選別の道具と書き換えたい属性が出そろった。これらを
「マッチ+アクション」の形に束ねる構造が **route-map** である。

先に慣用名の注意を述べる。「route-map」は RFC 4271 に登場しない。
仕様が求めるのは「ローカルなポリシーに基づいて経路の選別と属性の操作を行う」
という抽象だけであり、その表現形式は実装ごとに異なる
(Junos の policy-statement、BIRD のフィルタ言語など。
ポリシーを組織間で機械可読に公開するための記述言語 RPSL(**RFC 2622**)も存在する)。
本書は FRRouting / Cisco 系の route-map の形で説明するが、
「順序付きのマッチ+アクション規則列」という概念はどの形式にも共通する。

route-map は**番号付きエントリの列**で、各エントリは
permit/deny・match 句(0個以上)・set 句(0個以上)を持つ:

```text
route-map <名前> permit|deny <番号>
 match <条件>     ← prefix-list、AS_PATH フィルタ、コミュニティ等を参照
 set <アクション>  ← local-preference、community、as-path prepend 等
```

評価規則は prefix-list と同じ「先勝ち+暗黙の deny」である:

1. エントリを番号順に評価し、**match 句をすべて満たした**(複数の match 句は
   AND)最初のエントリで確定する
2. そのエントリが permit なら、set 句を適用して**通す**。deny なら**落とす**
3. どのエントリにもマッチしなければ**暗黙の deny** — 落ちる

2つの帰結を強調しておく。第一に、**match 句のない permit エントリは
すべての経路にマッチする**。「一部の経路だけ属性を変え、残りはそのまま通す」
つもりなら、最後に素の permit エントリ(受け皿)が**必須**である。
第二に、route-map の deny と、参照される prefix-list 内の deny は役割が違う —
prefix-list の deny は「この match 句に**マッチしない**」を意味するだけで、
経路を落とすかどうかは route-map のエントリが決める。この2層の
permit/deny の絡み合いが route-map の読み間違いの大半を占める。

### 既定ポリシーの問題 — 関所に何も置かなかったら

最後に、道具を**置かなかった場合**の規定を確認する。これは安全性の問題である。

歴史的に多くの実装は「ポリシー未設定なら**全部通す**」を既定としてきた。
この既定の下では、設定の書き忘れがそのまま**経路リーク(route leak)**になる —
意図した範囲を超えて経路広告が伝搬する事故である(**RFC 7908** が類型を定義する)。
典型例は、2つの上流を持つ AS が、上流 A から受けたフルルートを
出力フィルタなしで上流 B へ広告してしまうケースである。
B から見ると「A のすべての宛先にその顧客経由で行ける」ように見え
(AS_PATH はより長いが、顧客経路を LOCAL_PREF で最優先する
[前章](03_path_attributes.md) の慣行が裏目に出て選ばれてしまう)、
大量のトラフィックが細い顧客回線に流れ込んで広域障害になる。
実際にこの形の事故は繰り返し発生している。

**RFC 8212** はこの既定を反転させた: **eBGP は、明示的にポリシーが
設定されるまで、経路を受け入れても広告してもならない**(default deny)。
FRRouting はこれを既定で実装しており(`bgp ebgp-requires-policy`)、
ポリシーなしの eBGP セッションは Established になっても経路が流れない。
「関所に何も置かなければ何も通らない」— フェイルセーフの原則が
プロトコルの規定にまで昇格した例である(トラブルシューティングで再訪する)。

## プロトコル動作の詳細

### ポリシーはいつ評価されるか — 再評価と Route Refresh

ポリシーの評価は**経路が関所を通過する瞬間**に行われる。
ここに BGP の差分更新設計([BGP の基礎の章](01_bgp_basics.md))との
相互作用がある。IGP なら情報は定期再送されるからポリシー変更は自然に
行き渡るが、BGP の経路は一度受信されたら**再送されない**。
つまり**ポリシーを変更しても、すでに受信済みの経路には自動では適用されない**
— 何らかの方法で経路を関所に**通し直す**必要がある。手段は3つ:

1. **セッションリセット**: 全経路が再交換されるので確実だが、
   セッション断そのものである。転送への影響が大きく、最後の手段
2. **soft reconfiguration inbound**: 加工前の受信経路(Adj-RIB-In の生のコピー)を
   メモリに保持しておき、ポリシー変更時にローカルで再評価する。
   セッションに触れない代わりに、経路数分のメモリを常時消費する
3. **Route Refresh(RFC 2918)**: ピアに「この AFI/SAFI の経路を
   再送してほしい」と依頼するメッセージを送る。
   [BGP の基礎の章](01_bgp_basics.md) で「定期再広告の不在を補う機構」として
   予告したものの正体で、ケイパビリティ交渉で双方が対応していれば使える。
   メモリ消費なしに入力ポリシーの再評価ができ、現在の標準的な手段

FRR / Cisco 系の `clear bgp <ピア> soft in|out` は、in なら
3(または 2 の再評価)、out なら Adj-RIB-Out の再生成と再広告を指す。
現代の実装(FRR 等)はポリシーの設定変更を検知して自動で再評価を
走らせることも多いが、「**ポリシーは変更しただけでは既存の経路に効かない
ことがある**」という原理は切り分けの前提として持っておくべきである。

### ウォークスルー — 商業関係に基づくポリシー

道具がそろったので、実務で最も普遍的なポリシーを最初から組み立てる。
AS 間の接続は、契約上ほぼ3種類の**関係**に分類できる:

- **顧客(customer)**: 自分に金を払って接続している AS。
  顧客宛て・顧客発のトラフィックを運ぶことが収入になる
- **ピア(peer)**: 互いの顧客宛てのトラフィックを**無償で交換**する対等な相手
  (双方が運搬コストを節約できる場合に成立する)
- **上流(upstream / transit provider)**: 自分が金を払って、
  残りすべての宛先への到達性を買う相手

この経済構造から、広告のルールが**演繹**できる。
「経路を広告する」とは「その宛先へのトラフィックを引き受ける」と
宣言することである([前章](03_path_attributes.md) の
「トラフィックは広告と逆向きに流れる」)。したがって:

- **顧客の経路は全員に広告する** — 顧客宛てのトラフィックを世界中から
  引き込むことは、顧客が金を払っている当のサービスである
- **ピア・上流から学んだ経路は、顧客にだけ広告する** — これをピアや上流に
  広告すると、他人どうしのトラフィックを**自腹で**運ぶことになる
  (先ほどの経路リークとは、まさにこのルールの破れである)

このルールを、**入口で関係をタグ付けし、出口でタグを見る**という
パターンで実装する。ポリシー設計を表1枚に集約すると:

| 関係 | 入口: 貼るタグ | 入口: LOCAL_PREF | 出口: この経路を広告する相手 |
|---|---|---|---|
| 顧客から受信 | 64500:100 | 200 | **全員**(顧客・ピア・上流) |
| ピアから受信 | 64500:200 | 150 | 顧客のみ |
| 上流から受信 | 64500:300 | 100 | 顧客のみ |

LOCAL_PREF の序列(顧客 > ピア > 上流)も同じ経済原理の帰結である —
同じ宛先に複数の経路があるなら、金をもらえる道 > 無償の道 > 金を払う道、
の順で選ぶ。[前章](03_path_attributes.md) のトラブルシューティングで
「顧客から学んだ経路は無条件に最優先、という商用 ISP のほぼ普遍的な慣行」
と述べたものの正体がこの表である。

このパターンの美点は**規模への耐性**にある。ピアが 100 本に増えても、
出口ポリシーは「64500:100 が付いた経路だけ通す」の1本のままでよい。
ピアごと・経路ごとの個別判断(N×M の組み合わせ)が、
**関係の種類数(3)**に圧縮される。これがコミュニティという
「文脈の書き留め」の真価である。
さらに、全 AS がこのルールに従うと、経路広告は「顧客 → 上流方向へ登り、
一度でも下ったら二度と登らない」形(谷なし・valley-free)に保たれ、
経路リークが**構造的に**起こらなくなる。

もう1つ、入口には関係ごとに**選別**も要る。特に顧客からの受信は、
「その顧客(とその顧客の顧客)のプレフィックス**だけ**」を
prefix-list で厳密に許可するのが鉄則である — 顧客の設定ミスや悪意による
経路(他人のプレフィックスの広告=ハイジャック、フルルートの逆流=リーク)を
入口で堰き止める、インターネット全体の安全に対する各 AS の責務である
(広告元 AS の正当性を暗号的に検証する RPKI / Origin Validation
(RFC 6811)という枠組みも実用化されているが、本書の範囲を超えるため
言及にとどめる)。

### アクション依頼の実例 — 標準化された2つの荷札

「アクション依頼」型のコミュニティの動作を、標準化された2例で確認する。

**GRACEFUL_SHUTDOWN(RFC 8326)** はメンテナンスの道具である。
eBGP セッションを止めると、経路の撤回 → 再収束の間、トラフィックが失われる。
そこで**止める前に**、そのセッションで広告している経路に
GRACEFUL_SHUTDOWN(65535:0)を貼って再広告する。受信側はこの荷札を見て
LOCAL_PREF を 0(最低)にする — 経路は残したまま最下位に落ちるので、
代替経路があれば経路選択プロセスが**無停止で**そちらへ切り替える。
全トラフィックが退避したのを確認してからセッションを落とせば、
パケットは1つも失われない。「経路の優先度を下げてほしい」という依頼が
標準の荷札になった例である。

**BLACKHOLE(RFC 7999)** は DDoS 対策の道具である(RTBH:
Remotely Triggered Black Hole と呼ばれる手法)。攻撃で回線が溢れているとき、
被害者側の対処は「攻撃対象アドレス宛てのトラフィックを、
**自分に届く前に**上流で捨ててもらう」ことである。
攻撃対象の /32 に BLACKHOLE(65535:666)を貼って上流へ広告すると、
上流はその宛先への経路を廃棄経路([第1部の Null ルート](../01_fundamentals/02_routing_table_basics.md)
の BGP 版)に向ける。そのアドレスへの正当なトラフィックも
道連れになるが、1アドレスを犠牲に回線と他のサービスを守る取引である。
なお /32 という「最も細かい経路」を受け入れる特例になるため、
上流は BLACKHOLE 付き経路に限り prefix-list の長さ制限を緩める —
本章の道具が総動員される場面でもある。

## 設定例(補助)

ウォークスルーの表の「顧客」行を FRRouting で実装する。
顧客 AS 64511(198.51.100.1、正当なプレフィックスは 203.0.113.0/24)と、
上流(192.0.2.9)を持つ構成である:

```text
ip prefix-list CUST-64511 seq 5 permit 203.0.113.0/24 le 25
!
bgp community-list standard FROM-CUSTOMER seq 5 permit 64500:100
!
route-map CUSTOMER-IN permit 10
 match ip address prefix-list CUST-64511
 set community 64500:100
 set local-preference 200
!
route-map TO-UPSTREAM permit 10
 match community FROM-CUSTOMER
!
router bgp 64500
 address-family ipv4 unicast
  neighbor 198.51.100.1 route-map CUSTOMER-IN in
  neighbor 192.0.2.9 route-map TO-UPSTREAM out
 exit-address-family
```

本章の理論がそのまま形になっていることを確認してほしい:

- CUSTOMER-IN は「登録済みプレフィックス(/24〜/25)なら、
  関係タグを貼り、LOCAL_PREF 200 で土俵に上げる」。受け皿の permit を
  **意図的に置いていない**ので、リスト外の経路は暗黙の deny で落ちる —
  顧客入口の鉄則(登録外は受けない)が、暗黙の deny の正しい使い方になる
- TO-UPSTREAM は「顧客タグの付いた経路だけ通す」。ピア・上流から学んだ経路は
  64500:100 を持たないので暗黙の deny で落ち、valley-free が保たれる

動作確認は荷札で引くのが早い。`show bgp ipv4 unicast community 64500:100` は
そのタグを持つ経路だけを一覧し、個別経路の詳細表示には Community 行が現れる:

```text
router# show bgp ipv4 unicast 203.0.113.0/24
  64511
    198.51.100.1 from 198.51.100.1 (203.0.113.1)
      Origin IGP, metric 0, localpref 200, valid, external, best (First path received)
      Community: 64500:100
```

「入口で貼った文脈を、AS 内のどこからでも読み出せる」というコミュニティの
働きが、そのまま確認コマンドの引数になっている。

## トラブルシューティング

### 症状: route-map を適用したら経路が全部消えた / そもそも1本も入らない

- まず**暗黙の deny** を疑う。「一部の経路に set したいだけ」の route-map に
  受け皿(match 句なしの permit エントリ)を置き忘れると、
  set 対象以外の全経路が落ちる。設定例の CUSTOMER-IN のように
  「落とすのが正しい」場面と、受け皿が必須の場面を意識的に区別する
- match 句が参照する prefix-list や community-list が**未定義・空**の場合の
  挙動は実装差がある(全マッチ扱いと全不一致扱いの両方が存在する)。
  リスト名の打ち間違い1つで「全部通る」「全部落ちる」のどちらかが
  黙って起こるので、参照先の存在確認は切り分けの最初期に行う
- ポリシーを何も設定していないのに経路が流れない場合は、
  RFC 8212 の既定(FRR の `bgp ebgp-requires-policy`)である。
  故障ではなく「関所に何も置いていないから何も通さない」という仕様であり、
  対処は無効化ではなく、意図を表すポリシーを明示的に置くことである

### 症状: コミュニティを貼ったのに相手に届いていない

- **送信設定**を確認する。コミュニティを広告に含めるかは設定項目であり、
  既定値が実装で異なる(Cisco IOS 系は既定で送らず
  `neighbor <ピア> send-community` が必要、FRR は既定で送る)。
  さらに standard / extended / large の**3種が別スイッチ**になっている
  実装が多く、「標準コミュニティは届くのにラージだけ消える」形で現れる
- 自分と相手の設定が正しくても、**経路上の第三者が洗い流す**ことがある。
  optional transitive という属性の性質上、原理的には世界を巡れるが、
  多くの ISP は境界で他組織のコミュニティを削除・選別するポリシーを
  持っている(自組織の名前空間の偽装や、荷札の氾濫を防ぐため)。
  隣までは合意で届けられるが、**遠くまで届く保証は仕様上もない**と考える

### 症状: ポリシーを変更したのに経路に反映されない

- 評価のタイミングの問題である。ポリシーは関所の通過時に評価されるので、
  受信済み・広告済みの経路には再評価の契機が要る。
  `clear bgp <ピア> soft in`(入力の再評価。Route Refresh または
  soft reconfiguration による)/ `soft out`(再広告)を実行する
- 自動で再評価する実装でも、反映は非同期である(経路数によっては時間がかかる)。
  「変更 → 即confirm」で観測すると未反映に見えることがあるため、
  再評価の完了を待ってから判断する

### 症状: prefix-list の ge / le の解釈を誤り、意図しない経路が通る・落ちる

- 最頻の誤解は `0.0.0.0/0 le 32` である。これは「デフォルトルートのみ」では
  なく「**すべての経路**」を意味する(上位 0 ビットの一致=無条件)。
  デフォルトルートのみは ge/le なしの `0.0.0.0/0` である
- 「10.0.0.0/8 の配下」のつもりの `permit 10.0.0.0/8`(ge/le なし)は
  **/8 ちょうど**にしかマッチせず、10.1.0.0/16 などの詳細経路はすべて
  暗黙の deny に落ちる。配下を含めるなら `le` の明示が必要である
- 逆に le を大きく取りすぎる(`le 32`)と、顧客からホストルートの
  雪崩を受け入れることになる。切り分けには FRR / Cisco 系の
  `show ip prefix-list <名前>`(マッチ回数の表示)と、
  疑わしい経路のプレフィックス長を1つずつ範囲条件に当てはめる
  地道な確認が確実である

## 演習・確認問題

1. BGP のポリシーの適用点2箇所を、3段の RIB(Adj-RIB-In / Loc-RIB /
   Adj-RIB-Out)との位置関係とともに図示せよ。また入力ポリシーが
   経路選択プロセスの Phase 1 とどう関係するかを述べよ。
2. `ip prefix-list X permit 172.16.0.0/12 ge 16 le 24` にマッチする経路の
   条件を正確に述べよ。また 172.16.0.0/12、172.20.4.0/24、172.16.1.1/32、
   172.32.0.0/16 のそれぞれがマッチするか判定せよ。
3. 「`0.0.0.0/0 le 32` はすべての経路にマッチする」のはなぜか。
   prefix-list のマッチ条件が2つの独立した検査の AND であることから説明せよ。
4. AS_PATH の正規表現 `^$` は何にマッチするか。またこれを出力ポリシーに
   使うことがすべての末端 AS に推奨されるのはなぜか、経路リークの
   観点から述べよ。
5. コミュニティの2つの使い方(情報タグ/アクション依頼)を、
   荷札を「貼る者」と「読む者」がどこにいるかの違いとして対比せよ。
   それぞれの実例を1つずつ挙げよ。
6. ラージコミュニティ(RFC 8092)が必要になった理由を、RFC 1997 の
   表記慣例と 32 ビット AS 番号(RFC 6793)の関係から説明せよ。
7. 商業関係に基づくポリシー(顧客/ピア/上流)における広告ルールを述べ、
   それが各関係の経済構造からどう導かれるかを説明せよ。また
   「入口でタグ・出口でタグを見る」実装パターンが、ピア数の増加に対して
   ポリシーの複雑さをどう抑えるかを述べよ。
8. RFC 8212 が eBGP の既定ポリシーを default deny に改めたのはなぜか。
   route-map の暗黙の deny と共通する設計原則を挙げて説明せよ。

## まとめ

- BGP のポリシーは「どこで(ピア・方向)・どの経路に(マッチ)・
  何をするか(アクション)」に分解でき、適用点は Adj-RIB-In の後と
  Adj-RIB-Out の前の2つの関所である
- 選別の道具は prefix-list(宛先と長さの範囲)と AS_PATH フィルタ(来歴)。
  評価は「先勝ち+暗黙の deny」で、これは route-map にも共通する
- コミュニティ(RFC 1997)は経路に貼る荷札であり、意味は貼る者と読む者の
  合意で決まる。入口で文脈を記録するタグ、相手のポリシーを遠隔操作する
  依頼(GRACEFUL_SHUTDOWN、BLACKHOLE)、型付きの拡張(RT)と
  32 ビット AS 対応(ラージコミュニティ)がある
- 実務ポリシーの骨格は商業関係(顧客/ピア/上流)から演繹される:
  顧客経路は全員へ、それ以外は顧客のみへ。LOCAL_PREF も同じ序列で付け、
  入口のタグ付けと出口のタグ選別で実装する
- ポリシーは関所の通過時にしか評価されない(再評価には Route Refresh /
  soft reconfiguration)。関所に何も置かないときの既定は RFC 8212 で
  default deny に反転した — 書き忘れは経路リークではなく不通で現れる
