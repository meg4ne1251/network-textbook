# 分類とマーキング — 荷札には何を書き、誰の荷札を信じるか

## 概要

この章では、QoS パイプラインの入口側 — 分類(classification)とマーキング
(marking)— を扱う。DSCP の値の体系(EF / AF / クラスセレクタ)と PHB の
関係、L2 の荷札である 802.1Q の PCP(いわゆる CoS)、層をまたぐ荷札の写像、
そして「誰のマークを信じるか」という信頼境界の設計までを見る。
前提は[前章(QoS の基礎)](01_qos_basics.md)であり、
[VLAN タグのフォーマット](../02_vlan_vxlan_evpn/01_vlan_basics.md)と
[MPLS ラベル](../05_mpls_srv6/01_mpls_basics.md)の知識を随所で使う。

## 導入 — そもそも何のための作業か

### 分類とマーキングは別の行為である

前章で見たとおり、DiffServ の骨格は「エッジで分類し、コアはマークを読んで
PHB を適用する」だった。この一文には、実は2つの異なる行為が含まれている。

- **分類** — このパケットはどのクラスに属するか、を**判断する**こと。
- **マーキング** — 判断の結果をヘッダの荷札に**書き留める**こと。

分類だけならどのホップでもできる。あらゆるルータが 5 タプルを見て
「これは音声だ」と毎回判断してもよい。ではなぜ書き留めるのか。

第一に、**深い分類はエッジでしかできない**からである。「このパケットは
経理部の IP 電話の音声だ」という判断には、送信元がつながるポート、
VLAN、アドレス、ポート番号といった文脈が要る。その文脈は端末に近い
場所にしかない。網の奥のコアルータには、毎秒数百万のパケットの1つ
1つについてそんな詮索をしている時間も情報もない。

第二に、**同じ判断を繰り返すのは無駄**だからである。エッジが一度
下した判断を荷札に書いておけば、以後の全ホップは 6 ビットの
DSCP を読むだけでよい。マーキングとは、**高価な分類の結果を
パケット自身にキャッシュする**行為である。

この構図は本書の読者には見覚えがあるはずだ —
[MPLS の FEC](../05_mpls_srv6/01_mpls_basics.md) である。入口で一度だけ
分類し、結果をラベルに焼き付け、以後は機械的に処理する。違いは
縮約する内容である。MPLS のラベルは「**どこへ**運ぶか」(転送先)を、
DSCP は「**どう**運ぶか」(輻輳時の扱い)を縮約する。だから両者は
排他ではなく重ねて使われる — MPLS ラベルの中に QoS 用の TC フィールドが
あるのは、この二重の縮約の現れである(後述)。

### 荷札は自己申告である — 信頼の問題

もう1つ、この章を貫く論点を先に出しておく。DSCP はパケットの
送り手側が書く**自己申告**であり、ヘッダのその他のフィールドと違って
**書いてあることの真偽を転送の仕組みが保証しない**。宛先アドレスが
嘘ならパケットは届かないが、DSCP が嘘でもパケットは届く —
むしろ良い席で届く。

つまり、荷札を読んで優遇する仕組みを入れた瞬間、
「誰の書いた荷札なら信じるか」という問題が必ず発生する。
端末は自分のトラフィックを最優先だと申告できてしまうし、
悪意がなくても OS やアプリケーションは思い思いの DSCP を付けてくる。
これに線を引くのが**信頼境界(trust boundary)**であり、
分類・マーキングの設計とは、値の選定と信頼境界の設定の2つを
合わせたものになる。

## 理論

### ToS オクテットの3世代 — 同じ8ビットの再定義史

IPv4 ヘッダの2バイト目 — RFC 791 の ToS オクテット、現在の
**DS フィールド** — は、同じ 8 ビットが 3 回意味を変えた歴史を持つ。
tcpdump が今も `tos` と表示するのはこの歴史の名残であり、
この読み替えを知らないと現場の 16 進数が読めない。

```text
  bit:      0     1     2     3     4     5     6     7
         +-----+-----+-----+-----+-----+-----+-----+-----+
RFC 791  |   Precedence    |  D  |  T  |  R  |  0  |  0  |  (1981)
         +-----+-----+-----+-----+-----+-----+-----+-----+
RFC 1349 |   Precedence    |        ToS            | MBZ |  (1992)
         +-----+-----+-----+-----+-----+-----+-----+-----+
RFC 2474 |             DSCP                  |    CU     |  (1998)
         +-----+-----+-----+-----+-----+-----+-----+-----+
RFC 3168 |             DSCP                  |    ECN    |  (2001)
         +-----+-----+-----+-----+-----+-----+-----+-----+
```

- **RFC 791(1981)** — 上位 3 ビットが **IP Precedence**(0〜7 の優先度。
  6 = Internetwork Control、7 = Network Control)、続く 3 ビットが
  遅延(D)・スループット(T)・信頼性(R)の希望フラグ。
  優先度の概念は最初からあったが、フラグ部分はほぼ使われなかった。
- **RFC 1349(1992)** — フラグ部分を 4 ビットの ToS 値に再定義したが、
  これも普及しなかった。
- **RFC 2474(1998)** — DiffServ の登場とともに全体を再設計。
  上位 6 ビットを **DSCP(Differentiated Services Codepoint)**とし、
  8 ビット全体を **DS フィールド**と改称した。IPv6 の Traffic Class
  オクテット([第4部の対比表](../04_ipv6/01_why_ipv6.md)で見た)にも
  同じ構造がそのまま適用される — つまり **DSCP の議論は IPv4/IPv6 共通**である。
- **RFC 3168(2001)** — 残りの下位 2 ビットが **ECN**(明示的輻輳通知)に
  割り当てられた。ECN は「捨てる代わりに混雑を予告する」仕組みであり、
  詳細は後述の章(`04_shaping_policing.md`)で扱う。

実務上の重要な帰結が1つある。tcpdump 等が表示する `tos` は
**8 ビット全体の 16 進値**であり、DSCP はその上位 6 ビットである。
したがって換算は:

```text
ToS バイト = DSCP × 4    (DSCP を左に2ビットシフト。ECN = 00 のとき)

例: CS6 = DSCP 48 → 48 × 4 = 192 = 0xc0   ← 前章の tcpdump で見た値
    EF  = DSCP 46 → 46 × 4 = 184 = 0xb8
```

前章の tcpdump 出力にあった `tos 0xc0` が DSCP 48(CS6)だったのは、
この ×4 の関係である。この換算を体で覚えておくことが、
QoS のトラブルシューティングの第一歩になる。

### DSCP と PHB — 名前空間と動作は別物

RFC 2474 の設計で最も誤解されやすいのは、**DSCP という値と
PHB(Per-Hop Behavior)という動作が別のものとして定義されている**
ことである。

- **DSCP** は 6 ビット(64 通り)の単なる名前空間である。
- **PHB** は「そのクラスのパケットを各ホップがどう扱うか」という
  転送動作の仕様である(RFC 2475。優先的に送る、一定の帯域を保証する、等)。
- 両者の対応 — どの値がどの動作を意味するか — は、原理的には
  **各 DiffServ ドメインが自由に決めてよい**。IETF が定めるのは
  「この PHB にはこの値を推奨する」という推奨コードポイントである。

なぜこんな二段構えにしたのか。それは前章で見たとおり、QoS が
自分のドメインの中でしか効かないからである。ドメインごとにクラス設計は
異なる(4 クラスの事業者もあれば 8 クラスの企業網もある)から、
値の意味を世界共通に固定しても守られない。それなら「動作の部品」だけを
標準化し、値との対応はドメインの設計に委ね、境界では再マーキングで
翻訳する — という割り切りである。とはいえ実務では、相互運用のために
推奨コードポイントをそのまま使うのが圧倒的多数であり、以下で見る
EF = 46 などの対応は事実上の共通語彙になっている。

標準化されている PHB は次の4系統である。

**① Default PHB(DF、DSCP 0 = 000000)— RFC 2474**

何も約束しないベストエフォート。マークされていないパケットの既定値であり、
「DSCP 0 のパケットを差別してはならない(通常のベストエフォートは
受けられる)」ことだけが規定される。

**② クラスセレクタ PHB(CS0〜CS7 = xxx000)— RFC 2474**

下位 3 ビットが 000 の 8 個の値。これは **IP Precedence との後方互換**の
ために予約された系列である — 上位 3 ビットの位置が Precedence と
一致するため、CS6(110000)は Precedence 6 としか解釈できない古い機器にも
「Internetwork Control」として通じる。要求される動作も「番号が大きいほど
劣らない扱い」という緩やかな順序のみ。前章で見た**制御プレーンの CS6** は
この系列であり、ルーティングプロトコルが 1990 年代の機器とも通じる
最も保守的な荷札を選んでいることが分かる。

**③ EF(Expedited Forwarding、DSCP 46 = 101110)— RFC 3246(旧 RFC 2598)**

「低遅延・低ジッタ・低損失」— 仮想的な専用線を目指す PHB。
実装のイメージは「EF のキューは常にほぼ空であるべし」、すなわち
EF パケットの到着レートを送出能力が常に上回るように運用する。
音声(VoIP)のための席だと考えてよい。

ただし EF には**毒**がある。厳密優先で送る席は、そこに大量のトラフィックが
流れ込むと**他のすべてのクラスを飢えさせる**。だから RFC 3246 は
EF に流入制限(ポリシング)を課すことを前提にしている —
特等席には定員がある。この「優先とポリシングはセット」という規律は
後述の章(`04_shaping_policing.md`)の主題につながる。

**④ AF(Assured Forwarding)— RFC 2597**

最も構造的な PHB 群。**4 つの独立なクラス × 3 段階の廃棄優先度**の
行列として定義される。

| | 廃棄優先度 低(輻輳時に最後まで残る) | 中 | 高(輻輳時に最初に捨てられる) |
|---|---|---|---|
| **AF1x**(クラス1) | AF11 = 10 | AF12 = 12 | AF13 = 14 |
| **AF2x**(クラス2) | AF21 = 18 | AF22 = 20 | AF23 = 22 |
| **AF3x**(クラス3) | AF31 = 26 | AF32 = 28 | AF33 = 30 |
| **AF4x**(クラス4) | AF41 = 34 | AF42 = 36 | AF43 = 38 |

値は `AFxy` と読み、x がクラス、y が廃棄優先度。DSCP 値は
**8x + 2y** で機械的に計算できる(AF31 = 8×3 + 2×1 = 26)。

2つの軸は役割が違う。**クラス間(AF1x と AF2x)は別々のキュー・
別々の資源配分**を受ける独立な席である。一方**クラス内の廃棄優先度は
同じキューの中での「捨てられやすさ」**であり、輻輳したとき
AFx3 → AFx2 → AFx1 の順に犠牲になる。「同じクラスの中に
捨てられやすさの階級を作る」ことの用途は次章と 04 章で本領を発揮する —
契約帯域内のトラフィックは AFx1、超過分は捨てずに AFx2/AFx3 へ
**降格マーキング**して通し、輻輳したときだけ超過分から死んでいく、
という使い方である(三色マーカー。後述: `04_shaping_policing.md`)。
廃棄優先度を読んで実際に選択的に捨てる機構(WRED)は
次章(`03_queuing_scheduling.md`)で扱う。

**その他 — LE(Lower Effort、DSCP 1 = 000001)— RFC 8622**

「ベストエフォート**より下**」を明示する PHB。夜間バックアップや
OS アップデートなど「帯域が余っているときだけ流れてほしい」
トラフィックのための席で、伝統的に CS1 が慣用されてきた
スカベンジャークラスの役割を正式に置き換えるものである(RFC 8622 は
RFC 4594 を更新して CS1 の位置づけを改めた)。

### RFC 4594 — 値の語彙集

どのトラフィックにどの値を使うかの体系的な指針が
**RFC 4594(Configuration Guidelines for DiffServ Service Classes)**である。
Informational(実装や運用への要求ではない)だが、実務のクラス設計は
おおむねこの語彙の部分集合になる。抜粋:

| サービスクラス | 推奨 DSCP | 例 |
|---|---|---|
| Network Control | CS6(48) | ルーティングプロトコル(前章の伏線) |
| Telephony | EF(46) | VoIP 音声 |
| Signaling | CS5(40) | SIP 等の呼制御 |
| Multimedia Conferencing | AF41〜43 | ビデオ会議 |
| Real-Time Interactive | CS4(32) | 対話型リアルタイム(ゲーム等) |
| Multimedia Streaming | AF31〜33 | VoD ストリーミング |
| Broadcast Video | CS3(24) | 放送型映像 |
| Low-Latency Data | AF21〜23 | 対話型業務データ |
| OAM | CS2(16) | 運用管理(SNMP 等) |
| High-Throughput Data | AF11〜13 | バルク転送 |
| Standard | DF(0) | 既定 |
| Low-Priority Data | LE(1)※ | スカベンジャー(※ RFC 8622 による更新) |

12 クラスをすべて使う網はまれで、実務は「EF / CS6 / AF 数クラス / DF」の
4〜6 クラス程度に集約するのが普通である。重要なのは全部を使うことではなく、
**自ドメインの中で語彙を1つに決め、文書化し、全機器で一貫させる**ことである。

### 分類の2方式 — BA と MF

RFC 2475 は分類器(classifier)を2種類に整理している。
この区別は「どこで何をすべきか」の指針としてそのまま役に立つ。

- **BA 分類器(Behavior Aggregate)** — DSCP **だけ**を見る。
  荷札を読むだけなので高速・無状態。**コア向き**。
- **MF 分類器(Multi-Field)** — 5 タプル(送信元/宛先アドレス、
  プロトコル、ポート)、入力インタフェース、VLAN など**複数フィールド**を
  見る。文脈を使える代わりに高価。**エッジ向き**。

DiffServ の分業を分類器の言葉で言い直せば、
「**エッジで MF 分類してマーキングし、コアは BA 分類だけで済ませる**」
となる。

ここで現代的な注意がある。MF 分類の古典的な頼みの綱だった
「ポート番号でアプリを見分ける」は、**通信の暗号化と HTTPS への集約で
効きにくくなっている**。あらゆるものが TCP/UDP 443 に載る時代、
ペイロードを覗く深い分類(DPI)も暗号化の前には無力である。
この流れは、分類の重心を**網の中から端点へ**押し戻しつつある —
アプリケーション自身(または端末 OS)が送信時に正しい DSCP を
マークし(Linux なら `setsockopt(IP_TOS)`、Windows ならグループ
ポリシーの QoS 設定)、網はそれを信頼境界の設計で受け止める、
という形である。「何のトラフィックかを一番よく知っているのは
送り手自身」なのだから、これは理にかなった揺り戻しでもある。

### L2 の荷札 — PCP(802.1p、CoS)

[第2部で見た 802.1Q タグ](../02_vlan_vxlan_evpn/01_vlan_basics.md)の
TCI の先頭 3 ビットが **PCP(Priority Code Point)**である。

```text
   802.1Q タグ(TCI 部分)
   +-----------+-----+---------------------------+
   | PCP (3bit)| DEI | VID (12bit)               |
   +-----------+-----+---------------------------+
```

「802.1p」という呼び名は、この優先度機構を導入した 1990 年代の
802.1D への追補プロジェクトの名前であり、現在の仕様上は 802.1Q 本体の
一部である(仕様に存在しない慣用名の常として、由来を知っていれば
混乱しない)。Cisco 系の文書では **CoS(Class of Service)**とも
呼ばれるが、本書は仕様の用語である PCP を標準表記とする。

PCP は 3 ビット = 8 値。802.1Q が示す既定の用途(traffic type)は
次のとおりで、DSCP のクラスセレクタとよく似た序列を作る。

| PCP | 略号 | 用途 | 備考 |
|---|---|---|---|
| 7 | NC | Network Control | |
| 6 | IC | Internetwork Control | |
| 5 | VO | Voice(遅延 < 10ms 級) | |
| 4 | VI | Video(遅延 < 100ms 級) | |
| 3 | CA | Critical Applications | |
| 2 | EE | Excellent Effort | |
| 0 | BE | **Best Effort(既定)** | タグ付けの既定値 |
| 1 | BK | Background | **0 より低い最下位** |

表の下2行に有名な癖がある。**序列の最下位は 0 ではなく 1**である。
「何も考えずに付く既定値(0)」と「意図的に最下位を申告する値」を
分けたかったが、既定値を 0 以外にはできなかった — 結果として
1(Background)だけが 0 の下に潜り込む形になった。DSCP の
LE(RFC 8622)と同じ「ベストエフォート未満」の思想が、L2 では
番号の折れ曲がりとして現れている。

PCP の最重要の性質は、**荷札の寿命がタグの寿命と同じ**ことである。

- PCP は 802.1Q タグの中にしか存在しない。**タグなしフレームに
  PCP という概念はない**。アクセスポートでタグが外れれば消える。
- ルータが L3 転送すればフレームは作り直され、タグも PCP も残らない。

つまり PCP は**隣のスイッチまでしか届かない、リンクローカルな荷札**である
(BPDU や LACPDU がリンクローカルだったのと同じ感覚で捉えるとよい)。
エンドツーエンドの意図を運ぶ主役は常に DSCP であり、PCP は
「IP ヘッダを見ない(見られない)L2 機器のために、DSCP の要約を
タグに転記したもの」と位置づけるのが正しい。IP を持たない
ストレージ系プロトコルなど「DSCP を持てないトラフィック」の荷札としては
PCP が唯一の手段になる。

PCP の隣の **DEI(Drop Eligibility Indicator)**は「廃棄されてよい」
ことを示す 1 ビットで、AF の廃棄優先度の L2 版に相当する
(輻輳時に DEI = 1 から捨てる)。[第2部](../02_vlan_vxlan_evpn/02_trunking_native_vlan.md)で
CFI からの再定義の経緯に触れたフィールドの、これが本来の用途である。
契約超過分に DEI を立てる使い方は 04 章の三色マーカーと同じ思想である。

### 層をまたぐ写像 — 荷札のリレー

前章の「マークの座席」の表を思い出してほしい。DSCP(L3)、PCP(L2)、
MPLS TC(ラベル)— 座席が層ごとにあるのは、処理する機器が見る層が
違うからだった。すると必然的に、**層の境界で荷札を引き継ぐ写像**が
必要になる。

好都合なことに、PCP も MPLS TC も 3 ビットであり、DSCP の上位 3 ビット
(= クラスセレクタの系列 = 旧 Precedence)と幅が揃っている。
だから既定の写像は素直である:

```text
   DSCP(6bit)の上位3ビット ──┬──→ PCP(3bit)
                              └──→ MPLS TC(3bit)

   例: EF = 101110 → 上位3ビット 101 = 5 → PCP 5(VO)/ TC 5
       CS6 = 110000 → 110 = 6 → PCP 6(IC)/ TC 6
```

逆向き(3 ビット → 6 ビット)は情報が足りないことに注意する。
PCP 2 から戻せるのは CS2 までであり、それが元々 AF21 だったのか
AF23 だったのかは失われている — **写像は不可逆**であり、
特に AF の廃棄優先度は 3 ビットの座席を通ると消える。
これが「エンドツーエンドの意図は DSCP に持たせ、L2/MPLS の荷札は
区間ごとの要約とみなす」べきもう1つの理由である。

**トンネルと外側ヘッダ** — カプセル化はこの問題の変奏である。
[VXLAN](../02_vlan_vxlan_evpn/03_vxlan_fundamentals.md) の外側 IP ヘッダ、
MPLS のラベル、IPsec の外側ヘッダ — アンダーレイの機器が見るのは
外側の荷札だけであり、輻輳が起きるのはまさにそのアンダーレイである。
内側の DSCP がどれほど正しくても、外側に引き継がれていなければ
トンネルの中身は全部ひとまとめのベストエフォートになる。

RFC 2983 はトンネルの荷札の扱いを2つのモデルに整理している。

- **uniform モデル** — トンネルは透明であるべきという立場。
  入口で内側 DSCP を外側へコピーし、出口では(トンネル区間で外側が
  再マーキングされていれば)それを内側へ書き戻す。網全体が
  1つの DiffServ ドメインである場合の自然な選択。
- **pipe モデル** — トンネルは別ドメインを通る土管だという立場。
  外側の値はトンネル運用者のポリシーで独立に決め、出口では
  外側を捨てて内側の値をそのまま使う。事業者網をまたぐ VPN の
  自然な選択(顧客のマークは触らず運び、事業者の扱いは事業者の
  クラス設計で決まる)。

MPLS でも同じ問いが立ち、RFC 3270 が uniform / pipe(と short-pipe)を
定義している。なお MPLS には
[PHP の章で触れた罠](../05_mpls_srv6/01_mpls_basics.md)がある —
最終ホップ手前でラベルが pop されると **TC も一緒に消える**ため、
TC を出口まで届けたい場合は explicit NULL を使う必要があった。
荷札はカプセル化と共に生まれ、デカプセル化と共に死ぬ、
という原則の好例である。

### 信頼境界 — どこから先の申告を信じるか

理論の最後に、冒頭で予告した信頼の問題に答えを与える。

RFC 2475 は、DiffServ ドメインの縁で行う処理を
**トラフィック条件付け(traffic conditioning)**と呼び、
分類器 → メータ → マーカー → シェーパ/ドロッパという部品の連鎖で
定義している(メータ以降は 04 章の主題)。このうち境界の分類・
マーキングが担うのが、外から来た荷札の**検認**である。
選択肢は3つしかない。

1. **信頼する(trust)** — 付いてきた DSCP/PCP をそのまま受け入れる。
2. **書き換える(re-mark)** — 自ドメインの語彙へ翻訳する。
3. **洗い流す(bleach)** — 0(DF)に上書きし、必要なら自分で
   MF 分類してマーキングし直す。

そして設計の定石は次のようにまとめられる。

- **端末(PC、サーバの一般ポート)は信頼しない。** ユーザ端末の申告を
  信じることは、優先席の切符を自由に自己発行させることと同じである。
  境界(アクセススイッチ)で洗い流し、網の側の MF 分類で必要な
  マークを付け直す。
- **管理された機器は例外的に信頼する。** IP 電話が代表で、
  「電話機の付けた EF/CS5 は信頼するが、電話機の背面ポートに
  つながった PC のマークは信頼しない」という条件付き信頼が典型。
  電話を LLDP 等で認証・識別して初めて信頼を与える実装が多い。
- **自ドメインの内部リンク(スイッチ間・ルータ間)は信頼する。**
  信頼境界の内側では、エッジで付けたマークこそが分類の成果物であり、
  これを毎ホップ疑っては DiffServ の分業が成立しない。
- **他ドメインとの境界は契約で決める。** 事業者は顧客の DSCP を
  既定では信頼せず、契約したクラス数へ再マーキング(または pipe モデルで
  透過)する。逆に自分が顧客の立場なら、「どの DSCP で渡せば
  どのクラスに乗るか」は事業者のサービス仕様で確認する事項である
  ([L3VPN](../05_mpls_srv6/03_l3vpn_l2vpn.md) の QoS オプションが
  まさにこれである)。

一言でいえば、**信頼境界はできるだけ端に置くが、管理の及ぶ範囲より
外には出さない**。境界が端に近いほど分類の文脈は豊かになり、
網の奥の仕事は軽くなる。しかし管理外の機器まで信頼した瞬間、
QoS はポリシーではなく自己申告制になる。

なお、機器の設定語彙としての「trust dscp」「trust cos」や、
ポートの既定が信頼か非信頼か(プラットフォームにより異なる)は
ベンダー実装の話であり、本書では立ち入らない。重要なのは
「すべてのポートについて、信頼するのか・書き換えるのか・洗い流すのかが
設計として決まっている」ことである。決めていないポートの既定動作は、
プラットフォーム次第で「全部信頼」かもしれない。

## プロトコル動作の詳細

### 3つの座席の位置 — ワイヤ上のどこにあるか

分類・マーキングの対象フィールドの位置を、ワイヤ上のフレームとして
1枚にまとめておく(VLAN タグ付き・MPLS 1 段の場合)。

```text
 +----------------+------+----------------+-----------------+----------------
 | 宛先/送信元 MAC | 802.1Q タグ          | (MPLS ラベル)   | IP ヘッダ
 |                | PCP|DEI|VID           | Label|TC|S|TTL  | ...|DSCP|ECN|...
 +----------------+------+----------------+-----------------+----------------
                    ↑ 3bit                   ↑ 3bit             ↑ 6bit
                    L2 スイッチが見る         MPLS LSR が見る    ルータが見る
                    (タグと共に生滅)         (ラベルと共に生滅) (末端まで生存)
```

- **DSCP** は IP ヘッダにあり、[L3 アドレスと同様に](../01_fundamentals/01_l2_l3_recap.md)
  原則として末端まで書き換えられずに生き残る(書き換えるのは
  ポリシーの意思だけ)。
- **PCP/DEI** はタグと運命を共にする。タグなし区間・L3 ホップで消える。
- **MPLS TC** はラベルと運命を共にする。push で生まれ pop で消える。

### DSCP 値の総覧 — 現場の換算表

理論で見た語彙を、現場で目にする全表記(2進・10進・ToS 16進)で
1枚にまとめる。tcpdump の `tos 0x??` をこの表で引けるようにしておくと、
トラブルシューティングが速くなる。

| 名前 | DSCP(2進) | DSCP(10進) | ToS バイト(16進) | 意味 |
|---|---|---|---|---|
| DF(CS0) | 000000 | 0 | 0x00 | ベストエフォート(既定) |
| LE | 000001 | 1 | 0x04 | BE 未満(RFC 8622) |
| CS1 | 001000 | 8 | 0x20 | (旧スカベンジャー慣用) |
| AF11 / 12 / 13 | 001010/001100/001110 | 10 / 12 / 14 | 0x28 / 0x30 / 0x38 | 大容量データ |
| CS2 | 010000 | 16 | 0x40 | OAM |
| AF21 / 22 / 23 | 010010/010100/010110 | 18 / 20 / 22 | 0x48 / 0x50 / 0x58 | 低遅延データ |
| CS3 | 011000 | 24 | 0x60 | 放送映像 |
| AF31 / 32 / 33 | 011010/011100/011110 | 26 / 28 / 30 | 0x68 / 0x70 / 0x78 | ストリーミング |
| CS4 | 100000 | 32 | 0x80 | リアルタイム対話 |
| AF41 / 42 / 43 | 100010/100100/100110 | 34 / 36 / 38 | 0x88 / 0x90 / 0x98 | ビデオ会議 |
| CS5 | 101000 | 40 | 0xa0 | シグナリング |
| **EF** | **101110** | **46** | **0xb8** | **音声** |
| **CS6** | **110000** | **48** | **0xc0** | **ネットワーク制御** |
| CS7 | 111000 | 56 | 0xe0 | (予約的。通常は使わない) |

ToS 16進の下位 2 ビットが 00 でない値(例: `0xb9`、`0x2f`)を見たら、
それは ECN ビットが立っている(輻輳通知が動いている)ことを意味する。
DSCP だけを知りたければ下位 2 ビットを無視して 4 で割ればよい。

### ウォークスルー — 音声パケットの荷札のリレー

IP 電話の音声パケット(RTP)が、企業網から事業者 MPLS 網(L3VPN)を
通って対向拠点に届くまで、どの荷札がどこで生まれ・読まれ・消えるかを
追跡する。

```text
 [IP電話]──(音声VLANタグ付き)──[アクセスSW]──trunk──[L3SW]──(タグなし)──[WANルータ CE]
                                                                        │
 [対向拠点]──…──[CE]──(タグなし)──[PE]═══════ MPLS L3VPN ═══════[PE]──┘
```

| # | 区間/地点 | PCP | DSCP | MPLS TC | 何が起きるか |
|---|---|---|---|---|---|
| 1 | 電話 → アクセス SW | **5**(電話が付ける) | **EF**(電話が付ける) | — | 荷札の誕生。電話は音声 VLAN のタグ付きで送る |
| 2 | アクセス SW(**信頼境界**) | 5 | EF | — | LLDP で電話と確認済みのポートなので条件付き信頼。同ポートの PC 由来フレームは DSCP 0 へ洗い流し |
| 3 | トランク区間 | 5 | EF | — | L2 機器は PCP を読んで優先キューへ。DSCP は素通り |
| 4 | L3 SW → WAN ルータ(タグなし) | **消滅** | EF | — | L3 転送でフレーム再生成。以後の分類は DSCP のみが頼り |
| 5 | CE → PE(事業者との境界) | — | EF | — | 契約: 「EF は音声クラスに載る」ことをサービス仕様で確認済み |
| 6 | PE(MPLS 入口) | — | EF(不変) | **5**(PE が写像) | pipe モデル: 顧客 DSCP には触れず、TC 5 で事業者の音声クラスへ |
| 7 | MPLS コア(P) | — | (見ない) | 5 | P ルータは TC だけで PHB を適用(BGP フリーコアは IP を見ない) |
| 8 | 出口 PE → CE | — | EF | 消滅 | pop で TC は消える。内側 DSCP がそのまま現れる(pipe) |
| 9 | 対向拠点内 | (再び付与されうる) | EF | — | 拠点のエッジが DSCP → PCP を写像し直す |

この表から3つの教訓が読める。第一に、**端から端まで生き続ける荷札は
DSCP だけ**である(それも「触らない」というポリシーの合意があってこそ)。
第二に、PCP と TC は**区間ごとに DSCP から再生成される使い捨ての要約**である。
第三に、リレーのどこか1箇所 — 信頼境界の設定漏れ、写像の設定漏れ、
契約クラスの認識違い — が切れると、そこから先の全区間で意図が失われる。
QoS が「全ホップの合意」だと言われるのはこのためである。

## 設定例 — Linux でマーキングと荷札を観察する

原理の確認として、Linux でマーキングを行い、荷札の変化を観察する。

**nftables で分類・マーキングする**(エッジルータの MF 分類の最小形):

```bash
# SIP(5060/udp)を CS5、RTP の範囲を EF にマーキング
nft add table inet qos
nft add chain inet qos post '{ type filter hook postrouting priority mangle ; }'
nft add rule inet qos post udp dport 5060 ip dscp set cs5
nft add rule inet qos post udp dport 16384-32767 ip dscp set ef

# 信頼しない端末からのマークを洗い流す例(LAN から来たものを 0 へ)
nft add rule inet qos post iifname "eth1" ip dscp != 0 ip dscp set 0
```

nftables は `ef`、`cs5`、`af41` といった**推奨コードポイント名を
そのまま語彙として持っている**。標準の語彙が実装の設定言語に
なっている好例である(最後の行のような「全部洗い流し」は、実際には
信頼する例外をこの規則の前に置いてから使う)。

**マークを確認する** — tcpdump の `-v` で tos を読む:

```bash
$ tcpdump -v -n -i eth0 udp port 5060 -c 1
IP (tos 0xa0, ttl 64, id 4711, offset 0, flags [DF], proto UDP (17), length 628)
    192.0.2.10.5060 > 198.51.100.20.5060: SIP ...
```

`tos 0xa0` → 0xa0 ÷ 4 = 40 = CS5。換算表のとおりである。

**試験パケットで信頼境界を探す** — ping は任意の ToS バイトを付けられる:

```bash
$ ping -Q 0xb8 -c 5 198.51.100.20      # EF(DSCP 46 → ToS 0xb8)を付けて送る
```

これを経路の各区間で tcpdump しながら送ると、**どこで荷札が
書き換えられた(洗い流された)かが特定できる**。`-Q` に渡すのは
DSCP 値ではなく ToS バイト全体である点に注意(後述の症状1)。

**L2 の荷札(PCP)を観察する** — `-e` でリンク層ヘッダを表示する:

```bash
$ tcpdump -e -v -n -i eth0.30 -c 1
00:11:22:33:44:55 > 66:77:88:99:aa:bb, ethertype 802.1Q (0x8100), length 632:
    vlan 30, p 5, ethertype IPv4, (tos 0xb8, ttl 64, ...)
```

`vlan 30, p 5` の `p 5` が PCP である。DSCP(tos 0xb8 = EF)と PCP 5 の
両方の荷札が確認でき、写像(EF の上位 3 ビット = 5)が機能していることが
読み取れる。なお Linux で DSCP → PCP の写像を担うのは VLAN デバイスの
`egress-qos-map`(内部的には skb の priority を介する)だが、
その配管の詳細は本書の範囲を超えるので触れない。

## トラブルシューティング

### 症状1: EF を付けたつもりが、別の値になっている

**×4 の換算ミス**が最も多い。「EF = 46 だから」と `ping -Q 46` と
打つと、46 = 0x2e は ToS バイトとして解釈され、DSCP は 0x2e の
上位 6 ビット = **11**(意図しない値)、しかも下位 2 ビット `10` が
ECN の ECT(0) として立つ。逆に、設定 UI が DSCP(0〜63)を求めている
ところに ToS バイト(0xb8 = 184)を入れて範囲エラーになる事故もある。
**「この欄はどちらの単位か」を常に確認し、tcpdump の tos 表示は
4 で割ってから語彙表を引く** — これだけで荷札の誤読はなくなる。

### 症状2: エッジで付けた DSCP が、途中で消えている/変わっている

送信側では確かに EF なのに、受信側で tos 0x00 になっている。
`ping -Q` の試験パケットを使い、**区間ごとに tcpdump して
荷札が変わる地点を挟み撃ちで特定する**のが基本動作である。
典型的な犯人は次の順で疑う。

1. **意図した信頼境界** — アクセススイッチやファイアウォールが
   設計どおり洗い流している(その場所で信頼 or 再マーキングする設計に
   直すべきかを判断する)。
2. **意図しない既定動作** — 機器の既定が「非信頼」で、QoS 機能を
   有効にした瞬間に全ポートで洗い流しが始まった、という形。
   信頼設定はキュー設定と別項目であることが多い。
3. **事業者網** — インターネット経由なら DSCP が無傷で届く保証は
   そもそもない([前章](01_qos_basics.md)のとおり、QoS は自ドメインの
   中の道具である)。閉域網(L3VPN 等)なら契約のクラス写像を確認する。

### 症状3: PCP(CoS)ベースで設計したら、荷札が途中で消えた

L2 スイッチ網で PCP を頼りに優先制御を組んだが、経路の途中に
アクセスポート(タグなし)区間、あるいは L3 ホップが1つあり、
そこから先で全フレームが PCP なし(= 0 扱い)になっていた —
という形の事故。原因は障害ではなく PCP の仕様である。
**PCP はタグの中にしか存在せず、タグなし区間と L3 転送を越えられない。**
対策は、意図をエンドツーエンドで運ぶ荷札を DSCP に一本化し、
PCP は「DSCP から区間ごとに再生成する要約」と位置づけ直すこと
(L2 区間の入口ごとに DSCP → PCP の写像を設定する)。
逆に、IP を持たないトラフィックを扱う区間では PCP しか荷札がないので、
その区間にタグなしリンクを作らないことが設計条件になる。

### 症状4: トンネルを越えた区間だけ QoS が効かない

拠点間 VPN(VXLAN、IPsec、GRE 等)を導入した途端、トンネル経由の
音声だけ品質が落ちた。中身の DSCP は正しい — しかし**外側ヘッダの
DSCP が 0** であり、アンダーレイの輻輳点ではトンネル全体が
ベストエフォートの1本のフローに見えていた、という構図である。
確認は簡単で、トンネル区間を tcpdump し、**外側 IP ヘッダの tos** を
読む(VXLAN なら UDP 4789 のパケットの tos)。対策はカプセル化点で
内側 → 外側のコピー(uniform モデル相当。Linux の vxlan/gre には
外側 ToS を内側から継承させる設定がある)か、トンネル自体を
1つのクラスとして外側に固定マークする(pipe モデル相当)かを
**意図として選んで**設定することである。MPLS 網では同型の問題が
PHP で起きる — 最終ホップ手前で TC が
[ラベルごと消える](../05_mpls_srv6/01_mpls_basics.md)ため、
出口直前のリンクだけクラスが失われる(explicit NULL が処方箋)。

## 演習・確認問題

**問1.** 分類とマーキングの違いを述べ、「深い分類はエッジで行い、
結果をマーキングする」という分業が合理的である理由を2つ挙げよ。

**問2.** tcpdump で `tos 0x68` と表示された。DSCP の値と名前、
およびそのクラスの意味(RFC 4594 の語彙で)を答えよ。また、
`tos 0x6a` と表示された場合、追加で何が読み取れるか。

**問3.** AF31 と AF33 の違い、および AF31 と AF41 の違いを、
「キュー」と「廃棄」という言葉を使って説明せよ。DSCP 値も計算せよ。

**問4.** クラスセレクタ(CSx)の下位 3 ビットが 000 に固定されている
理由を、IP Precedence との関係から説明せよ。制御プレーンの慣行値
CS6 がこの系列から選ばれていることの利点にも触れること。

**問5.** PCP(802.1p CoS)だけを頼りに QoS を設計してはならない
理由を、PCP という荷札の「寿命」の観点から説明せよ。また、
PCP にしか荷札を書けないトラフィックとは何か。

**問6.** VXLAN で拠点間を結んだ網で、内側パケットの DSCP は正しいのに
アンダーレイの輻輳点で音声が劣化する。原因と、RFC 2983 の2つの
トンネルモデルそれぞれに基づく対策を述べよ。

---

**解答**

**問1.** 分類は「このパケットがどのクラスに属するか」を判断する行為、
マーキングは判断結果を DSCP 等の荷札としてヘッダに書き留める行為である。
分業が合理的な理由: (1) 深い分類(MF 分類)に必要な文脈 — どのポート・
どの VLAN・どの端末から来たか — はエッジにしかなく、コアでは
得られない。(2) エッジで一度下した判断を荷札にキャッシュすれば、
以後の全ホップは 6 ビットを読む BA 分類だけで済み、高速・無状態で
処理できる(状態はパケット自身が運ぶ、という DiffServ の原則)。

**問2.** 0x68 ÷ 4 = 26 = **AF31**。RFC 4594 の語彙では Multimedia
Streaming(ストリーミング系)のクラスで、AF3x クラスの中で最も
捨てられにくい(廃棄優先度・低)。`tos 0x6a` なら下位 2 ビットが
`10` — DSCP は同じ AF31 のまま、**ECN の ECT(0)** が立っており、
そのフローが ECN 対応(輻輳を廃棄でなく通知で受け取れる)であることが
読み取れる。

**問3.** AF31 と AF33 は**同じクラス(AF3x)= 同じキュー**に入り、
同じ資源配分を受けるが、輻輳時には廃棄優先度の高い AF33 から
先に捨てられる(クラス内の捨てられやすさの階級)。AF31 と AF41 は
**別のクラス = 別のキュー**であり、資源配分そのものが独立している
(どちらが優遇されるかはドメインのクラス設計次第)。DSCP は
8x + 2y より AF31 = 26、AF33 = 30、AF41 = 34。

**問4.** RFC 2474 は DS フィールドの上位 3 ビットが RFC 791 の
IP Precedence と同じ位置になるよう DSCP を設計し、下位 3 ビットが
000 の 8 値をクラスセレクタとして予約した。CSx の上位 3 ビットは
Precedence の x と一致するため、DSCP を解釈しない古い機器にも
「Precedence x」として意味が通じる(後方互換)。CS6 は Precedence 6
(Internetwork Control)として 1990 年代の機器にも通じる最も保守的な
選択であり、ルーティングプロトコルの荷札として機器の新旧を問わず
機能することが利点である。

**問5.** PCP は 802.1Q タグの中にしか存在しないため、荷札の寿命が
タグの寿命と等しい。タグなし(アクセス/ネイティブ VLAN)区間では
そもそも存在せず、ルータが L3 転送すればフレームの再生成と共に消える。
つまり PCP はリンク(L2 区間)ローカルな荷札であり、エンドツーエンドの
意図を運ぶことはできない — 意図は DSCP に持たせ、PCP は L2 区間ごとに
DSCP から再生成する要約とすべきである。PCP にしか荷札を書けないのは
**IP を持たないトラフィック**(非 IP のストレージ/制御プロトコル等)
である。

**問6.** 輻輳が起きるアンダーレイの機器が見るのは VXLAN の**外側**
IP ヘッダであり、カプセル化時に内側 DSCP が外側へ引き継がれていなければ、
トンネル全体が DSCP 0 の単一フローとして扱われるからである。
対策は、(uniform モデル)VTEP のカプセル化時に内側 DSCP を外側へ
コピーし、アンダーレイにも同じクラス体系の PHB を設定する。
(pipe モデル)トンネルを1つのサービスとして外側に固定の DSCP を
マークし、アンダーレイはトンネル単位のクラスで扱う(内側の細かい
クラス分けはトンネル内では効かなくなることを設計として受け入れる)。

## まとめ

- マーキングとは、エッジでしかできない高価な分類(MF 分類)の結果を
  パケット自身にキャッシュする行為である。以後のホップは DSCP を読む
  BA 分類だけで済む — MPLS の FEC と同型の「エッジで一度だけ」の分業。
- DSCP(6 ビット)は名前空間、PHB は動作であり、対応はドメインごとに
  決めてよい(実務は推奨コードポイントに従う)。語彙は EF(46、音声・
  ポリシングとセット)、AF(4 クラス × 3 廃棄優先度、8x + 2y)、
  CS0〜7(Precedence 後方互換。CS6 = 制御プレーン)、DF、LE。
  tcpdump の tos は ToS バイトであり DSCP × 4 で読む。
- 荷札は層ごとにあり、寿命が違う。DSCP は末端まで生きるが、
  PCP(802.1Q、3 ビット)はタグと共に、MPLS TC はラベルと共に生滅する
  リンク/区間ローカルな要約であり、DSCP の上位 3 ビットから
  区間ごとに再生成する(逆写像は不可逆)。
- トンネルでは外側ヘッダの荷札がアンダーレイでの扱いを決める。
  内側からコピーする uniform モデルと、トンネル運用者が独立に決める
  pipe モデル(RFC 2983 / MPLS は RFC 3270)から意図して選ぶ。
- DSCP は自己申告であるため、境界での検認 — 信頼・再マーキング・
  洗い流し — が必須である。信頼境界はできるだけ端末の近くに、
  ただし管理の及ぶ範囲の内側に引く。端末は信頼せず、管理された
  機器(IP 電話)は条件付きで信頼し、他ドメインとは契約で合意する。
