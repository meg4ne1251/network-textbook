# キューイングとスケジューリング — 待たせ方と捨て方の設計

## 概要

この章では、QoS パイプラインの出力側の心臓部 — キューイング(どの列に積み、
満杯なら誰を捨てるか)とスケジューリング(どの列から先に送るか)— を扱う。
厳密優先(PQ)・WFQ・DRR というスケジューラの系譜、テールドロップの病理、
RED/WRED と AQM、バッファブロートと CoDel/fq_codel までを見る。
前提は[第7部の01章](01_qos_basics.md)(輻輳・遅延の4成分・パイプライン)と
[02章](02_classification_marking.md)(DSCP/PHB、AF の廃棄優先度)である。

## 導入 — そもそも何のための機構か

### 荷札はまだ何もしていない

前章までで、パケットには正しい荷札(DSCP)が貼られ、信頼境界で検認され、
層をまたいで写像されるようになった。しかし思い出してほしい —
**マーキングは転送の振る舞いを 1 bit も変えない**。荷札は「どう扱ってほしいか」
の宣言にすぎず、それを読んで実際に扱いを変える機構がなければ、
EF も CS6 も飾りである。

その機構が働く場所は決まっている。**輻輳点の出力ポート**である。
[01章](01_qos_basics.md)で見たとおり、輻輳とは出力ポートへの到着レートが
送出レートを超える瞬間であり、超過は遅延と損失に両替される。
QoS にできるのはこの両替の**配分を変えること**だけだった。
そして配分の変更は、具体的には2つの操作に分解される。

### 2つの独立な問い — 順番と廃棄

出力ポートで機器が下す判断は、突き詰めれば次の2問である。

```text
        到着したパケット
             │
             ▼
   ┌──────────────────────┐
   │ 問い1(廃棄の問題)   │  このパケットをキューに積むか、捨てるか?
   │                      │  — 損失を誰に配るか
   └──────────┬───────────┘
              ▼
      ┌─────┐ ┌─────┐ ┌─────┐
      │キューA│ │キューB│ │キューC│   クラス別のキュー
      └──┬──┘ └──┬──┘ └──┬──┘
         └───────┼───────┘
                 ▼
   ┌──────────────────────┐
   │ 問い2(順番の問題)   │  次の1個をどのキューから取るか?
   │                      │  — 遅延と帯域を誰に配るか
   └──────────┬───────────┘
              ▼
          ワイヤへ直列化
```

- **問い2 = スケジューリング**は、遅延と帯域の配分を決める。
  音声を先に送れば音声の遅延が縮み、バルクの帯域を絞ればその分が他に回る。
- **問い1 = 廃棄**は、損失の配分を決める。誰のパケットを捨てるかであり、
  さらに後で見るように「**いつ**捨てるか」(満杯になってからか、その前か)が
  想像以上に重要になる。

この2つは独立な設計軸である。何も設定しない機器 — 単一 FIFO +
テールドロップ — は、問い2に「到着順」、問い1に「満杯なら後着を捨てる」と
答えているだけであり、どちらの答えも痛みの違いを無視する。
本章の前半はスケジューリング(問い2)、後半は廃棄(問い1)を解剖する。

なお、キューの実体について1つだけ現実を述べておく。ハードウェアの
出力ポートが持つキューの数は有限で、**8 本**というのが伝統的な定番である
(PCP が 3 ビット = 8 値であることと符合する)。12 クラスの精緻な設計を
しても、最後は物理キューの数に畳み込まれる — [02章](02_classification_marking.md)で
「実務は 4〜6 クラスに集約する」と述べた背景にはこの事情もある。

## 理論

### スケジューリング(1)— 厳密優先(PQ)とその毒

最も単純で、最も強力なスケジューラが**優先キューイング
(PQ: Priority Queuing、厳密優先)**である。規則は1行で書ける —
**上位キューが空でない限り、下位キューには一切手を付けない**。

音声にとってこれは理想の席である。EF パケットは、先に到着していた
バルク転送が何百パケット並んでいようと、それらを追い越して
次の送出機会をもらえる。待たされるのは最悪でも
「いま直列化中の1パケットが出きるまで」だけであり
(送出中のパケットは中断できない — 非プリエンプティブ)、
その上限は 1500 バイト @ 10Mbps でも 1.2ms である。
[01章の 48ms の算数](01_qos_basics.md)は、PQ によって
**48ms → 最大 1.2ms** に縮む。遅延の上限が縮むということは、
揺らぎ(ジッタ)の上限も同時に縮むということである。

しかし PQ には構造的な毒がある。**飢餓(starvation)**である。
上位キューにトラフィックが絶えない限り、下位キューは永遠に順番が
回ってこない。誰かが(悪意または誤設定で)EF を大量に流し込めば、
他の全クラス — CS6 の KEEPALIVE も含めて — が窒息する。
優遇のつもりで作った特等席が、網全体を人質に取る単一障害点になる。

だから規律はこうなる。**厳密優先の席は原則1クラス(音声)だけに絞り、
必ず流入制限(ポリシング)とセットで使う**。これは実装の知恵である
と同時に仕様の要求でもある — [02章](02_classification_marking.md)で見たとおり、
**RFC 3246** は EF PHB にポリシングを課すことを前提にしている。
特等席には定員がある。定員を執行する機構(トークンバケット)は
[04章](04_shaping_policing.md)の主題である。

### スケジューリング(2)— 公平に混ぜる: ラウンドロビンから WFQ へ

PQ が「絶対の序列」なら、残りのクラスに必要なのは「重み付きの分配」である。
バルクを飢えさせずに、対話型データにはより多くの帯域を —
つまり比率の保証である。その素朴な出発点が**ラウンドロビン(RR)**、
各キューから1個ずつ順繰りに取る方式である。

RR の欠陥はすぐ分かる。**パケット単位の公平はバイト単位では不公平**である。
キュー A が 1500 バイト級、キュー B が 64 バイト級のパケットで
埋まっていたら、「1個ずつ交互」は帯域にして約 23 : 1 の配分になる。
重み付き版(WRR)も同じ病気を引き継ぐ。公平に配りたいのは
パケット数ではなく**ビット**なのである。

理論の世界はここに美しい理想解を持っている。
**GPS(Generalized Processor Sharing)** — 全キューを無限に細かく
(ビットよりさらに細かく、流体のように)同時に重み比で送る、
という思考実験上のスケジューラである。GPS は完全な重み比例配分と
遅延の解析的な保証を与えるが、現実のリンクはパケットを
1個ずつしか送れないから、そのままでは実装できない。

**WFQ(Weighted Fair Queuing)**は、GPS をパケット単位で近似する
アルゴリズムである(1989 年の Demers らの論文と Parekh–Gallager の
解析に由来する。IETF の RFC ではなく学術由来の技術であり、
各ベンダーの「WFQ」はこの系譜の実装群を指す)。発想は、
「もし GPS で送っていたら、このパケットはいつ送り終わるか」
という**仮想終了時刻**を到着のたびに計算し、常に仮想終了時刻が
最小のパケットを次に送る、というものである。バイト単位の公平と
良い遅延特性が得られるが、代償として全キューの中から最小値を
選び続ける計算(ソート、O(log n))がパケットごとに要る。
高速リンクではこの計算コストが痛い。

### スケジューリング(3)— DRR: 引き算だけで公平を作る

実装の世界の答えが **DRR(Deficit Round Robin)**である
(1995 年の Shreedhar–Varghese の論文に由来。これも RFC ではないが、
後述の fq_codel = RFC 8290 がフロー間スケジューラとして採用しており、
IETF 文書の中に生きている)。

DRR は WRR の骨格(順繰り)を保ったまま、**バイトの帳簿**を導入する。

- 各キューに **quantum**(1巡あたりの持ち点。バイト単位)と
  **deficit counter**(繰り越し残高)を持たせる。
- 自分の番が来たら、deficit に quantum を加算し、
  **先頭パケットのバイト数が deficit 以下である限り**送り続ける。
  送るたびに deficit からそのバイト数を引く。
- 先頭パケットが残高を超えたら、そこで打ち切って次のキューへ。
  残高は繰り越される(だから deficit = 「使い残し」)。

大きいパケットは1巡では送れないことがあるが、残高が貯まった巡で
必ず送れる。長期的には各キューの送出バイト数が quantum の比に収束する —
つまり**重みは quantum の比で表現される**。そして計算は加減算だけ、
quantum を最大パケット長(MTU)以上に取れば1巡あたりの処理は
O(1) である。ソートも仮想時刻も要らない。
「WFQ の公平性を、WRR のコストで」— これが DRR が実装の主流に
なった理由である。

### 実務の合成形 — 厳密優先1本+重み付き分配

実務のスケジューラ構成は、ここまでの部品の定番の合成になる。

```text
   EF(音声)       ──→ [厳密優先キュー]──┐   ← ポリシングで定員制(04章)
                                          │ 空でない限り最優先
   CS6(制御)      ──→ [キュー1 保証小]──┤
   AF4x(会議)     ──→ [キュー2 重み大]──┼──[DRR/WFQ 系の重み付き分配]──→ ワイヤ
   AF1x(バルク)   ──→ [キュー3 重み小]──┤
   DF(既定)       ──→ [キュー4 重み中]──┘
```

- **音声だけが厳密優先**。定員はポリシングで執行する。
- **制御プレーン(CS6)は厳密優先にはせず**、小さな帯域保証を持つ
  専用キューに入れるのが定石である。制御トラフィックは極小
  ([01章](01_qos_basics.md)の表のとおり)なので小さな保証で足り、
  厳密優先の席を増やして飢餓のリスクを広げる必要がない。
- 残りは重み付きで分配する。重みは**輻輳時の最低保証**であり、
  上限ではない(次項)。

この「厳密優先1本 + クラス別の重み付き保証」という合成は、
Cisco の文書では **LLQ(Low Latency Queuing)**と呼ばれる
(CBWFQ = クラスベース WFQ に厳密優先キューを1本足したもの、
という同社の実装群の名前であり、仕様上の用語ではない —
慣用名の常として、由来を知っていれば混乱しない)。
呼び名はどうあれ、ほぼすべてのベンダー・OS が同型の合成を提供している。

### work-conserving — 重みは「輻輳時の最低保証」である

スケジューラの重要な性質を1つ言語化しておく。ここまでのスケジューラは
すべて **work-conserving**(仕事保存的)である — 送るべきパケットが
1個でもあれば、リンクを遊ばせない。キュー3の重みが 10% でも、
他のキューが空なら、キュー3は 100% を使ってよい。

つまり重みとは帯域の**上限ではなく、輻輳時の最低保証**である。
「バルクに 10% の重み」は「バルクは 10% しか使えない」ではなく
「全員が押し寄せた最悪の瞬間でも、バルクに 10% は残す」を意味する。
[01章](01_qos_basics.md)の「輻輳していなければ QoS は何もしていない」は、
スケジューラの言葉ではこう言い直せる — 空いているとき、
work-conserving なスケジューラは FIFO と区別がつかない。

逆に、あえてリンクを遊ばせてでも送出レートに上限を課す
non-work-conserving な機構がシェーピングであり、
これは[04章](04_shaping_policing.md)で扱う。

### 廃棄(1)— テールドロップの3つの病理

ここから問い1(廃棄)に移る。既定の廃棄戦略 — キューが満杯なら
後着を捨てる**テールドロップ(tail drop)** — には、
「痛みを区別しない」以上に深い病理があることが 1990 年代に
明らかになった。IETF はそれを **RFC 2309**(1998)で整理し、
現在は後継の **RFC 7567**(BCP 197、2015)が
「すべてのルータはテールドロップよりましな廃棄をせよ」と勧告している。
病理は3つある。

**① 満杯キューの恒常化(full queues)** — テールドロップが廃棄を
始めるのは、定義上、**キューが満杯になってから**である。
ところが TCP は損失があるまで送信レートを上げ続けるから、
輻輳したリンクのキューは満杯またはその近傍で運転され続ける。
満杯のキューとは、[01章の遅延の分解](01_qos_basics.md)で見たとおり
**最大のキューイング遅延が全パケットに恒常的に課されている**状態である。
バッファは本来バーストを吸収するための遊びなのに、
常時満杯では吸収の余地もない — 遅延だけ払って弾力は失う。

**② ロックアウト(lock-out)** — 満杯のキューに新しいフローが
参入しようとしても、席はすでに埋まっており、後着はすべて捨てられる。
キューを埋めているのは大きなバーストを送り込む少数の攻撃的なフローで
あることが多く、テールドロップは**先に埋めた者を守り、
新参と行儀のよい者を罰する**配分になる。

**③ グローバル同期(global synchronization)** — 満杯の瞬間に
到着した**全フロー**のパケットが同時に捨てられるため、そのキューを
通る多数の TCP が一斉に損失を検知し、一斉にレートを半減させ、
一斉に回復して、また一斉に満杯にする。全フローの鋸歯が同期して
振動する結果、リンクは「全員が減速した谷」で遊び、
**満杯で捨てているのに平均利用率は上がらない**という
最悪の組み合わせになる。

### 廃棄(2)— AQM: 満杯になる前に、ランダムに捨てる

3つの病理はどれも「満杯になってから・機械的に」捨てることに由来する。
なら裏返せばよい — **満杯になる前に・確率的に**捨てる。
これが **AQM(Active Queue Management)**の発想であり、
その古典が **RED(Random Early Detection)**である(1993 年の
Floyd–Jacobson の論文に由来。RED 自体は RFC ではなく、
RFC 2309/7567 が AQM の代表例として参照する)。

RED の動作は1本の折れ線で描ける。

```text
 廃棄確率
   │                            ┌────── 1.0(強制廃棄 = テールドロップ)
   │                            │
   │                       ×   
   │                   ×       
   │               ×          ← min_th〜max_th の間は
   │           ×                 平均キュー長に比例した確率で捨てる
   │       ×                     (上限 max_p)
   │   ×
 0 ┼──────┬───────────┬─────────→ 平均キュー長
   0    min_th      max_th
```

- キュー長は瞬間値ではなく**指数移動平均(EWMA)**で見る。
  ミリ秒のバーストには寛容に(吸収はバッファの本業)、
  **持続的な**キューの成長にだけ反応するためである。
- 平均が min_th 以下なら何もしない。min_th〜max_th の間は、
  平均キュー長に比例した確率でパケットを捨てる。max_th を超えたら
  全部捨てる(テールドロップに退化)。

早期のランダム廃棄は、3つの病理を同時に治す。損失が確率的に
**別々のフローの別々の時刻**に落ちるから同期が壊れる。キューが
満杯になる前に TCP へ減速の合図が届くから、平均キュー長は
min_th 付近の浅いところに保たれ、遅延は下がりバーストの吸収余地は
戻る。多く送る者ほど高い確率で当たる(パケット数に比例して
くじを引く)から、配分もおおむね公平になる。

ここで見方を変えると、AQM の本質が見える。廃棄はもはや
「あふれた結果」ではなく、**送信者への輻輳信号を意図的に生成する
行為**である。[01章](01_qos_basics.md)で「網は捨てることで送信者と
対話している」と述べた — AQM はその対話を、破綻(満杯)の前に
礼儀正しく始める仕組みである。そして「信号を送るのが目的なら、
捨てる必要すらないのではないか?」という当然の問いが、
ECN(廃棄の代わりにヘッダの2ビットに印を付ける —
[02章](02_classification_marking.md)の DS フィールドの残り2ビット)につながる。
これは[04章](04_shaping_policing.md)で扱う。

RED の実務上の泣きどころは**チューニング**である。min_th/max_th/max_p の
適値はリンク速度・RTT・フロー数に依存し、外すと「効かない」か
「捨てすぎる」に振れる。RFC 7567 自身が「自動チューニングされる AQM を
使え」と述べており、これが後述の CoDel の登場動機になる。

### 廃棄(3)— WRED: 廃棄優先度の伏線回収

**WRED(Weighted RED)**は、RED の折れ線を**クラス(マーク)ごとに
複数本持つ**拡張である。ここで [02章の AF の行列](02_classification_marking.md)が
伏線を回収する — AF の廃棄優先度(AFx1/x2/x3)は、
**WRED のプロファイルを選ぶための値**だったのである。

```text
 廃棄確率          AFx3            AFx2           AFx1
   │             (超過分)                      (契約内)
   │               │               │               │
   │           ┌───┘           ┌───┘               │
   │       ×               ×                   ×
   │   ×               ×                   ×
 0 ┼───┬───────────┬───────────────┬─────────────→ 平均キュー長
     min_th(x3)   min_th(x2)      min_th(x1)
```

同じキュー(同じ AF クラス)の中で、AFx3 は浅いキュー長から
捨てられ始め、AFx1 は最後まで守られる。輻輳が深まるにつれて
**超過分(降格マーキングされたパケット)から順に死んでいく** —
[02章](02_classification_marking.md)で予告した「契約内は AFx1、超過分は
AFx2/x3 に降格して通し、輻輳時だけ超過分から捨てる」という設計は、
入口の降格マーキング(三色マーカー。[04章](04_shaping_policing.md))と
輻輳点の WRED が対になって初めて完成する。

注意すべきは、**WRED は TCP(輻輳制御を持つトラフィック)のための
道具**だということである。早期廃棄が意味を持つのは、捨てられた側が
それを合図に減速するからである。減速しない UDP の実時間系(音声)に
early drop を適用しても、得られるのは純粋な品質劣化だけである。
音声(EF)は PQ + ポリシングで守り、WRED は AF 系(TCP)のキューにだけ
適用する — 廃棄戦略もクラスの性質に合わせて選ぶ。

### 廃棄(4)— バッファブロート: 深すぎるバッファという病

AQM が普及しないまま、もう1つの変化が静かに進行した。
メモリの低価格化により、機器 — 特に家庭用ルータやモデム、
ホスト NIC — のバッファが**桁違いに深く**なったのである。

深いバッファ + テールドロップ + 「損失があるまで送り続ける」TCP。
この組み合わせの帰結は病理①の極端形である。TCP はその深いバッファを
**満たしきるまで**レートを上げ、以後キューは常時ほぼ満杯で運転される。
1000 パケットのバッファは、10Mbps の上り回線では
1.2ms × 1000 = **1.2 秒**の恒常遅延になる。帯域は出ている
(バッファは絶えず流れている)のに、RTT が数百ミリ秒〜秒単位に膨張し、
同じ経路を通る音声・対話型のすべてが巻き添えになる —
この病理が**バッファブロート(bufferbloat)**である。
[01章のトラブルシューティング](01_qos_basics.md)で「帯域は出るのに
操作がもたつく」として予告した症状の正体がこれである。

「バッファは大きいほど安全」という直感への反証は、実は IP の
歴史の初期からあった。**RFC 970**(1985)で J. Nagle は
「無限のバッファを持つパケットスイッチ」の思考実験により、
バッファを増やしても輻輳は解決せず、待ち時間が TTL を超えた
パケットが無駄に運ばれて**スループットがゼロに近づく**ことを示している。
バッファの適正サイズについては「帯域 × RTT(BDP)」という古典的な
目安(および多数フロー下ではその 1/√n で足りるという後年の解析)が
知られているが、適正値が状況依存であること自体が、
静的なサイズ調整では病理を根治できないことを示している。

### 廃棄(5)— CoDel と fq_codel: 長さではなく時間で測る

バッファブロート問題への現代の回答が **CoDel(Controlled Delay、
RFC 8289)**である。CoDel の洞察は測定量の転換にある —
**キューの長さではなく、パケットがキューに滞在した時間
(sojourn time)を測る**。

- 長さ(バイト/パケット数)の適正値はリンク速度に依存するが、
  滞在時間はどの速度のリンクでも同じ物差しで比べられる。
  遅延こそがユーザが被る量そのものでもある。
- **良いキューと悪いキュー**の区別: バーストを受け止めて
  速やかに掃ける(滞在時間がすぐ戻る)のはバッファの本業 = 良いキュー。
  滞在時間が下がらないまま持続するのが定在キュー = 悪いキュー。
  CoDel は後者だけを攻撃する。

動作は2つの定数で駆動される(既定値: **target 5ms、interval 100ms**)。
enqueue 時に時刻を記録し、dequeue 時に滞在時間を得る。滞在時間が
target を下回れない状態が interval 続いたら「悪いキュー」と判定して
廃棄モードに入り、先頭から1個捨てる。以後、キューが治るまで
捨てる間隔を interval/√n(n = 廃棄数)で徐々に詰めていく —
穏やかに始めて、効かなければ圧を上げる制御則である。
滞在時間が target を切れば廃棄モードを抜ける。
リンク速度も RTT もフロー数も入力に要らない。
**RED が果たせなかった「チューニング不要の AQM」**(RFC 7567 の勧告)を、
測る量の転換によって実現したのが CoDel である。
なお同時期の別解として、滞在時間を推定して確率制御する
**PIE(RFC 8033)**もある(ケーブルモデム規格 DOCSIS が採用)。

そして Linux の既定 qdisc として [01章](01_qos_basics.md)で登場済みの
**fq_codel(RFC 8290)**は、本章の2つの軸の合成そのものである —

- **FQ(スケジューリング)**: 5 タプルのハッシュでパケットを
  多数(既定 1024)のフローキューへ振り分け、フロー間を
  **DRR**(quantum 既定 1514 バイト)で公平に回す。さらに
  新規フロー(キューが空だったフロー)を1巡だけ優先する仕掛けがあり、
  DNS クエリや TCP ハンドシェイクのような散発小フローは
  バルクの列に並ばずに済む。
- **CoDel(廃棄)**: フローキューの**それぞれ**に独立の CoDel を適用する。
  バルクフローの定在キューはそのフローの中だけで削られ、
  他のフローは巻き添えにならない。

DSCP を一切見ていないことに注意してほしい。fq_codel が作る公平は
クラス設計に基づく意図の配分ではなく「フロー間の機械的な公平 +
遅延の自動抑制」である。それだけでバッファブロートと
「1本の象(大転送)が全員を潰す」問題は消えるから、
**明示的なクラス設計に値する輻輳点(WAN 境界など)には
クラス別キュー + スケジューラを、それ以外には fq_codel を**、
という使い分けが現代の実務の相場になっている。

## プロトコル動作の詳細

キューイングとスケジューリングはワイヤ上にメッセージを流さない
(この点は [01章](01_qos_basics.md)で述べた QoS 全般の性質のとおりである)。
「動作の詳細」は、機器内部のアルゴリズムの精密な追跡になる。

### DRR のウォークスルー — 帳簿の動き

quantum をどちらも 1500 バイトとし、2本のキューを DRR で回す。
キュー A には 1500B のパケットが連続、キュー B には 600B が連続して
積まれているとする(重み 1 : 1 なので、送出バイト数が 1 : 1 に
なるのが正解である)。

| 巡 | キュー | deficit(+quantum 後) | 送出 | deficit(残) |
|---|---|---|---|---|
| 1 | A | 0 + 1500 = 1500 | 1500B × 1 | 0 |
| 1 | B | 0 + 1500 = 1500 | 600B × 2 = 1200B | 300 |
| 2 | A | 0 + 1500 = 1500 | 1500B × 1 | 0 |
| 2 | B | 300 + 1500 = 1800 | 600B × 3 = 1800B | 0 |
| 3 | A | 1500 | 1500B × 1 | 0 |
| 3 | B | 1500 | 600B × 2 | 300 |

2巡ごとに見ると、A は 3000B、B も 3000B(1200 + 1800)—
パケット数は 2 : 5 と大きく違うのに、**バイトでは正確に 1 : 1** に
収束している。使い残し(deficit 300)が次の巡に繰り越されて
帳尻を合わせる様子が読み取れる。重みを 2 : 1 にしたければ
quantum を 3000 : 1500 にすればよい。
なお quantum を MTU 未満(例: 600)にすると、キュー A は
1500B のパケットを送るのに3巡かけて残高を貯めることになり、
「1巡あたり少なくとも1個は送れる」という O(1) の前提が崩れる —
**quantum ≥ MTU** が実装の約束事である。

### 厳密優先の遅延上限 — 01章の算数、ふたたび

PQ の下での音声パケットの待ち時間を正確に書くと:

```text
  音声の最大待ち = いま送出中のパケットの残り直列化時間
                 + 先に並んでいる音声パケットの直列化時間の合計
```

第2項はポリシング(定員)で小さく保たれている前提なので、
支配項は第1項 — **最悪で最大パケット1個分**である。
10Mbps なら 1500B = 1.2ms、100Mbps なら 0.12ms。
[01章](01_qos_basics.md)の「バルク 40 個の後ろで 48ms」は、
PQ の導入により**上限 1.2ms(+自クラスの分)**に変わる。
ジッタも同じ上限に縛られる — 音声品質にとって、これが
スケジューリングの効果のほぼすべてである。
(数 Mbps を大きく下回る低速リンクでは 1 パケットの直列化自体が
予算を食うため、大きなフレームを割って音声を挟み込む
LFI: Link Fragmentation and Interleaving という古典的手法が
存在したが、現代の回線速度では出番はまれである。)

### WRED プロファイルの数値例

AF1x クラスのキュー(物理長 300 パケットとする)に、
3本のプロファイルを張った例:

| プロファイル | min_th | max_th | max_p | 意味 |
|---|---|---|---|---|
| AF13(超過・高廃棄) | 60 | 150 | 20% | 平均 60 で捨て始め、150 で全滅 |
| AF12(超過・中廃棄) | 100 | 200 | 10% | |
| AF11(契約内) | 150 | 280 | 5% | 最後まで守られる |

平均キュー長が 120 のとき、AF13 は確率約 13% で捨てられ、
AF12 は約 2%、AF11 は無傷である。輻輳が深まるほど格差が開き、
しかし**順番は常に「超過分から」** — 入口の三色マーカー
([04章](04_shaping_policing.md))が付けた階級が、
輻輳点でそのまま死ぬ順番になる。
ECN 対応のフロー([02章](02_classification_marking.md)の
下位2ビット)に対しては、この「捨てる」を「印を付ける」に
置き換えられる — それも 04 章で扱う。

### CoDel の状態遷移

```text
                滞在時間 < target が一度でも観測される
      ┌──────────────────────────────────────────────┐
      ▼                                              │
 ┌─────────┐  滞在時間 ≥ target が interval 継続 ┌──┴───────┐
 │ 通常状態 │ ────────────────────────────────→ │ 廃棄状態  │
 │(何もしない)│                                   │ 先頭から1個廃棄│
 └─────────┘                                    │ 次回は interval/√n 後│
                                                 └──────────┘
```

- 測るのは dequeue 時の滞在時間(enqueue 時刻との差)。
- 廃棄は**先頭から**行う(head drop)。テールドロップと逆である。
  先頭 = 最も長く待った = 最も情報が古いパケットを捨てるほうが
  損失信号が最速で送信者に届き、かつ新しいデータが生き残る。
- 廃棄間隔の √n 縮小は、TCP の応答(1 RTT かけて減速)を待ちながら
  圧を単調に強める制御である。キューが治れば即座に通常状態へ戻る。

### 全体の合成 — 1つの輻輳点の完全な絵

本章の部品を [01章のパイプライン](01_qos_basics.md)の④⑤に
はめ込むと、WAN 境界ルータの出力ポートはこうなる:

```text
                     ④ キュー投入(廃棄判定)          ⑤ スケジューラ
                    ┌─────────────────────────┐
  EF ──────────────→│ 優先キュー(浅い・定員済)  │──→ 厳密優先 ─┐
  CS6 ─────────────→│ 制御キュー(テールドロップで可)│─┐          │
  AF4x ────────────→│ 会議キュー + WRED 3面     │─┼→ DRR ────┼→ 直列化
  AF1x ────────────→│ バルクキュー + WRED 3面   │─┤ (重み比)  │
  DF ──────────────→│ 既定キュー + fq_codel/RED │─┘          │
                    └─────────────────────────┘             │
                                                 上が空のときだけ下段
```

クラスごとに**廃棄戦略まで違う**ことに注目してほしい。
EF は浅いキュー(深くする理由がない — 深い優先キューは
ジッタの原資にしかならない)、制御は極小トラフィックゆえ
単純廃棄で足り、TCP 系の AF には WRED、雑多な既定クラスには
フロー公平の fq_codel。「どの列に積むか・どの列から送るか・
どう捨てるか」の3点セットがクラスの性質ごとに設計されて、
初めて荷札(02章)は意味を持つ。

## 設定例 — Linux で PQ と AQM を観察する

**fq_codel の統計を読む** — [01章](01_qos_basics.md)で見た既定 qdisc を、
今度は本章の語彙で精読する:

```bash
$ tc -s qdisc show dev eth0
qdisc fq_codel 0: root refcnt 2 limit 10240p flows 1024 quantum 1514
    target 5ms interval 100ms memory_limit 32Mb ecn drop_batch 64
 Sent 987654321 bytes 1234567 pkt (dropped 42, overlimits 0 requeues 3)
 backlog 0b 0p requeues 3
  maxpacket 1514 drop_overlimit 0 new_flow_count 8021 ecn_mark 12
  new_flows_len 0 old_flows_len 2
```

- `flows 1024 quantum 1514` — 1024 本のフローキューを quantum
  1514B の **DRR** で回している(quantum ≥ MTU の約束どおり)。
- `target 5ms interval 100ms` — **CoDel** の2定数。
- `new_flow_count` — 新規フロー優遇が発動した回数。
- `ecn_mark` — 捨てる代わりに ECN 印を付けた回数(04章の予告)。
- `drop_overlimit` — limit(10240 パケット)超過の強制廃棄。
  ここが増えるのは AQM が追いつかない異常事態である。

**厳密優先を最小構成で作る** — `prio` qdisc は設定した瞬間から
厳密優先で動く3バンドの PQ である。DSCP で振り分けるフィルタを
足し、バルクのバンドには fq_codel を張る:

```bash
# 3バンドの PQ(band 0 が最優先)。priomap の既定は ToS ベースなので明示フィルタで上書きする
tc qdisc add dev eth0 root handle 1: prio bands 3

# EF(ToS 0xb8)→ band 0、CS6(0xc0)→ band 1、それ以外 → band 2
tc filter add dev eth0 parent 1: protocol ip prio 1 u32 \
    match ip dsfield 0xb8 0xfc flowid 1:1
tc filter add dev eth0 parent 1: protocol ip prio 1 u32 \
    match ip dsfield 0xc0 0xfc flowid 1:2
tc filter add dev eth0 parent 1: protocol ip prio 2 matchall flowid 1:3

# 既定バンドにはフロー公平 + AQM
tc qdisc add dev eth0 parent 1:3 fq_codel
```

`match ip dsfield 0xb8 0xfc` のマスク `0xfc` は上位6ビット(DSCP)
だけを比較し、ECN の2ビットを無視する — [02章の症状1](02_classification_marking.md)で
見た「ECN ビットが立っていると一致しない」事故を避ける実務の作法である。

観察には、band 2 を飽和させながら(iperf3 等)band 0 宛ての
遅延を測ればよい:

```bash
$ ping -Q 0xb8 -c 100 -i 0.2 192.0.2.1     # EF: 飽和中も RTT がほぼ不変
$ ping -Q 0x00 -c 100 -i 0.2 192.0.2.1     # DF: RTT がキュー分だけ膨らむ
```

2つの `mdev` の差が、スケジューリングが音声に与えるものの実測になる。
なお `prio` には定員の機構がない — band 0 に流し込んだ分だけ
下位バンドは無条件に飢える。ポリシングを重ねる方法は
[04章](04_shaping_policing.md)で扱う。

## トラブルシューティング

### 症状1: 音声を最優先にしたら、輻輳のたびに他の全通信が止まる

厳密優先の飢餓である。優先キューに入るトラフィックが想定
(音声数本 = 数百 kbps)を超えている — 典型は分類の緩みで、
「UDP は全部音声扱い」のような雑なフィルタや、信頼境界の漏れで
EF を自己申告した端末のバルク転送が特等席に流れ込んでいる。
確認は優先キューの送出レート(`tc -s class` や機器の
per-queue カウンタ)で、音声の理論値(同時通話数 × コーデック帯域)と
比べれば一目で分かる。対策は2つで1組 — **分類を締める**ことと、
**優先キューにポリサ(定員)を必ず併設する**こと。
定員があれば、事故が起きても被害は「音声クラス内の劣化」に
封じ込められ、網全体は人質に取られない。

### 症状2: 満杯で捨てているのに、リンクの平均利用率が 80% 止まり

グローバル同期の典型的な足跡である。輻輳点のキューはテールドロップで
廃棄を刻んでいるのに、利用率グラフは鋸歯状に振動して天井に届かない —
多数の TCP フローが同時に損失を食らい、一斉に減速・一斉に回復を
繰り返している。フロー数が多いのに全体のスループットが
「谷」のたびに落ち込むのが特徴で、単一フローの限界(ウィンドウ不足)とは
区別できる。処方箋は AQM の導入(fq_codel、または AF クラスなら WRED)。
ランダムな早期廃棄が損失の時刻をフローごとに散らし、
鋸歯の位相をずらすことで、合計のスループットが平らに近づく。

### 症状3: 帯域は出ているのに、アップロード中だけ全操作が重くなる

バッファブロートである。スピードテストは回線速度どおり、
しかし大きな送信(バックアップ、クラウド同期)の最中だけ RTT が
数百 ms〜秒に膨らみ、Web も SSH も音声も一緒に沈む —
[01章の症状4](01_qos_basics.md)の後者の正体である。
犯人は輻輳点(たいてい WAN 境界の上り)の深い FIFO で、
TCP がそれを満たしきって恒常遅延に変えている。
切り分けは簡単で、**負荷をかけながら ping** し、無負荷時との
RTT 差を見る(この差がそのままキュー滞在時間である)。
処方箋は輻輳点の qdisc を fq_codel に替えること —
定在キューだけが削られ、負荷中の RTT 増分が target(5ms)級まで
落ちる。輻輳点が自分の機器でない(ISP のモデム内など)場合は、
その手前で自分がボトルネックを作る — 回線速度の少し下に
シェーピングして輻輳点を自機に引き取る — 必要があり、
これは[04章](04_shaping_policing.md)で扱う。

### 症状4: WRED を入れたら、音声の品質がかえって悪化した

廃棄戦略とトラフィックの性質の不整合である。WRED の早期廃棄は
「捨てられたら減速する」TCP への信号として設計されている。
輻輳制御を持たない UDP の実時間系(音声・映像)に適用すると、
まだ席があるのに音声パケットを確率的に捨てるだけ —
信号としては誰にも届かず、純粋な劣化にしかならない。
音声は WRED の対象から外し、PQ + ポリシングで守るのが正しい。
同系の事故として、AF クラスの WRED 閾値の設定ミス
(AFx1 と AFx3 のプロファイルを逆に張る、min_th を浅くしすぎて
バースト吸収の余地まで削る)もある — プロファイルの意図
「輻輳時に**超過分から**捨てる。契約内は最後まで守る」に照らして
3面の閾値の順序を検算すること。

## 演習・確認問題

**問1.** 輻輳点の出力ポートで機器が下す判断を「2つの独立な問い」に
分解し、それぞれがどの品質量(遅延・帯域・損失)の配分を
決めるかを述べよ。

**問2.** 厳密優先(PQ)の下で、音声パケットが被る待ち時間の上限は
何によって決まるか。10Mbps のリンク(MTU 1500)での上限値を計算し、
01章の「バルク 40 個の後ろで 48ms」と比較せよ。また、
PQ を安全に使うために必ず併設すべき機構とその理由を述べよ。

**問3.** WRR(重み付きラウンドロビン)がバイト単位で不公平になる
理由を述べ、DRR がそれを quantum と deficit counter でどう解決するかを
説明せよ。quantum を MTU 以上に取るべき理由にも触れること。

**問4.** テールドロップの3つの病理(満杯キューの恒常化・ロックアウト・
グローバル同期)を説明し、RED の「満杯になる前に・確率的に捨てる」が
それぞれをどう治すかを対応付けよ。

**問5.** AF31 と AF33 のパケットが同じキューに積まれているとき、
輻輳の進行とともに何が起きるか、WRED のプロファイルの言葉で説明せよ。
また、この機構が入口側のどの機構(04章の主題)と対になって
初めて意味を持つか。

**問6.** CoDel が「キュー長」ではなく「滞在時間」を測ることにした
理由を2つ挙げよ。また fq_codel が「スケジューリング」と「廃棄」の
両方の問いにどう答えているか、本章の部品の名前(DRR、CoDel)を
使って説明せよ。

---

**解答**

**問1.** ①廃棄の問い — 到着したパケットをキューに積むか捨てるか
(捨てるならいつ・誰を)。これは**損失**の配分を決める。
②順番の問い(スケジューリング)— 複数のキューから次の1個を
どこから取るか。これは**遅延**(先に送られるほど待ちが短い)と
**帯域**(取られる頻度・量の比)の配分を決める。両者は独立な
設計軸であり、既定の FIFO + テールドロップは両方に
「痛みを区別しない」答えを与えているにすぎない。

**問2.** 送出中のパケットは中断できない(非プリエンプティブ)ため、
上限は「いま直列化中の最大長パケット1個の残り時間 + 自クラス
(音声)の先行パケット分」で決まる。後者はポリシングで小さく
保たれる前提なので、支配項は最大パケット1個分 = 1500B × 8 ÷ 10Mbps
= **1.2ms**。FIFO での 48ms(40 個分)が、PQ では 40 分の 1 の
上限に縮み、ジッタも同じ上限に縛られる。併設すべきは
**ポリシング(流入の定員)**である。PQ は上位キューが空になるまで
下位に手を付けないため、定員がなければ誤分類・悪意の流入が
他の全クラス(制御プレーン含む)を飢餓させる。RFC 3246 も
EF PHB にポリシングを前提として課している。

**問3.** WRR は「パケット数」を数えて公平にするため、平均パケット長が
違うキュー間では送出バイト数が長さの比だけ歪む(1500B のキューは
64B のキューの約 23 倍の帯域を取る)。DRR は各キューにバイト建ての
持ち点(quantum)を1巡ごとに与え、送ったバイト数を残高
(deficit counter)から引き、先頭パケットが残高を超えたら打ち切って
残高を繰り越す。長期的には送出バイト数が quantum の比に収束し、
バイト単位の公平が加減算だけ(O(1))で得られる。quantum が
MTU 未満だと、最大長パケットを送るのに複数巡の貯金が必要になり
「1巡で最低1個送れる」前提が崩れて計算量とジッタが悪化するため、
quantum ≥ MTU とする。

**問4.** ①満杯キューの恒常化: TCP は損失まで加速するので、
テールドロップ(満杯で初めて捨てる)の下ではキューが満杯近傍で
運転され、全パケットが最大キューイング遅延を恒常的に被り、
バースト吸収の余地も消える。→ RED は min_th から捨て始めるので
平均キュー長が浅く保たれ、遅延低減と吸収余地が戻る。
②ロックアウト: 満杯のキューでは後着(新規・小フロー)が
一方的に捨てられ、キューを埋めた攻撃的フローが守られる。
→ RED のくじはキュー内に多くパケットを持つ(= 多く送る)者ほど
高確率で当たるため、配分がフローの送出量に比例して公平化する。
③グローバル同期: 満杯の瞬間に全フローが同時に損失を検知し、
一斉減速・一斉回復の振動で利用率が落ちる。→ ランダムな早期廃棄は
損失をフローごとに別の時刻へ散らし、鋸歯の位相をずらして
振動を解消する。

**問5.** 同じキュー = 同じスケジューリング(資源配分)を受けるが、
WRED は廃棄優先度ごとに別のプロファイル(min_th/max_th/max_p)を
持つ。平均キュー長が伸びるにつれ、まず AF33 のプロファイルが
発火して確率廃棄が始まり、さらに深まると AF31 も捨てられ始める —
輻輳の進行とともに「超過分から順に死ぬ」。この階級付け(降格
マーキング)を入口で行うのが**三色マーカー(ポリシング)**であり、
入口が付けた階級を輻輳点の WRED が執行する、という分業で
初めて「契約内は守り、超過は空きがあるときだけ通す」が完成する。

**問6.** 理由①: キュー長(バイト/個数)の適正値はリンク速度に
依存するため万能の閾値が存在しない(RED のチューニング問題)が、
滞在時間はどの速度でも同じ物差しで比較でき、既定値(target 5ms /
interval 100ms)のままで自動チューニングが成立する。理由②:
ユーザが被る量は遅延そのものであり、制御したい量を直接測るのが
正確である(バーストによる一時的な長いキュー = 良いキューと、
掃けない定在キュー = 悪いキューを、滞在時間の持続で区別できる)。
fq_codel は、スケジューリングの問いに「5タプルハッシュで分けた
フローキュー群を DRR で公平に回す(+新規フローを1巡優遇)」と答え、
廃棄の問いに「各フローキューに独立の CoDel を適用し、定在キューを
作ったフローの中だけで先頭から捨てる」と答える合成である。

## まとめ

- 輻輳点の判断は2軸に分解できる。スケジューリング(どの列から送るか =
  遅延と帯域の配分)と廃棄(誰をいつ捨てるか = 損失の配分)。
  荷札(02章)はこの2軸の設計があって初めて意味を持つ。
- スケジューラの定石は「**厳密優先1本(音声、ポリシングで定員制)+
  残りは DRR/WFQ 系の重み付き分配**」。厳密優先は遅延・ジッタの上限を
  「最大パケット1個分」まで縮めるが、定員なしでは飢餓の単一障害点になる。
  重みは輻輳時の最低保証であり上限ではない(work-conserving)。
- テールドロップには満杯キューの恒常化・ロックアウト・グローバル同期の
  3病理がある(RFC 2309 → RFC 7567)。AQM は「満杯になる前に・確率的に」
  捨てることで3つを同時に治す — 廃棄を、あふれの結果から
  送信者への意図的な輻輳信号へ変える発想である。
- WRED は AF の廃棄優先度(02章)の執行機構であり、輻輳の進行とともに
  超過分(AFx3)から捨てる。ただし早期廃棄は TCP のための信号であり、
  減速しない実時間系には適用しない。
- 深すぎるバッファはテールドロップと TCP の性質により恒常遅延に化ける
  (バッファブロート)。CoDel(RFC 8289)は長さでなく**滞在時間**を測る
  ことでチューニング不要の AQM を実現し、fq_codel(RFC 8290)は
  フロー公平(DRR)と各フロー独立の CoDel を合成した現代の既定解である。
