# 静的ルーティング vs 動的ルーティング

## 概要

この章では、ルーティングテーブルに経路を載せる2つの方法 —
管理者が手で書く**静的ルーティング**と、ルータどうしが情報交換して自動で学ぶ
**動的ルーティング** — を比較し、それぞれが解いている問題と限界を整理する。
前提知識は [前章](./02_routing_table_basics.md) のルーティングテーブルの構造、
管理距離、再帰的ルックアップである。

## 導入 — テーブルの行は誰が書くのか

前章で、ルーティングテーブルの経路には3つの情報源(直結・静的・動的)が
あることを見た。直結経路はインタフェースの設定から自動的に生まれるので、
管理者に選択の余地はない。問題は**直結していない宛先への経路**である。
これをテーブルに載せる方法は、原理的に2つしかない:

1. **静的ルーティング**: 管理者が「この宛先はこの隣人へ」と1行ずつ手で書く
2. **動的ルーティング**: ルータどうしがプロトコルで経路情報を交換し合い、
   各ルータが自動で経路を計算してテーブルに載せる

一見すると「自動のほうが便利に決まっている」と思えるが、話はそう単純ではない。
動的ルーティングはプロトコルの理解・設計・運用というコストを要求し、
「他のルータからの伝聞」を信じるというリスクも持ち込む。
一方の静的ルーティングは、単純で確実だが**ネットワークの変化に追従できない**。

実際のネットワークはほぼ例外なく両者を併用して作られる。
「どちらが優れているか」ではなく、**それぞれがどんな問題を解き、
どんな限界を持つか**を理解することが本章の目的である。
そしてこの比較は、なぜ OSPF や BGP のような複雑なプロトコルが
存在しなければならないのか、という以後の全章の動機付けでもある。

## 理論

### 静的ルーティングの本質 — 「意思」を書く

静的経路は、管理者の意思をそのままテーブルに写したものである:

```text
198.51.100.0/24 → ネクストホップ 192.0.2.1
```

この1行の意味を、前章までの道具立てで正確に述べると
「宛先が 198.51.100.0/24 にロンゲストマッチしたパケットは、
192.0.2.1 へ(再帰解決の上で)引き渡せ。この情報源の管理距離は 1
(Cisco 系慣例)であり、直結経路に次いで信頼せよ」となる。

静的ルーティングの性質は、すべてこの「人が書いた静止した1行」であることから
導かれる:

- **決定的である**: 経路は書いたとおりにしか変わらない。トラフィックが
  どこを通るかを完全に予測・統制できる
- **プロトコルのオーバーヘッドがない**: 経路情報のやり取りが発生しないため、
  帯域・CPU・メモリを消費せず、攻撃面(不正な経路情報の注入)も存在しない
- **変化に追従しない**: 書いた本人が消すか書き換えるまで、ネットワークが
  どう変わろうと経路は残り続ける。これが最大の弱点である

3つ目の弱点を正確に理解することが重要である。静的経路も
**まったく無反応というわけではない**: 前章で見たとおり、出力インタフェースが
down すれば直結経路の消滅に連鎖して静的経路も無効化されるし、
ネクストホップが再帰解決できなくなれば FIB から外れる。
問題は、静的経路が検知できるのが**その程度まで**だということである。

```text
[R1] ───────── [R2] ──✕── [R3] ── 198.51.100.0/24
      リンクA         リンクB(障害)

R1 の静的経路: 198.51.100.0/24 → R2
```

R2 の先のリンク B が切れても、R1 から見えるリンク A は健在であり、
ネクストホップ R2 への再帰解決も成功し続ける。R1 の静的経路は
テーブルに残り、パケットは R2 まで届いてから行き場を失う。
つまり静的ルーティングは**自分の1ホップ先までしか見えず、
その先の障害には原理的に気づけない**。届かない宛先への経路が
テーブルに残り続けてトラフィックを吸い込む状態を
**ブラックホール**と呼び、静的ルーティングの典型的な故障モードである。

もう1つの限界は**規模**である。ルータが N 台、プレフィックスが P 個ある
ネットワークで完全な到達性を静的経路だけで作ると、考慮すべき設定は
最大で N × P 行に達する。しかも障害迂回・機器追加・アドレス変更のたびに
**全ルータの整合性を人間が保証しなければならない**。数台のネットワークでは
現実的でも、数十台を超えると設定量よりも「変更のたびに全体の整合を
検証するコスト」が破綻点になる。

### 動的ルーティングが解く問題 — 到達性の自動維持

動的ルーティングの本質は、上の2つの限界をプロトコルで解決することにある:

1. **トポロジ変化への追従**: ルータどうしが経路情報を交換し続け、
   障害や構成変更を検知したら**経路を自動で再計算**する。
   さきほどの図でいえば、R2 がリンク B の断を検知して R1 に伝え、
   R1 が代替経路(あれば)へ切り替える — 人間の介入なしに
2. **規模への対応**: 各ルータは自分の直結情報だけを正しく申告すれば、
   ネットワーク全体の経路は交換と計算で自動的に組み上がる。
   設定量はネットワーク全体の経路数ではなく、そのルータ自身の
   接続数に比例する程度で済む

障害を検知してから、全ルータのテーブルが新しいトポロジを反映した
一貫した状態に落ち着くまでの過程を**収束(convergence)**と呼ぶ。
収束が完了するまでの間、ネットワークには古い情報と新しい情報が混在し、
パケットの損失やループが起こりうる。**収束の速さと正確さは、
ルーティングプロトコルの設計を貫く中心的な評価軸**であり、
[次章](./04_distance_vector_link_state.md) 以降で繰り返し登場する。

### 動的ルーティングのコスト — 「伝聞」を信じるということ

自動化はただでは手に入らない。動的ルーティングが持ち込むコストを
正しく認識しておく:

- **資源の消費**: 経路情報の交換に帯域を、経路計算に CPU を、
  経路データベースの保持にメモリを使う。特に経路数が大きい環境
  (インターネットのフルルートなど)ではメモリと計算量が機器選定を左右する
- **複雑さ**: プロトコルの動作を理解していないと、障害時に
  「なぜこの経路が選ばれているのか」を説明できない。
  自動化されたシステムの故障解析は、手作業のシステムより難しい
- **信頼の問題**: 動的経路は本質的に**他のルータからの伝聞**である。
  隣のルータが(バグ・設定ミス・悪意によって)誤った経路を広告すれば、
  それを信じたルータは誤った転送を行う。1台の設定ミスがネットワーク全体、
  ときにインターネット規模の障害へ波及した事例は数多い。
  経路情報のフィルタリングと認証が動的ルーティング運用の重要課題になるのは
  このためである(BGP のポリシー制御として第3部で本格的に扱う)
- **予測可能性の低下**: 経路はトポロジとプロトコルの計算結果で決まるため、
  「このトラフィックは必ずここを通る」と言い切るには
  プロトコルの選択規則まで含めた理解が必要になる

この対比を一言でまとめると:
**静的ルーティングは「人間の意思」を、動的ルーティングは「ネットワークの現状」を
テーブルに反映する仕組み**である。意思で固定すべき経路は静的に、
現状に追従すべき経路は動的に載せる — 使い分けの原則はこれに尽きる。

### 使い分けの定石

実務上の構成パターンは、この原則から素直に導かれる:

- **スタブネットワークの出口は静的(またはデフォルトルート)で足りる**:
  外部への出口が1つしかない拠点では、迂回先が存在しないので
  動的に経路を学ぶ意味がない。出口へのデフォルトルート1行が最適解である。
  出口が1つなら、そのリンクの断はすなわち全断であり、
  静的経路の「先の障害に気づけない」弱点も問題にならない
- **迂回路が存在するところは動的にする**: 経路が複数あるのに静的で固定すると、
  障害時の切り替えが人間の作業速度に律速される。
  自動迂回こそ動的ルーティングの存在意義である
- **中核は動的、末端は静的、という階層化**: 多数の拠点を持つネットワークでは、
  中核(コア)は IGP で自動化し、単一接続の末端拠点は静的+集約で
  中核に取り込む構成が定石である。末端の経路を動的プロトコルに
  参加させないことで、中核の経路データベースを小さく保てる

### フローティングスタティック — 管理距離を道具として使う

静的と動的の境界で、前章の**管理距離**が設計の道具として現れる例を1つ見ておく。

静的経路の管理距離は通常最強(Cisco 系で 1)だが、多くの実装では
**経路ごとに管理距離を任意の値へ変更できる**。これを使い、
動的プロトコルより**わざと弱い**静的経路を仕込んでおくのが
**フローティングスタティックルート(floating static route)**である:

```text
通常時:
  10.0.0.0/16  OSPF 経由(AD 110)     ← 勝者としてテーブルに載る
  10.0.0.0/16  静的 バックアップ回線へ(AD 250)  ← 負けて待機

OSPF 経路が消えた(主回線断):
  10.0.0.0/16  静的 バックアップ回線へ(AD 250)  ← 浮上してテーブルに載る
```

通常時は動的経路が勝ち、動的経路が撤回された瞬間に静的経路が「浮上」して
バックアップ経路として機能する。管理距離が「情報源間の優先順位を決める
内部的な物差し」であることを逆手に取った、古典的なバックアップ設計である。

ただしこの仕組みが働くのは**主経路がテーブルから消えたとき**だけである。
主経路が消えない障害(さきほどの「リンクは up だが先が死んでいる」型)では
切り替わらない。検知の仕組みは次節で扱う。

## プロトコル動作の詳細

この章では特定プロトコルには立ち入らず、**あらゆる動的ルーティング
プロトコルに共通する動作の骨格**を押さえる。OSPF も BGP も、
細部は違えどこの骨格の上に成り立っている。

### 動的ルーティングの共通ライフサイクル

どの動的ルーティングプロトコルも、おおむね次の5段階で動く:

```mermaid
graph LR
    A[① ネイバー発見] --> B[② 隣接関係の確立]
    B --> C[③ 経路情報の交換]
    C --> D[④ 経路計算・テーブル登録]
    D --> E[⑤ 維持と障害検知]
    E -->|変化を検知| C
```

1. **ネイバー発見**: 経路情報を交換する相手(ネイバー、隣接ルータ)を
   見つける。OSPF のように専用パケットで自動発見するものと、
   BGP のように管理者が相手を明示設定するものがある
2. **隣接関係の確立**: 発見した相手と、バージョン・各種パラメータの
   合意を取り、情報交換できる状態(隣接、アジャセンシー/ピア)を作る
3. **経路情報の交換**: 自分の知る経路情報を相手に広告し、相手からも受け取る。
   **何を**交換するか(距離か、トポロジ情報か)がプロトコル設計の
   最大の分岐点であり、次章の主題である
4. **経路計算・テーブル登録**: 受け取った情報から最良経路を計算し、
   プロトコル自身のデータベースを経て RIB に差し出す(前章で見た構造)
5. **維持と障害検知**: 定期的な生存確認(keepalive / hello)を交わし続け、
   相手の沈黙や明示的な撤回通知を検知したら③に戻って再交換・再計算する —
   これが収束の入口である

### 障害検知の速さが収束の速さを決める

⑤の障害検知には、速い順に3つの階層がある:

1. **物理層の検知(ミリ秒未満〜)**: リンク断を物理的に検知できれば最速である。
   ただし検知できるのは直結リンクだけで、間に L2 スイッチや光伝送装置が
   挟まると「自分側のポートは up のまま、対向だけ down」という
   検知不能パターンが生じる
2. **プロトコルのタイマー(数秒〜数十秒)**: keepalive / hello の応答が
   一定時間(hold time / dead interval)途絶えたら相手を死んだとみなす。
   確実だが遅い。タイマーを詰めれば速くなるが、CPU 負荷や
   誤検知(輻輳で keepalive が落ちただけ)とのトレードオフになる
3. **BFD(数十ミリ秒〜)**: 障害検知だけを専門に行う軽量プロトコル
   **BFD(Bidirectional Forwarding Detection)**が **RFC 5880** で
   定義されている。小さなパケットを高頻度(数十ミリ秒間隔)で交換し、
   途絶を検知したら登録元のプロトコル(OSPF、BGP、そして静的経路)へ
   即座に通知する。「検知」を各プロトコルのタイマーから分離して
   高速化・共通化する設計である

BFD の重要な含意は、**静的経路にも「先の障害への追従」を持ち込める**ことである。
静的経路のネクストホップを BFD で監視し、途絶したら経路を無効化する —
これにより、フローティングスタティックが「リンク up のまま先が死ぬ」型の
障害でも切り替わるようになる。静的/動的の境界は
「検知の仕組みを持つか否か」だと言い換えることもできる。

### 動的ルーティングの2つの分類軸

動的ルーティングプロトコルは伝統的に2つの軸で分類される。
本章では見取り図だけを示し、詳細は続く2つの章に委ねる:

- **情報の交換方式による分類**: 「宛先までの距離」を教え合う
  **ディスタンスベクタ型**と、「ネットワークの地図」を共有して各自で計算する
  **リンクステート型**。アルゴリズムの本質的な違いであり、
  [次章](./04_distance_vector_link_state.md) の主題である
- **適用範囲による分類**: 1つの管理主体の内部で使う **IGP**
  (Interior Gateway Protocol: OSPF など)と、管理主体どうしの間で使う
  **EGP**(Exterior Gateway Protocol: 事実上 BGP のみ)。
  この区別は「経路の良さ」より「誰を信頼しどんなポリシーで繋がるか」という
  問題意識の違いに根ざしており、[第1部最終章](./05_igp_overview.md) で
  IGP 側を整理した上で、第3部 BGP への導入とする

### 混在時の相互作用 — 静的経路は再配送で動的になりうる

静的と動的を併用するとき、両者は管理距離で競合するだけでなく、
**静的経路を動的プロトコルに注入して他のルータへ広めてもらう**
(再配送、redistribution)という連携も行われる。
末端拠点への静的経路を中核の IGP に注入すれば、中核の各ルータは
その拠点への経路を「動的に」学ぶ。

これは便利な反面、「人が書いた静止した情報」がプロトコルに乗って
ネットワーク全体へ自動拡散することを意味する。誤った静的経路
(タイポした広いプレフィックスなど)が全ルータへ配られる事故は
この経路で起こる。再配送の制御は IGP・BGP 双方で重要な設計点であり、
該当章でそれぞれ扱う。

## 設定例(補助)

Linux(iproute2)で、静的経路と管理距離(Linux では `metric` /
FRR では `distance` に相当する優先制御)によるフローティング構成の
最小例を示す。概念の確認が目的である。

```bash
# 静的経路の追加(主経路)
$ ip route add 10.0.0.0/16 via 192.0.2.1 metric 100

# バックアップの静的経路(metric を大きく=弱くして待機させる)
$ ip route add 10.0.0.0/16 via 198.51.100.1 metric 200

$ ip route show 10.0.0.0/16
10.0.0.0/16 via 192.0.2.1 dev eth0 metric 100
10.0.0.0/16 via 198.51.100.1 dev eth1 metric 200
```

この状態で転送に使われるのは metric 100 の経路のみである
(`ip route get 10.0.1.1` で確認できる)。eth0 が down して主経路が
無効化されると、metric 200 の経路が転送に使われるようになる。
ただし前述のとおり、この切り替えが起こるのは**主経路がテーブルから
消える障害**に限られる。ネクストホップの先の障害まで追従させるには
BFD などの監視と組み合わせる必要がある。

## トラブルシューティング

### 症状: 静的経路の追加後、行きは通るが応答が返らない

- 静的ルーティングでは**戻り経路も誰かが手で書かなければならない**。
  行きの経路を追加して満足し、相手側ネットワークに自分への戻り経路を
  入れ忘れるのは、静的運用で最も頻度の高いミスである。
  動的ルーティングなら双方向の経路が自動で整合するため、
  このミスは静的特有といえる
- 切り分け: 前章と同じく、相手側ルータで `ip route get <自分のアドレス>` を
  確認する。行き・帰りのテーブルは独立に壊れる、が原則である

### 症状: 主回線断のはずなのにバックアップへ切り替わらない

- フローティングスタティックや BFD なし静的経路は、
  **主経路がテーブルに残ったままの障害**(対向側だけ down、
  途中の L2 区間の障害、先のルータの故障)では切り替わらない
- 切り分け: 主経路がテーブルに残っているか(`ip route show`)をまず確認する。
  残っているなら「機器は障害を検知できていない」ということであり、
  検知手段(BFD、リンク監視)の設計問題である。
  ping が通らないのに経路が残っている、はこの型の典型症状

### 症状: 撤去したはずの経路にトラフィックが吸い込まれる

- 静的経路は消し忘れても誰も撤回してくれない。移設・廃止後に残った
  静的経路が、ロンゲストマッチで現行経路より勝ってしまう
  (前章の「具体経路の吸い込み」)事故は、静的経路の
  「変化に追従しない」性質の裏面である
- 切り分け: `ip route get <問題の宛先>` で使われている経路の情報源を確認する。
  `proto static`(または show ip route の `S`)なら、その行を書いた理由が
  現在も有効かを疑う。**静的経路には設定管理(いつ・誰が・何のために)が
  不可欠**なのはこのためである

### 症状: 動的プロトコルを導入したのに経路が一部だけ古いまま

- 静的経路(AD 1)が残っている機器では、動的経路(OSPF なら AD 110)は
  永久に勝てない。「導入時の仮設定の静的経路を消し忘れ、その機器だけ
  古い経路で動き続ける」パターンは、静的→動的の移行期に頻出する
- 切り分け: 機器間で同じプレフィックスの情報源(proto 欄)を比較する。
  1台だけ static になっていれば、それが消し忘れである

### 症状: 再配送した静的経路が原因で広域に誤経路が配られた

- 静的経路の設定ミス(プレフィックス長の誤り、宛先のタイポ)は、
  再配送されると**その機器だけの問題ではなくなる**。ネットワーク全体、
  外部と BGP 接続していればその先まで波及しうる
- 切り分け: 誤経路の情報源をたどる。動的に学んだ経路でも、
  広告元をたどっていくと再配送点(静的→動的の注入点)に行き着くことが多い。
  再配送点にはプレフィックスフィルタを置くのが定石(詳細は第3部)

## 演習・確認問題

1. 静的経路が「無効化される」条件を2つ挙げよ。また、それでも検知できない
   障害の例を図示し、その場合にパケットがどうなるかを説明せよ。
2. 外部への出口が1本しかない拠点ルータに動的ルーティングを導入しても
   得られるものが少ないのはなぜか。「迂回」という言葉を使って説明せよ。
3. フローティングスタティックルートの仕組みを、管理距離という言葉を
   使って説明せよ。また、この仕組みが機能しない障害パターンを1つ挙げよ。
4. 動的ルーティングにおける障害検知の3つの階層(物理層・プロトコルタイマー・
   BFD)を、検知の速さと検知できる範囲の観点から比較せよ。
5. 「動的経路は伝聞である」とはどういう意味か。この性質が
   もたらすリスクと、その対策の方向性(キーワードで可)を述べよ。
6. ルータ20台・全拠点単一接続のハブ&スポーク型ネットワークがある。
   中核と末端それぞれに静的/動的のどちらを使うべきか、理由とともに設計せよ。
7. 静的経路の再配送とは何か。それが便利さと引き換えに持ち込むリスクを、
   静的ルーティングの本質的性質と結びつけて説明せよ。

## まとめ

- 静的ルーティングは「人間の意思」を、動的ルーティングは「ネットワークの現状」を
  テーブルに反映する。決定性と単純さを取るか、変化への自動追従を取るかの選択である
- 静的経路は直結リンク断とネクストホップ解決不能にしか反応できず、
  その先の障害には原理的に気づけない(ブラックホール)。規模の面でも
  変更時の整合性維持が人手では破綻する — これが動的ルーティングの存在理由である
- 動的ルーティングの代償は資源消費・複雑さ・「伝聞を信じる」リスクであり、
  障害検知から収束までの速さと正確さがプロトコル設計の中心的評価軸となる
- 障害検知は物理層・プロトコルタイマー・BFD(RFC 5880)の3階層で行われ、
  BFD は静的経路にも動的な障害追従を与える
- 実務の定石は併用である: 迂回路のある中核は動的、単一接続の末端は
  静的+集約。フローティングスタティックは管理距離を道具化した
  バックアップ設計の古典である
