# ルーティングテーブルと経路選択の原理

## 概要

この章では、ルータの頭脳であるルーティングテーブルの構造と、
「複数の候補からどの経路を使うか」を決める選択規則を学ぶ。
前提知識は [L2/L3 の役割分担](./01_l2_l3_recap.md) で説明したホップバイホップ転送と
ロンゲストマッチの概念である。この章の内容は、以後のすべてのルーティングプロトコル
(OSPF、BGP)の議論の共通土台になる。

## 導入 — ルーティングテーブルとは「次の一歩」の辞書である

前章で、ルータは「宛先までの全経路を知らず、次の一歩(ネクストホップ)だけを
決めて転送する」ことを見た。ではその「次の一歩」はどこに書かれているのか —
それが**ルーティングテーブル**である。

ルーティングテーブルは本質的に、

> 「この範囲の宛先(プレフィックス)宛てなら、この隣人(ネクストホップ)に渡せ」

という行の集まりにすぎない。1行の最小構成は次の3要素である:

```text
宛先プレフィックス    ネクストホップ      出力インタフェース
198.51.100.0/24  →  192.0.2.1      via  eth0
```

ここで重要なのは、宛先が**ホスト単位ではなくプレフィックス(範囲)単位**で
書かれていることである。IP アドレスが階層的に割り当てられているからこそ、
「203.0.113.0/24 はまとめてあっち」と1行で数百のホストへの経路を表現できる。
インターネット全体(2026年時点でフルルートは IPv4 で約100万経路)ですら
テーブルに収まるのは、この集約のおかげである。プレフィックスによる経路の集約は
**CIDR(Classless Inter-Domain Routing)**として **RFC 4632** で整理されている。

ただし範囲で書く以上、**1つの宛先アドレスが複数の行にマッチする**ことが起こる。
たとえば宛先 198.51.100.20 は `198.51.100.0/24` にも `198.51.0.0/16` にも
`0.0.0.0/0` にもマッチする。さらに、同じプレフィックスへの経路を
複数の情報源(静的設定と OSPF など)が同時に主張することもある。
この「候補が複数あるとき、どれを使うか」を決める規則の全体像が本章の主題である。

## 理論

### ルーティングテーブルの1行に入っているもの

実際のルーティングテーブルの1エントリは、最小構成の3要素に加えて
次の情報を持つ:

| 要素 | 意味 | 例 |
|---|---|---|
| 宛先プレフィックス | 経路がカバーする宛先範囲 | 198.51.100.0/24 |
| ネクストホップ | パケットを引き渡す隣接ルータのアドレス | 192.0.2.1 |
| 出力インタフェース | フレームを送出するポート | eth0 |
| 情報源(プロトコル) | この経路を誰が載せたか | connected / static / OSPF / BGP |
| 管理距離(AD) | 情報源そのものの信頼度 | static=1, OSPF=110 |
| メトリック | 同一プロトコル内での経路の良さ | OSPF コスト 20 |

情報源は大きく3種類に分かれる:

1. **直結経路(connected)**: インタフェースに IP アドレスを設定すると
   自動的に載る、そのサブネットへの経路。インタフェースが down すると消える
2. **静的経路(static)**: 管理者が手で書いた経路
3. **動的経路(dynamic)**: OSPF や BGP などのルーティングプロトコルが
   学習して載せた経路

静的と動的の使い分けは [次章](./03_static_vs_dynamic.md) の主題なので、
本章では「複数の情報源が同居しうる」ことだけ押さえればよい。

### 経路選択は2つの独立した問いからなる

「どの経路を使うか」という問題は、実は**時間軸の異なる2つの問い**に分解される。
これを混同すると経路制御の議論全体が混乱するので、最初に明確に分けておく。

- **問い1(テーブル構築時)**: 同じプレフィックスへの経路候補が複数あるとき、
  どれをルーティングテーブルに**載せる**か
- **問い2(パケット転送時)**: 届いたパケットの宛先に対して、テーブル上の
  複数のエントリがマッチするとき、どの行を**使う**か

問い2の答えは前章で登場した**ロンゲストマッチ**であり、
問い1の答えが**管理距離とメトリック**である。順に見ていく。

### 問い2から: ロンゲストマッチ — 転送時の唯一の規則

パケット転送時の規則は1つしかない:
**マッチするプレフィックスのうち、プレフィックス長が最も長い(範囲が最も狭い)
行を使う**。IPv4 ルータの要件を定めた **RFC 1812** Section 5.2.4.3 でも、
最長一致が転送の基本規則として規定されている。

```text
ルーティングテーブル:
  (a) 0.0.0.0/0          → ネクストホップ X   ... デフォルトルート
  (b) 198.51.0.0/16      → ネクストホップ Y
  (c) 198.51.100.0/24    → ネクストホップ Z

宛先 198.51.100.20 のパケット → (a)(b)(c) すべてにマッチ
                             → 最長の /24 すなわち (c) が勝ち、Z へ転送
宛先 198.51.99.1 のパケット   → (a)(b) にマッチ → (b) が勝ち、Y へ転送
宛先 8.8.8.8 のパケット       → (a) のみマッチ → X へ転送
```

この規則の含意を2つ押さえておく:

- **プレフィックス長 /0 のデフォルトルート**(0.0.0.0/0)は
  「すべての宛先にマッチするが、長さ 0 なので常に最後の受け皿になる」
  経路である。「他にどこにも載っていなければこちら」を1行で表現する仕組みであり、
  特別扱いのルールがあるわけではない — ロンゲストマッチの自然な帰結である
- **具体的な経路は広い経路より常に強い**。/32(ホスト単位の経路)は
  他のどんな経路よりも優先される。これは経路制御の道具として意図的に
  使われる一方(特定宛先だけ別経路に逃がす等)、障害の原因にもなる
  (意図しない具体経路がトラフィックを吸い込む)。トラブルシューティングの節で再訪する

なおロンゲストマッチは**宛先アドレスのみ**で判定される。送信元アドレスや
ポート番号は通常の転送判断に関与しない(関与させる技術はポリシーベース
ルーティングと呼ばれ、本書では扱わない)。

### 問い1その1: 管理距離 — 情報源どうしの信頼度比較

同じプレフィックス(プレフィックス長も同じ)への経路を、複数の情報源が
同時に主張することがある。たとえば `10.1.0.0/24` への経路を静的設定と OSPF の
両方が持っているとき、テーブルにはどちらか一方だけが載る。

この決着に使われるのが**管理距離(Administrative Distance、AD)**である。
「経路の良さ」ではなく「**情報源そのものの信頼度**」を数値化したもので、
小さいほど信頼される。代表的なデフォルト値(Cisco 系の慣例):

| 情報源 | AD |
|---|---|
| 直結経路(connected) | 0 |
| 静的経路(static) | 1 |
| eBGP | 20 |
| OSPF | 110 |
| RIP | 120 |
| iBGP | 200 |

注意すべき点として、**管理距離は RFC で標準化された概念ではなく、
ベンダー実装上の慣例**である。値もベンダーによって異なる
(Linux/FRR では同種の概念を distance と呼び、Juniper では route preference と
呼んで値も異なる)。しかし「複数の情報源の優先順位を機器内部で決める仕組みが必要」
という問題自体は普遍的であり、どの実装にも同等の機構が存在する。

直結(0)と静的(1)が最強なのは直感に合う: 「自分のインタフェースに直接
つながっている」という事実と「管理者が明示的に書いた」意思は、
プロトコルが伝聞で学んだ情報より信頼される。
また iBGP(200)が全プロトコル中で最も弱い理由は、
BGP の設計思想(経路の到達性は IGP に委ねる)と関係する — 第3部で詳述する。

### 問い1その2: メトリック — 同一プロトコル内での経路の良さ

同じプレフィックスへの経路候補が**同一プロトコル内で**複数あるとき
(OSPF が2方向から同じ宛先を学んだ等)は、そのプロトコル固有の
**メトリック**で優劣を決める。小さいほど良い。

重要なのは、**メトリックの意味と計算方法はプロトコルごとに完全に別物**であり、
プロトコル間で比較できないことである:

- RIP: ホップ数(経由ルータ台数)
- OSPF: インタフェース帯域から導かれるコストの合計
- BGP: 単一のメトリックではなく、複数のパスアトリビュートによる多段判定
  (第3部の主題)

「OSPF のコスト 20 と RIP のホップ数 3 はどちらが良いか」という問いは
無意味である — だからこそ、プロトコル**間**の決着には別の物差し
(管理距離)が必要になる、という構造になっている。

### 選択の全体像 — 3段のフィルタ

ここまでを1本の流れに整理する。ある宛先へのパケットがどの経路で
転送されるかは、次の3段のフィルタで決まる:

```text
[経路の候補たち]
     │
     ▼ ① 同一プレフィックス・同一プロトコル内 → メトリック最小が勝つ
     │
     ▼ ② 同一プレフィックス・異なる情報源間 → 管理距離最小が勝つ
     │        (①②を勝ち抜いた経路がルーティングテーブルに載る)
     ▼
[ルーティングテーブル]
     │
     ▼ ③ パケット到着: マッチする全エントリ → ロンゲストマッチが勝つ
     │
[転送]
```

①②はテーブル構築時(経路を学んだとき)、③は転送時(パケットが届いたとき)の
処理である。**③のロンゲストマッチは①②より常に優先される**ことに注意してほしい。
たとえば「OSPF が学んだ /24」と「静的に書いた /16」が両方テーブルに載っているとき、
/24 の範囲内の宛先は AD で勝る静的経路ではなく OSPF の /24 で転送される。
管理距離が比較されるのは**同じプレフィックスどうし**のときだけである。

### ECMP — 甲乙つけがたいときは両方使う

3段のフィルタを通しても決着がつかない、つまり
「同一プレフィックス・同一プロトコル・同一メトリック」の経路が複数残った場合、
多くの実装はそれらを**すべて**テーブルに載せ、トラフィックを分散する。
これを **ECMP(Equal-Cost Multi-Path)**と呼ぶ。

このときパケットをどう振り分けるかが問題になる。パケット単位のラウンドロビンで
分けると、同じ TCP コネクションのパケットが別経路を通り、順序の入れ替わり
(リオーダリング)が発生して TCP の性能を大きく損なう。そのため実装は
**フロー単位**の振り分けを行う: パケットの5タプル
(送信元/宛先 IP、送信元/宛先ポート、プロトコル番号)のハッシュ値で
経路を選ぶことで、同一フローのパケットは常に同じ経路を通る。
このハッシュ方式の性質は **RFC 2992**(Analysis of an Equal-Cost Multi-Path
Algorithm)で分析されている。

ECMP は現代のデータセンターネットワーク設計の根幹であり、
第2部で扱う leaf-spine 構成は「全リンクを ECMP で使い切る」ことを
前提に設計されている。

### RIB と FIB — 「知っていること」と「引くための索引」

ここまで「ルーティングテーブル」と一括りに呼んできたが、実際のルータの内部では
2つのテーブルに分かれている:

- **RIB(Routing Information Base)**: ルーティングプロトコルや静的設定が
  経路を登録する、制御用のテーブル。①②の選択はここで行われる。
  各プロトコルはさらに自前のテーブル(OSPF のリンクステートデータベース、
  BGP テーブル)を持ち、そこから最良経路を RIB に差し出す
- **FIB(Forwarding Information Base)**: RIB の勝者だけを転送に最適化した
  形に変換したテーブル。パケット転送(③)は FIB を引いて行われる。
  ハードウェアルータでは ASIC 上のメモリに展開される

この分離は「経路を学び選ぶ処理(コントロールプレーン)」と
「パケットを高速に転送する処理(データプレーン)」の分離に対応している。
コントロールプレーン/データプレーンという見方は VXLAN の制御(第2部)や
MPLS(第5部)でも繰り返し使う、本書の基本概念である。

RIB から FIB への変換で行われる代表的な処理が、次に述べる
**再帰的ルックアップの解決**である。

### 再帰的ルックアップ — ネクストホップ自身の経路を引く

静的経路や BGP 経路では、ネクストホップとして**直結していないアドレス**が
指定されることがある。たとえば:

```text
203.0.113.0/24 → ネクストホップ 10.255.0.1   (10.255.0.1 は直結でない)
```

このままではフレームの送り先が決まらない。そこでルータは
「では 10.255.0.1 へはどう行くのか」を**もう一度ルーティングテーブルで引く**。
これを**再帰的ルックアップ(recursive lookup)**と呼ぶ:

```text
① 203.0.113.0/24 → next-hop 10.255.0.1          ... 直結でない
② 10.255.0.1 は? → 10.255.0.0/30 via 192.0.2.9 eth1 にマッチ(直結)
③ 解決結果: 203.0.113.0/24 → 192.0.2.9 @ eth1   ... FIB にはこれが載る
```

再帰は直結インタフェースに行き着くまで繰り返され、解決結果が FIB に
展開される(転送のたびに引き直すわけではない)。
**ネクストホップが解決できない経路は無効となり、FIB に載らない**。
この仕組みは iBGP の設計(ネクストホップは遠くの eBGP 境界ルータのまま
伝搬され、IGP が再帰解決を支える)を理解する上で不可欠であり、
第3部で主役級の役割を果たす。

## プロトコル動作の詳細

### 経路の一生 — 載るとき、消えるとき

ルーティングテーブルは静的な一覧表ではなく、常に更新され続ける動的な
データ構造である。経路がテーブルに載る・消える契機を押さえておく:

- **直結経路**: インタフェースが up し、IP アドレスが設定されたときに載る。
  インタフェースが down すると**即座に消える**。さらに、そのインタフェースを
  出力とする静的経路や、そのネクストホップに依存していた再帰経路も連鎖的に無効になる
- **静的経路**: 設定時に載る。ただし多くの実装では、指定されたネクストホップが
  解決不能(該当経路がない)の間はテーブルに載らない
- **動的経路**: プロトコルが最良経路を選んだときに載り、
  ネイバー断・上位経路の消滅・より良い経路の出現などで差し替え・撤回される

「経路が消えると、それに依存していた経路も連鎖して消える」という依存関係が、
障害時にネットワーク全体へ変化が波及していく基本メカニズムである。
この波及の速さ(収束、convergence)はルーティングプロトコル設計の
中心的な評価軸であり、[ディスタンスベクタとリンクステートの章](./04_distance_vector_link_state.md)
で本格的に扱う。

### ロンゲストマッチ検索の実装 — なぜ高速に引けるのか

「全エントリと突き合わせて最長のものを選ぶ」を素朴にやると、
100万経路のテーブルでは転送のたびに100万回の比較が必要になり、現実的でない。
実装では、プレフィックスをビット列の木構造
(基数木/トライ木、radix trie)として持つのが古典的な方法である:

```text
宛先アドレスのビット列を上位から1ビットずつたどる:

            (root)
           /      \
          0        1
         / \        \
       ...  10.0.0.0/8      ← たどる途中でマッチした最長のものを覚えておく
              \
               10.1.0.0/16  ← さらに深くでマッチすればそちらで上書き
```

木を根から葉へ1回たどるだけで、途中で出会った「最も深い(=最長の)マッチ」が
答えになる。検索コストはテーブルサイズではなくアドレス長(IPv4 で最大32ステップ)
に比例する。ハードウェアルータではさらに TCAM(内容参照可能メモリ)や
専用 ASIC により、テーブルサイズによらず一定時間で検索される。

実装の詳細を暗記する必要はないが、
**「ロンゲストマッチは高速に引ける。ただしテーブルのメモリは有限」**という
感覚は持っておきたい。フルルートを保持できる機器とできない機器の違い、
経路集約がなぜ大切かは、この資源制約の話である。

### 経路集約と Null ルート

複数の具体的な経路を1つの広い経路にまとめることを**経路集約(route aggregation)**
と呼ぶ。`10.1.0.0/24` と `10.1.1.0/24` … `10.1.255.0/24` を外部には
`10.1.0.0/16` の1本として広告すれば、外部のテーブルを256分の1にできる。

ただし集約には副作用がある。集約経路 `10.1.0.0/16` を広告しているルータが、
実際には `10.1.42.0/24` への経路を持っていない場合、その宛先へのパケットは
集約ルータまで届いてから行き場を失う。このとき、集約ルータのテーブルに
デフォルトルートがあると、パケットはデフォルトルートに従って外へ送り返され、
**外部との間で永久ループ**が生じうる(TTL が尽きるまで往復する)。

この対策として、集約を行うルータは集約プレフィックスへの
**Null ルート(廃棄経路)**を自分のテーブルに置くのが定石である:

```text
10.1.0.0/16 → Null0(廃棄)     ← 集約の「底」
10.1.1.0/24 → eth1              ← 実在する具体経路
10.1.2.0/24 → eth2
```

実在する宛先はロンゲストマッチで具体経路に流れ、実在しない宛先だけが
/16 の底に落ちて安全に廃棄される。「ロンゲストマッチがあるからこそ、
広い廃棄経路と狭い有効経路を安全に同居させられる」好例である。

### uRPF — テーブルを逆向きに使う

ルーティングテーブルの応用として、**送信元**の検証にテーブルを使う
**uRPF(unicast Reverse Path Forwarding、RFC 3704)**にも触れておく。
受信したパケットの送信元アドレスをテーブルで引き、
「その送信元へ向かう経路が、受信したインタフェースを向いているか」を検査して、
向いていなければ偽装アドレスの疑いありとして破棄する仕組みである。
送信元アドレス偽装(スプーフィング)対策の基本であり、
「テーブルは宛先を引くためのもの」という固定観念を外してくれる例でもある。

## 設定例(補助)

Linux でルーティングテーブルと再帰解決を観察する。概念と表示の対応を
確認することが目的である。

```bash
# ルーティングテーブルの表示
$ ip route show
default via 192.0.2.1 dev eth0 proto dhcp metric 100
192.0.2.0/24 dev eth0 proto kernel scope link src 192.0.2.10
10.99.0.0/16 via 192.0.2.254 dev eth0 proto static
```

- 1行目: デフォルトルート(0.0.0.0/0 の表記が `default` になる)。
  `proto dhcp` は情報源が DHCP であることを示す
- 2行目: 直結経路。`proto kernel` はアドレス設定に伴い kernel が自動生成したもの。
  ネクストホップがなく `dev eth0` だけなのは「直接 ARP 解決して送る」ことを意味する
- 3行目: 静的経路

特定の宛先がどの経路で転送されるかは、テーブルを目で追わなくても
カーネルに直接聞ける。ロンゲストマッチと再帰解決を済ませた**結果**が返る:

```bash
$ ip route get 10.99.42.7
10.99.42.7 via 192.0.2.254 dev eth0 src 192.0.2.10
```

「この宛先、どの経路に乗る?」を確認するこのコマンドは、
トラブルシューティングの第一手として非常に有用である。

## トラブルシューティング

### 症状: 経路はあるのに通信できない — 戻り経路を疑う

- 行きの経路がテーブルにあっても、**相手側から自分への戻り経路**がなければ
  通信は成立しない。行きと帰りのテーブルは独立であり、独立に故障する
- 切り分け: 相手側(または経路上のルータ)で `ip route get <自分のアドレス>` を
  実行し、戻りの経路が期待どおりかを確認する。
  「ping は相手に届いている(相手側キャプチャで見える)のに応答が返らない」
  なら、ほぼ戻り経路の問題である

### 症状: 一部の宛先だけ意図しない経路に流れる — 具体経路の吸い込み

- テーブルに意図しない**より長いプレフィックス**が紛れ込むと、
  その範囲の宛先だけがロンゲストマッチで吸い込まれる。
  全体は正常なのに特定宛先だけおかしい、という症状の典型原因である
- 切り分け: `ip route get <問題の宛先>` で実際に使われる経路を確認し、
  期待した経路とプレフィックス長を比較する。
  「/0 や /16 を直したのに変わらない」ときは、より長いマッチが必ずどこかにいる

### 症状: 設定した静的経路がテーブルに現れない

- ネクストホップが再帰解決できない(そのアドレスへの経路がない、または
  解決に使うインタフェースが down)場合、経路は無効化されテーブルに載らない
- 切り分け: `ip route get <ネクストホップのアドレス>` でネクストホップ自身への
  経路を確認する。依存先の直結経路が生きているか、インタフェース状態も見る

### 症状: 同じ宛先への経路が機器によって違う情報源で載っている

- 複数プロトコルを併用している環境で、機器 A では OSPF 経路、機器 B では
  静的経路が勝っている、という不整合は管理距離の理解不足から起こりやすい。
  特に「静的経路(AD 1)を仮置きしたまま OSPF を導入し、その機器だけ
  永久に静的が勝ち続ける」パターンは頻出である
- 切り分け: 経路の情報源(proto 欄や show ip route の先頭コード)を機器間で
  比較する。消したはずの静的設定が残っていないかを確認する

### 症状: ECMP 環境で特定のフローだけ落ちる・遅い

- ECMP はフロー単位で経路を固定するため、**複数経路のうち1本だけが劣化**
  していると、「あるフローは常に正常、別のフローは常に異常」という
  再現性のある偏りが生じる。ping(ICMP)とアプリ(TCP)でハッシュが異なる
  経路に乗り、「ping は通るのにアプリだけ死ぬ」ことも起こる
- 切り分け: 送信元ポートを変えながら試験して症状が変わるかを見る
  (ポートが変わればハッシュが変わり、乗る経路が変わる)。
  traceroute でも実装によりポートを変えつつ探索できる

## 演習・確認問題

1. ルーティングテーブルに `0.0.0.0/0 via A`、`172.16.0.0/16 via B`、
   `172.16.5.0/24 via C` の3行があるとき、宛先 172.16.5.9、172.16.9.5、
   8.8.8.8 のパケットはそれぞれどこへ転送されるか。
2. 「OSPF で学んだ 10.0.0.0/24(AD 110)」と「静的に設定した 10.0.0.0/16(AD 1)」が
   同じテーブルにあるとき、宛先 10.0.0.5 のパケットはどちらの経路で転送されるか。
   理由とともに述べよ。
3. 管理距離とメトリックはそれぞれ「何と何を比較する」ための物差しか。
   OSPF のコストと RIP のホップ数を直接比較できないのはなぜか。
4. 再帰的ルックアップとは何か。ネクストホップが解決できない経路は
   どう扱われるか。
5. ECMP でパケット単位ではなくフロー単位の振り分けが使われるのはなぜか。
   また、この性質が障害切り分けにどんな影響を与えるか。
6. 経路集約を行うルータが、集約プレフィックスへの Null ルートを自分の
   テーブルに置くのはなぜか。置かない場合に何が起こりうるか。
7. RIB と FIB の役割の違いを、コントロールプレーン/データプレーンという
   言葉を使って説明せよ。

## まとめ

- ルーティングテーブルは「プレフィックス → ネクストホップ」の辞書であり、
  階層的アドレスによる集約(CIDR、RFC 4632)がその規模を支えている
- 経路選択は3段のフィルタで決まる: 同一プロトコル内はメトリック、
  情報源間は管理距離、そして転送時はロンゲストマッチ。
  ロンゲストマッチは常に最優先であり、管理距離は同一プレフィックス間でしか働かない
- 決着がつかない等コスト経路は ECMP としてフロー単位で分散される
- ルータ内部では経路を選ぶ RIB と転送に使う FIB が分離しており
  (コントロールプレーン/データプレーンの分離)、直結でないネクストホップは
  再帰的ルックアップで解決されてから FIB に載る
- 経路は依存関係を持って生き死にする。インタフェース断は直結経路を消し、
  それに依存する経路を連鎖的に無効化する — この波及が「収束」の正体である
